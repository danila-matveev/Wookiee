## 1. Градиент сложности: от скрипта до мульти-агентной системы

Не каждая задача требует агента. Часто достаточно промпта или скрипта.

| Уровень | Тип решения | Когда использовать | Примеры Wookiee |
|---------|-------------|-------------------|----------------|
| **0** | **Одноразовый промпт** | Простой вопрос, разовый анализ | "Покажи топ-5 моделей по марже за вчера" |
| **1** | **Prompt Template** | Повторяющаяся задача, фиксированная структура | Шаблон дневного отчёта (без доступа к БД) |
| **2** | **Скрипт с LLM** | Детерминированная логика + генерация текста | `scripts/abc_analysis.py` — расчёты в Python, интерпретация через LLM |
| **3** | **Single-Shot Agent** | Нужен доступ к инструментам, но без итераций | Василий v1 — один вызов `generate_localization_report()` |
| **4** | **ReAct Agent (базовый)** | Нужны итерации: запрос → анализ → уточнение | Олег v1 — ответы на вопросы пользователя через БД |
| **5** | **ReAct Agent + Memory** | Нужен контекст прошлых сессий | Людмила — ведёт историю переписки с клиентами |
| **6** | **Agentic Workflow** | Многошаговый процесс с проверками | Олег v2 — self-verification после расчётов |
| **7** | **Multi-Agent System** | Специализация + оркестрация | Будущее: Manager (планирование) → Олег (финансы) + Василий (логистика) |

**Золотое правило:** Начинай с минимального уровня. Повышай сложность только если доказана необходимость.

**Примеры эволюции:**
- **Василий:** уровень 2 (скрипт) → уровень 3 (single-shot agent) — когда добавили интерактивность через Telegram
- **Олег:** уровень 4 (базовый ReAct) → уровень 6 (agentic workflow) — когда добавили self-verification vs OneScreen

## 2. Анатомия агента: 4 компонента

Каждый агент в Wookiee состоит из 4 обязательных компонентов:

```
┌─────────────────────────────────────────┐
│              AI AGENT                   │
├─────────────────────────────────────────┤
│ 1. BRAIN (LLM + orchestration layer)   │
│ 2. PLAYBOOK (instructions + rules)      │
│ 3. TOOLS (functions + data access)      │
│ 4. MEMORY (context + history)           │
└─────────────────────────────────────────┘
```

### 2.1 Brain (Мозг)

**Brain** = LLM модель + оркестрационный слой (harness).

#### Выбор модели

| Задача | Модель | Почему |
|--------|--------|--------|
| **Финансовый анализ** | Claude Opus 4.6 | Сложные рассуждения, точность в расчётах, самокоррекция |
| **Классификация/извлечение** | Claude Sonnet 4.5 | Баланс скорости и качества |
| **Генерация шаблонного текста** | Claude Haiku 4.0 | Быстро и дешево |
| **Код-генерация** | Claude Opus 4.6 | Понимание контекста кодовой базы |

**Правила переключения:**
- Начинай с Sonnet. Повышай до Opus только если качество недостаточно.
- Понижай до Haiku если задача шаблонная и скорость критична.
- НЕ используй GPT-4 для финансов — Claude показывает лучшую точность в вычислениях.

#### Оркестрация (Harness)

Мы используем **LangGraph** для оркестрации агентов:
- Управление состоянием (StateGraph)
- Ветвление логики (conditional edges)
- Retry-логика и обработка ошибок
- Streaming вывода

Пример структуры:
```python
# agents/<name>/service.py
from langgraph.graph import StateGraph

def create_agent_graph():
    workflow = StateGraph(AgentState)
    workflow.add_node("think", thinking_node)
    workflow.add_node("act", action_node)
    workflow.add_node("verify", verification_node)
    workflow.add_conditional_edges("act", should_verify)
    return workflow.compile()
```

### 2.3 Tools (Инструменты)

**Tools** = функции, которые агент может вызывать для выполнения действий.

#### Дизайн-правила инструментов

**Золотое правило (Anthropic):**
> "Один инструмент = одна чётко определённая задача. Композируй простые инструменты, а не создавай сложные монолиты."

**7 правил дизайна инструментов:**

1. **Атомарность:** Один инструмент = одна операция
   - ❌ `get_analytics(type, period, channel, models, ...)` — слишком много параметров
   - ✅ `get_channel_finance(channel, date_from, date_to)` — одна чёткая задача

2. **Идемпотентность:** Повторный вызов с теми же параметрами = тот же результат
   - ✅ `get_margin_levers(date)` — всегда возвращает одно и то же для даты
   - ❌ `send_report()` — побочный эффект (отправка)

3. **Понятная сигнатура:** Агент должен понимать параметры без документации
   ```python
   # ❌ Плохо
   def get_data(dt, ch, md):
       ...

   # ✅ Хорошо
   def get_channel_finance(
       channel: Literal["wb", "ozon", "all"],
       date_from: str,  # ISO format YYYY-MM-DD
       date_to: str
   ) -> Dict[str, Any]:
       ...
   ```

4. **Валидация входов:** Проверяй параметры до выполнения
   ```python
   if channel not in ["wb", "ozon", "all"]:
       raise ValueError(f"Unknown channel: {channel}")
   ```

5. **Структурированный вывод:** JSON/Dict, не текст
   - ✅ `{"revenue": 1234567, "margin": 234567}`
   - ❌ `"Выручка: 1 234 567 руб, маржа: 234 567 руб"`

6. **Обработка ошибок:** Возвращай ошибки в структурированном виде
   ```python
   return {
       "success": False,
       "error": "No data for date 2026-02-30",
       "error_code": "INVALID_DATE"
   }
   ```

7. **Документация в docstring:** Описание, параметры, возвращаемое значение, примеры
   ```python
   def get_margin_levers(date: str) -> Dict[str, Any]:
       """
       Вычисляет 5 рычагов маржи для заданной даты.

       Args:
           date: Дата в формате YYYY-MM-DD

       Returns:
           {
               "price_before_spp": float,      # ₽/ед
               "spp_percent": float,            # %
               "drr_percent": float,            # %
               "logistics_per_unit": float,     # ₽/ед
               "cogs_per_unit": float           # ₽/ед
           }

       Example:
           >>> get_margin_levers("2026-02-10")
           {"price_before_spp": 1750.5, ...}
       """
   ```

#### Реестр инструментов Wookiee

Все инструменты живут в:
- `shared/tools/` — общие инструменты (доступ к БД, API клиенты)
- `agents/<name>/tools/` — специфичные для агента

| Инструмент | Назначение | Агенты |
|-----------|-----------|--------|
| `get_channel_finance()` | Финансовые метрики канала | Олег |
| `get_margin_levers()` | 5 рычагов маржи | Олег |
| `get_model_breakdown()` | Декомпозиция по моделям | Олег |
| `get_traffic_data()` | Данные трафика (content_analysis) | Олег |
| `verify_against_onescreen()` | Сверка с OneScreen | Олег |
| `generate_localization_report()` | Отчёт индексации WB | Василий |
| `get_bitrix_deals()` | Список сделок CRM | Людмила |
| `update_bitrix_deal()` | Обновление сделки | Людмила |

### 2.4 Memory (Память)

**Memory** = контекст, который агент сохраняет между вызовами.

#### 3 уровня памяти

| Уровень | Живёт | Хранение | Пример использования |
|---------|-------|----------|---------------------|
| **Short-term** | В рамках одной сессии | In-memory (LangGraph State) | Олег помнит предыдущие вопросы в чате |
| **Medium-term** | В рамках дня/недели | SQLite / Redis | Василий помнит последний отчёт (не пересчитывать) |
| **Long-term** | Постоянно | PostgreSQL / Supabase | Людмила помнит всю историю клиента |

**Правила работы с памятью:**

1. **Short-term (в сессии):**
   - Храни в `AgentState` (LangGraph)
   - Очищай после завершения сессии
   - Лимит: последние 10-20 сообщений (контекст окно)

2. **Medium-term (день/неделя):**
   - Используй для кеширования расчётов
   - TTL (time to live) обязателен
   - Пример: `cache_key = f"margin_levers_{date}", ttl=86400`

3. **Long-term (постоянная):**
   - Только для критически важного контекста
   - Индексируй для быстрого поиска
   - Пример: `agent_memory` таблица в Supabase с `user_id`, `context`, `timestamp`

**Анти-паттерны:**
- ❌ Хранить весь чат в памяти (context overflow)
- ❌ Не очищать устаревшую память (memory leak)
- ❌ Запоминать PII (персональные данные) без шифрования

## 3. ReAct Loop: Think → Act → Observe → Repeat

**ReAct** = Reasoning + Acting. Агент рассуждает перед действием и анализирует результат.

### Диаграмма ReAct цикла

```
┌──────────────┐
│   THINK      │ ← Получил вопрос пользователя
│ (Reasoning)  │   "Какая маржа за вчера?"
└──────┬───────┘
       │
       ▼
┌──────────────┐
│   ACT        │ ← Вызываю инструмент
│ (Action)     │   get_channel_finance("all", "2026-02-13", "2026-02-13")
└──────┬───────┘
       │
       ▼
┌──────────────┐
│  OBSERVE     │ ← Получил результат
│ (Observation)│   {"margin": 271770, "revenue": 1234567, ...}
└──────┬───────┘
       │
       ▼
┌──────────────┐
│  THINK       │ ← Анализирую: достаточно ли данных?
│ (Reasoning)  │   "Нужна ли декомпозиция по моделям?"
└──────┬───────┘
       │
       ├─── Да ─────► ACT: get_model_breakdown(...)
       │
       └─── Нет ────► RESPOND: "Маржа за вчера: 271 770 руб (22%)"
```

### 5 обязательных правил ReAct в Wookiee

1. **Явное рассуждение (thinking out loud):**
   - Агент ДОЛЖЕН объяснить почему выбирает действие
   - Логируется в `<thinking>...</thinking>` блоке
   - Пример:
     ```xml
     <thinking>
     Пользователь спрашивает про маржу за вчера.
     Мне нужны данные по WB и OZON за 2026-02-13.
     Вызову get_channel_finance("all", "2026-02-13", "2026-02-13")
     </thinking>
     ```

2. **Одно действие за итерацию:**
   - НЕ вызывай 5 инструментов сразу
   - Вызови один, проанализируй результат, решай что дальше
   - Исключение: параллельные независимые запросы (WB и OZON отдельно)

3. **Проверка результата перед ответом:**
   - После получения данных — верифицируй
   - Олег: обязательная сверка с OneScreen перед выводом маржи
   - Если расхождение > 1% → СТОП, логируй в DATA_QUALITY_NOTES.md

4. **Максимум итераций = 10:**
   - Защита от зацикливания
   - После 10 итераций → STOP, вернуть "Не смог выполнить задачу, нужна помощь"

5. **Сохранение трейса:**
   - Каждая итерация логируется: thinking → action → observation
   - Формат: JSON Lines в `logs/agent_traces/<agent_name>_<timestamp>.jsonl`
   - Используется для отладки и улучшения playbook

## 4. Self-Correction и обработка ошибок

Агенты НЕ идеальны. Они должны уметь исправлять свои ошибки.

### 4.1 Check-After-Action Pattern

**Паттерн:** После выполнения действия — проверь результат на корректность.

Пример для Олега:
```python
# 1. Вычисли маржу
margin_data = get_channel_finance("all", date, date)

# 2. Верифицируй против OneScreen
verification = verify_against_onescreen(margin_data, date)

# 3. Проверь расхождение
if verification["discrepancy_percent"] > 1.0:
    # Ошибка! Логируй и уведомляй
    log_data_quality_issue(verification)
    return {
        "success": False,
        "error": f"Расхождение с OneScreen: {verification['discrepancy_percent']}%"
    }

# 4. Только после проверки — отдавай результат пользователю
return margin_data
```

### 4.2 Правила самокоррекции

| Ситуация | Действие агента | Пример |
|----------|----------------|--------|
| **Ошибка в расчётах** | Пересчитать с правильной формулой | Олег использовал 11-полевую формулу маржи → обнаружил расхождение → пересчитал с `marga - nds - reclama_vn` |
| **Некорректный параметр** | Переспросить пользователя | `get_channel_finance(channel="wildberries")` → ошибка → "Канал должен быть 'wb', 'ozon' или 'all'" |
| **Недостаточно данных** | Запросить дополнительные инструменты | Спросили про драйверы маржи → `get_channel_finance()` не достаточно → вызвать `get_model_breakdown()` |
| **Расхождение с источником истины** | Логировать, уведомить, СТОП | Олег: маржа расходится с OneScreen > 1% → СТОП анализа, фиксация в DATA_QUALITY_NOTES.md |
| **Инструмент вернул ошибку** | Retry до 3 раз с exponential backoff | API timeout → retry через 1s, 2s, 4s → после 3 попыток → вернуть ошибку пользователю |

### 4.3 Рефлексия (Reflection)

После завершения задачи агент ДОЛЖЕН:
1. Оценить качество своего ответа (0-10)
2. Определить что можно улучшить
3. Записать в лог для последующего анализа

Формат reflection:
```json
{
  "task": "Анализ маржи за 2026-02-13",
  "quality_score": 8,
  "what_went_well": [
    "Верификация против OneScreen прошла (расхождение 0.03%)",
    "Декомпозиция по моделям показала драйверы"
  ],
  "what_to_improve": [
    "Не проверил трафик при анализе падения заказов",
    "Забыл показать средневзвешенный СПП%"
  ],
  "action_items": [
    "Добавить в playbook: обязательная проверка трафика при изменении заказов > 10%"
  ]
}
```

Reflection используется для:
- Автоматического обновления playbook (через LLM)
- A/B тестирования промптов (какой даёт выше quality_score)
- Обучения новых агентов (лучшие практики из reflection логов)

### 4.4 Fallback-иерархия

Если агент не может выполнить задачу:

```
1. Попробуй альтернативный инструмент
   ↓ (не помогло)
2. Упрости задачу (разбей на подзадачи)
   ↓ (не помогло)
3. Спроси пользователя уточнение
   ↓ (пользователь не ответил / не помогло)
4. Передай задачу человеку (escalation)
   ↓
5. Логируй инцидент в agent_failures.md
```

Пример escalation для Олега:
```python
if attempts > 3 and not success:
    return {
        "success": False,
        "escalate_to_human": True,
        "reason": "Не могу вычислить маржу — расхождение с OneScreen > 5%",
        "context": {...}  # Все попытки и результаты
    }
```

## 5. Чек-лист дизайна инструментов

Перед добавлением нового инструмента пройди по чек-листу:

- [ ] **Атомарность:** Инструмент делает ОДНУ вещь?
- [ ] **Идемпотентность:** Повторный вызов = тот же результат?
- [ ] **Понятная сигнатура:** Параметры названы понятно? Type hints добавлены?
- [ ] **Валидация входов:** Параметры проверяются до выполнения?
- [ ] **Структурированный вывод:** Возвращает JSON/Dict, не текст?
- [ ] **Обработка ошибок:** Ошибки возвращаются в структурированном виде?
- [ ] **Документация:** Docstring описывает назначение, параметры, возвращаемое значение, примеры?
- [ ] **Unit-тесты:** Покрыты основные сценарии и edge cases?
- [ ] **Integration test:** Тест вызова инструмента через агента?
- [ ] **Добавлен в реестр:** Инструмент описан в `docs/guides/agent-principles.md` (этот файл)?

## 6. Multi-Agent: когда добавлять второго агента?

Мульти-агентные системы сложнее. Добавляй второго агента ТОЛЬКО если:

### Критерии добавления второго агента

1. **Специализация:** Задачи требуют принципиально разных знаний
   - ✅ Финансовый анализ (Олег) + Логистическая оптимизация (Василий)
   - ❌ Анализ WB + анализ OZON (один агент справится)

2. **Параллелизация:** Задачи выполняются одновременно
   - ✅ Олег анализирует маржу, пока Василий оптимизирует склады
   - ❌ Последовательные этапы одной задачи

3. **Разная скорость/стоимость:** Быстрый agent фильтрует → медленный углубляется
   - ✅ Sonnet классифицирует запрос → Opus делает детальный анализ
   - ❌ Два Opus агента дублируют работу

4. **Изоляция рисков:** Критичный агент не должен ломаться из-за экспериментов
   - ✅ Production Олег (стабильный) + Experimental Аналитик (тестируем новые подходы)
   - ❌ Один агент с нестабильным кодом

### Manager-Workers Pattern

Когда у тебя 2+ специализированных агента — добавь Manager (менеджер-агент).

```
┌───────────────────┐
│     MANAGER       │ ← Получает задачу от пользователя
│  (координатор)    │   "Проанализируй маржу и оптимизируй логистику"
└─────────┬─────────┘
          │
    ┌─────┴─────┐
    │           │
    ▼           ▼
┌────────┐  ┌──────────┐
│  OLEG  │  │ VASILY   │ ← Workers (специалисты)
│финансы │  │логистика │
└────┬───┘  └────┬─────┘
     │           │
     └─────┬─────┘
           ▼
     ┌───────────┐
     │  MANAGER  │ ← Агрегирует результаты
     │  (синтез) │   "Маржа: ..., Склады: ..."
     └───────────┘
```

**Правила Manager-агента:**
- Использует быстрый LLM (Sonnet) — роутинг не требует глубоких рассуждений
- НЕ выполняет работу сам — только делегирует и агрегирует
- Знает capabilities каждого Worker-агента (через `tools/agent_registry.json`)
- Обрабатывает конфликты (если Workers дают противоречивые рекомендации)

**Когда НЕ нужен Manager:**
- У тебя 2 агента и задачи не пересекаются → прямой вызов агентов
- Пользователь сам знает какого агента вызывать → UI с выбором агента

## 7. Безопасность и governance

### Правила безопасности агентов

| Правило | Почему | Как проверить |
|---------|--------|---------------|
| **Агент НЕ имеет доступа к .env** | Предотвращение утечки секретов | `.env` в `.cursorignore` |
| **Все DB-запросы через data_layer.py** | Единая точка аудита и валидации | `grep -r "supabase.from_" agents/` должен быть пуст |
| **Агент НЕ может удалять данные** | Защита от ошибок | Инструменты только READ, UPDATE через approval |
| **Rate limits на API вызовы** | Предотвращение DDoS и перерасхода | `@rate_limit(calls=100, period=60)` decorator |
| **Логирование ВСЕХ действий** | Аудит для compliance | `logs/agent_traces/<agent>_<timestamp>.jsonl` |
| **PII НЕ логируется в plaintext** | GDPR / privacy | Фильтр PII перед записью в лог |
| **Approval для critical actions** | Человек в петле для рискованных операций | `if action.is_critical: await ask_user_approval()` |

### Дефолтные лимиты

| Параметр | Лимит | Обоснование |
|----------|-------|-------------|
| **Max iterations** | 10 | Защита от зацикливания |
| **Max tool calls per iteration** | 5 | Защита от неконтролируемого каскада |
| **Context window** | 20 сообщений | Баланс памяти и качества |
| **Response timeout** | 60 секунд | UX — пользователь не будет ждать дольше |
| **API rate limit** | 100 запросов/мин | Защита от перерасхода |
| **Cache TTL** | 24 часа | Актуальность данных vs нагрузка на БД |

## 8. Стандартная структура файлов агента

Каждый агент в Wookiee следует единой структуре:

```
agents/<agent_name>/
├── README.md                   # Описание агента, как запускать, примеры
├── service.py                  # Основная логика агента (LangGraph workflow)
├── playbook.md                 # Живая инструкция (playbook)
├── tools/                      # Специфичные инструменты агента
│   ├── __init__.py
│   ├── <tool_name>.py
│   └── tests/
│       └── test_<tool_name>.py
├── prompts/                    # Системные промпты и шаблоны
│   ├── system_prompt.txt
│   └── templates/
│       ├── daily_report.md
│       └── weekly_report.md
├── memory/                     # Память агента (если используется)
│   ├── cache.db               # SQLite для medium-term
│   └── context_manager.py     # Управление контекстом
├── tests/                      # Unit и integration тесты
│   ├── test_service.py
│   └── test_integration.py
├── logs/                       # Логи агента (git-ignored)
│   └── .gitkeep
├── feedback_log.md             # История feedback от пользователей
└── __init__.py
```

**Обязательные файлы:**
- `README.md` — как пользоваться агентом
- `service.py` — точка входа
- `playbook.md` — правила работы
- `feedback_log.md` — история улучшений

**Опциональные:**
- `tools/` — если есть специфичные инструменты
- `prompts/` — если промпты сложные и шаблонизированы
- `memory/` — если нужна persistent память

## 9. Чек-лист создания нового агента

Перед добавлением нового агента пройди по чек-листу:

### Этап 1: Обоснование

- [ ] **Нужен ли агент?** Может достаточно скрипта или промпта? (см. раздел 1)
- [ ] **Уровень сложности определён?** (0-7, см. таблицу градиента)
- [ ] **Задача не пересекается с существующими агентами?** Если пересекается — расширить существующего
- [ ] **Есть бизнес-метрика успеха?** Как измерим что агент полезен?

### Этап 2: Проектирование

- [ ] **Brain:** Выбрана модель (Opus/Sonnet/Haiku)
- [ ] **Playbook:** Написаны правила работы (используй шаблон из 2.2)
- [ ] **Tools:** Определены инструменты (минимум 2, максимум 10)
- [ ] **Memory:** Решено нужна ли память (short/medium/long)
- [ ] **Безопасность:** Проверены правила из раздела 7

### Этап 3: Разработка

- [ ] **Структура папок создана** (см. раздел 8)
- [ ] **service.py реализован** (LangGraph workflow)
- [ ] **Инструменты реализованы** (соответствуют чек-листу из раздела 5)
- [ ] **Промпты написаны** (system_prompt.txt)
- [ ] **Unit-тесты написаны** (покрытие > 70%)
- [ ] **Integration тесты написаны** (минимум 3 сценария)

### Этап 4: Тестирование

- [ ] **Тест на корректность:** Агент выполняет задачу правильно?
- [ ] **Тест на устойчивость:** Агент обрабатывает ошибки (некорректные входы, timeout API)?
- [ ] **Тест на безопасность:** Агент не может сделать запрещённые действия?
- [ ] **User Acceptance Test:** Показан реальному пользователю, feedback собран?

### Этап 5: Документация

- [ ] **README.md заполнен** (как запустить, примеры)
- [ ] **docs/agents/<agent_name>.md создан** (концептуальное описание)
- [ ] **Инструменты добавлены в реестр** (в этот файл, раздел 2.3)
- [ ] **ADR создан** если принято архитектурное решение (`docs/adr.md`)

### Этап 6: Деплой

- [ ] **Переменные окружения добавлены** (`.env.example` обновлён)
- [ ] **CI/CD настроен** (тесты запускаются автоматически)
- [ ] **Мониторинг настроен** (логи, метрики, алерты)
- [ ] **Rollback план готов** (как откатить если что-то сломалось)
