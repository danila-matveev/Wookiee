## 1. Контекст и цель

Текущий сервис `services/wb_localization/` (бывший «Escript», ex-Vasily) уже считает индекс локализации (ИЛ) и индекс распределения продаж (ИРП), генерирует рекомендации по перестановкам товаров между складами WB и экспортирует 8 листов в Google Sheets. Его логика разнесена по `calculators/`, `generate_localization_report_v3.py` и `sheets_export.py` (59KB).

**Что нужно улучшить:**

1. **Справочник** — сейчас это минимальная таблица КТР/КРП. Должен превратиться в полноценную визуальную документацию: как считается ИЛ/ИРП, как работает формула логистики WB, как работают перестановки, какие лимиты по складам, как интерпретировать 13-недельное скользящее окно.
2. **Экономика сценариев** — сейчас 3 сценария (40% / текущий / 80%). Нужна градация 30–90% с шагом 10%, с «кислотной» сводной таблицей по всему кабинету и топ-артикулами, которые тянут индекс вниз.
3. **Перестановки + Roadmap** — сейчас просто список перемещений. Нужен автоматический прогноз: если начнём делать перестановки сегодня, с учётом реальных лимитов складов — как будет меняться ИЛ/ИРП по неделям, когда пересечём порог 60% (КРП→0), когда достигнем 80% (КТР=0.80). Прогноз строится на реальных данных артикулов и учитывает 13-недельное скользящее окно WB.
4. **Синхронизация КТР** — текущая таблица в `irp_coefficients.py` расходится с официальной WB (например, 2.20 при 0–5% vs официальный 2.00). Нужно привести в соответствие.

**Цель дизайна:** единый модульный сервис из независимых блоков (анализ, экономика, прогноз, перестановки), каждый работает изолированно и вместе, результат публикуется в Google Sheets с подробной документацией по каждой колонке.

## 2. Архитектура

Подход: расширение существующего `services/wb_localization/` с разбиением монолитного `sheets_export.py` на модули по смыслу.

### 2.1 Структура файлов

```
services/wb_localization/
├── calculators/                      # БИЗНЕС-ЛОГИКА (чистые функции)
│   ├── il_irp_analyzer.py            ✅ existing — расчёт ИЛ/ИРП
│   ├── economic_analyzer.py          ✅ existing — 3-scenario economics
│   ├── scenario_engine.py            🆕 градация 30–90%, что-если
│   ├── relocation_forecaster.py      🆕 roadmap с 13-недельным окном
│   └── reference_builder.py          🆕 структура справочника
│
├── sheets_export/                    # ВЫВОД (разбиение монолита 59KB)
│   ├── __init__.py                   🆕 фасад (экспортирует export_to_sheets)
│   ├── formatters.py                 🆕 общие хелперы (_header_fmt, _borders, ...)
│   ├── core_sheets.py                🆕 Перемещения, Допоставки, Сводка, Регионы
│   ├── analysis_sheets.py            🆕 ИЛ Анализ (текущий)
│   ├── reference_sheet.py            🆕 Расширенный справочник
│   ├── scenario_sheet.py             🆕 Экономика сценариев (градация)
│   └── roadmap_sheet.py              🆕 Перестановки + Roadmap
│
├── irp_coefficients.py               🔧 СИНХРОНИЗАЦИЯ с офиц. WB КТР
├── run_localization.py               🔧 orchestration + новые шаги
└── history.py                        🔧 +weekly_snapshots для forecast
```

### 2.2 Принципы

1. **Независимость модулей.** Каждый калькулятор принимает DataFrame/dict, возвращает dict. Не знает про Sheets, API, БД. Можно тестировать изолированно. Можно вызвать любой калькулятор отдельно из CLI или сервиса.

2. **Pipeline опциональных шагов** в `run_localization.py`:
   ```
   core_analysis → il_irp → economics → scenarios → forecast → export
   ```
   Каждый шаг включается/выключается флагом CLI (`--skip-forecast`, `--skip-scenarios` и т.п.). Дефолт — всё включено.

3. **Разделение по слоям.** Калькуляторы не знают про Sheets, листы не знают про WB API. Данные текут в одну сторону: API → calculators → sheets.

4. **Совместимость.** Существующие листы (Перемещения, Допоставки, Сводка, Регионы, ИЛ Анализ) продолжают работать. Новые добавляются, старая «Экономика» заменяется на новый сценарный лист, старый «Справочник» заменяется на расширенный.

5. **Backward-compat history.** Существующая SQLite schema `reports` не меняется. Добавляется только новая таблица `weekly_snapshots` для forecast.

### 2.3 Новые CLI-флаги

```bash
# Полный запуск (всё включено, дефолт)
python -m services.wb_localization.run_localization --cabinet ooo

# Только базовый расчёт (без прогноза/сценариев)
python -m services.wb_localization.run_localization --cabinet ooo --skip-scenarios --skip-forecast

# Forecast с кастомным реалистичным лимитом слотов
python -m services.wb_localization.run_localization --cabinet ooo --realistic-limit-pct 0.5

# Экспорт только справочника (для тестирования)
python -m services.wb_localization.run_localization --cabinet ooo --only-reference
```

1. **Справочник** — общий, один для кабинета (🆕 расширен)
2. **Сводка {cabinet}** — существующий
3. **Экономика сценариев {cabinet}** — 🆕 заменяет старую «Экономика»
4. **Перестановки Roadmap {cabinet}** — 🆕 заменяет старый список
5. **ИЛ Анализ {cabinet}** — существующий
6. **Регионы {cabinet}** — существующий
7. **Проблемные SKU {cabinet}** — существующий
8. **Допоставки {cabinet}** — существующий

## 3. Лист 1: «Справочник» (пояснительная документация)

**Замещает:** текущий `_write_reference_sheet()` (сейчас минимальная таблица КТР/КРП + формулы).

**Цель:** одна страница-инструкция. Читают один раз, потом ссылаются из других листов.

### 3.1 Структура блоков

| Блок | Строки | Содержание |
|---|---|---|
| **1. Обложка + оглавление** | 1–4 | Заголовок на тёмно-синем фоне, дата обновления коэффициентов, оглавление (гиперссылки на блоки через `=HYPERLINK`) |
| **2. Основная формула логистики** | 6–15 | Формула `Логистика = (База × Коэф.склада × ИЛ) + (Цена × ИРП%)`, компоненты подсвечены разным цветом, пример на реальных числах (цена 1000₽, объём 3л), сравнение ИЛ=1.20 vs ИЛ=0.80 |
| **3. Индекс локализации (ИЛ)** | 17–45 | Определение «локального заказа», формула взвешенного среднего, период 13 недель + обновление воскресенье→понедельник, **таблица КТР 20 строк** с градиентной заливкой (🟢 скидка / 🟡 нейтрально / 🔴 штраф), пример |
| **4. Индекс распределения продаж (ИРП)** | 47–75 | Связь с локализацией per-article, формула, **ключевой порог 60%** — резкий переход с 2.00% на 0.00%, **таблица КРП 13 строк** с цветом, пример расчёта переплаты |
| **5. Исключения** | 77–88 | КГТ, СГТ, КБТ, FBS — КРП=0%, правило 35%, список категорий |
| **6. Перестановки** | 90–130 | Опт-ин сервис, комиссия **+0.5% на все продажи**, lock-in **90 дней**, **таблица 20+ складов с лимитами/день** (Котовск 499 300, Шушары 200), лимиты общие на всех продавцов, хранение по тарифу исходного склада, **callout «Экономика перестановок»** с точкой окупаемости |
| **7. Скользящее окно 13 недель** | 132–155 | Визуальная диаграмма, формула `loc_week_t = ((13-t)×old + t×new)/13`, **таблица «недели до порога 60%»**: от 40% — 8–9 недель, от 50% — 5–6, от 55% — 2–3, призыв «начинать СЕЙЧАС, эффект отложенный» |
| **8. Наш расчёт vs WB** | 157–165 | Пометка: наш расчёт на календарных днях, WB на ISO-неделях, расхождение ≤ 3 п.п., ссылка «правда в ЛК WB» |

### 3.2 Форматирование

- Заголовок каждого блока: тёмно-синий фон, белый bold 14pt, row height 36px
- Таблицы КТР/КРП: conditional formatting по значению (градиент зелёный→жёлтый→красный)
- Формулы: monospace шрифт, серая заливка, рамка
- Примеры и выводы: жёлтые «callout» блоки с эмодзи 💡
- Freeze первой строки с оглавлением

## 4. Лист 2: «Экономика сценариев» (градация 30–90%)

**Замещает:** текущий лист `Экономика {cabinet}` (сейчас 3 сценария).

**Цель:** показать как меняется вся экономика кабинета при разных уровнях локализации, на реальных артикулах и ценах. Всё в одном листе, «кислотная» сводка + топ-артикулы + экономика перестановок.

### 4.1 Структура блоков

| Блок | Строки | Содержание |
|---|---|---|
| **1. Паспорт отчёта** | 1–10 | Дата расчёта, период, источник данных, matched/skipped артикулов, текущий ИЛ/ИРП, текущая логистика факт, ссылка на Справочник |
| **2. Сводная таблица сценариев 30–90%** | 12–28 | Главная «кислотная» таблица (см. 4.2) |
| **3. KPI-плитки** | 30–35 | Три крупные плитки: «Сейчас платите», «Макс. потенциал», «До порога 60%» |
| **4. Топ-15 артикулов** | 37–55 | Артикулы, которые тянут индекс вниз, с потенциалом экономии |
| **5. Экономика перестановок** | 57–70 | Комиссия +0.5% от оборота, точка окупаемости, чистая выгода |

### 4.2 Сводная таблица сценариев (главный блок)

**Колонки:**

| # | Колонка | Описание (под заголовком, серый italic 9pt) |
|---|---|---|
| 1 | Сценарий | Уровень локализации и оценка |
| 2 | Целевая лок.% | Уровень локализации в этом сценарии (30, 40, ..., 90) |
| 3 | КТР | Коэффициент логистики для этого уровня (из таблицы WB) |
| 4 | КРП% | Процент надбавки к цене (0% если ИЛ≥60%) |
| 5 | Логистика ₽/мес | Объёмная часть: База × Коэф.склада × КТР |
| 6 | ИРП ₽/мес | Ценовая часть: Σ(Цена × КРП%) по всем артикулам |
| 7 | Итого ₽/мес | Логистика + ИРП за месяц |
| 8 | Δ vs Сейчас | Разница с текущим состоянием (красный = переплата, зелёный = экономия) |
| 9 | Δ vs Худший (30%) | Разница с худшим сценарием |

**Строки (8):** 30%, 40%, 50%, Сейчас (anchor, жирная рамка), 60%, 70%, 80%, 90%

Цветовая логика:
- 🔴 Красный фон строки — уровень ниже текущего
- 🟡 Жёлтый — текущий
- 🟢 Зелёный — выше текущего
- Колонка «Итого ₽/мес» — heat map от красного (дорого) к зелёному (дёшево)
- Колонка «Δ vs Сейчас» — красный/зелёный по знаку

Под заголовком каждой колонки — строка с пояснением, что эта метрика означает и как считается.

**Цель:** объяснить механику перестановок + показать понедельный прогноз: что будет через N недель, если начнём перестановки сегодня с учётом реальных лимитов складов и инерции 13-недельного окна.

### 5.1 Структура блоков

| Блок | Строки | Содержание |
|---|---|---|
| **1. Как работают перестановки** | 1–15 | Краткое объяснение (со ссылкой на Справочник), ключевые цифры в плитках, реальность vs идеал |

### 5.3 Понедельный roadmap (блок 3, главный)

**Колонки:**

| # | Колонка | Описание |
|---|---|---|
| 1 | Неделя | Номер недели с начала перестановок, 0 = старт |
| 2 | Дата | Календарная дата начала недели |

### 5.6 Математика симуляции (ключевое)

**Модуль:** `calculators/relocation_forecaster.py`

**Входные данные:**
- `articles` — список артикулов с текущими метриками (из `il_irp_analyzer`)
- `movements` — рекомендованные перестановки (из `generate_movements_v3`)
- `weekly_orders_history` — история заказов per-article × week (из `history.weekly_snapshots`, fallback на текущие данные)
- `realistic_limit_pct` — параметр, % реально получаемых слотов (дефолт 0.3)
- `redistribution_limits` — дневные лимиты складов WB
- `target_localization` — целевая локализация после переноса (дефолт 85%)

**Алгоритм:**

```python
def simulate_roadmap(
    articles: list[dict],
    movements: list[dict],
    weekly_orders_history: dict,
    redistribution_limits: dict[str, int],
    realistic_limit_pct: float = 0.3,
    target_localization: float = 85.0,
) -> dict:
    # Шаг 1: недельная пропускная способность складов
    weekly_capacity = {
        wh: REDISTRIBUTION_LIMITS[wh] * 7 * realistic_limit_pct
        for wh in target_warehouses_in_movements
    }

    # Шаг 2: распределяем движения по неделям (greedy по импакту)
    schedule = schedule_movements_by_week(movements, weekly_capacity)
    # Возвращает: {week_num: [movement, ...]}

    # Шаг 3: для каждой недели (0..13) считаем «blended» локализацию
    weeks_forecast = []
    for week_num in range(0, 14):
        # Для каждого артикула определяем его локализацию на эту неделю
        blended_articles = []
        for article in articles:
            moved_to_date = sum(m.qty for m in schedule_cumulative(schedule, up_to=week_num) if m.article == article.id)
            # Доля артикула, которая «переехала»
            move_fraction = moved_to_date / article.total_stock
            # Локализация артикула сейчас (до переноса)
            loc_before = article.current_loc_pct
            # Локализация артикула после полного переноса
            loc_after = target_localization if move_fraction > 0 else loc_before

            # Эффективная локализация на этой неделе
            # (старые 13-week_num недель + новые week_num недель)
            old_weeks = 13 - week_num
            new_weeks = week_num
            # Средневзвешенно
            effective_loc = (old_weeks * loc_before + new_weeks * (loc_before * (1 - move_fraction) + loc_after * move_fraction)) / 13

            ktr, krp = get_ktr_krp(effective_loc)
            blended_articles.append({
                "article": article.id,
                "orders": article.wb_total,
                "price": article.price,
                "loc": effective_loc,
                "ktr": ktr,
                "krp": krp,
            })

        # Шаг 4: взвешенное среднее по кабинету
        total_orders = sum(a["orders"] for a in blended_articles)
        il_kabinet = sum(a["ktr"] * a["orders"] for a in blended_articles) / total_orders
        irp_kabinet = sum(a["krp"] * a["orders"] for a in blended_articles) / total_orders

        # Шаг 5: новая логистика и экономия
        new_logistics = sum(base_cost[a["article"]] * a["ktr"] for a in blended_articles) * monthly_factor
        new_irp = sum(a["price"] * a["krp"]/100 * orders_monthly[a["article"]] for a in blended_articles)
        total_new = new_logistics + new_irp

        savings = total_current - total_new

        cumulative_moved = sum(m.qty for w in range(week_num+1) for m in schedule.get(w, []))

        weeks_forecast.append({
            "week": week_num,
            "date": week_start_date(week_num),
            "moved_units_cumulative": cumulative_moved,
            "plan_pct": cumulative_moved / total_plan * 100,
            "il_forecast": il_kabinet,
            "ktr_weighted": il_kabinet,  # уже КТР
            "logistics_monthly": total_new,
            "savings_vs_current": savings,
            "milestones": detect_milestones(il_kabinet, prev_il_kabinet),  # "60% crossed", "80% reached"
        })

    return {
        "params": {...},
        "roadmap": weeks_forecast,
        "schedule": schedule,  # для детального плана
        "milestones": {
            "week_60pct": first_week_crossing(weeks_forecast, 60),
            "week_80pct": first_week_crossing(weeks_forecast, 80),
        },
    }
```

### 5.7 Ключевое уравнение «инерции» 13-недельного окна

Для каждого артикула на неделю `t`:

```
effective_loc(t) = ((13 - t) × loc_before + t × loc_new(t)) / 13
```

где `loc_new(t)` — взвешенное между `loc_before` и `loc_after` в зависимости от того, какая доля stock уже перенесена к неделе `t`.

### 5.8 Реалистичные допущения

- **Доля лимитов:** только 30% запрошенных слотов (параметр CLI)
- **Расписание:** перестановки распределяются жадно по неделям, приоритет — по impact на ИЛ
- **Локализация после переноса:** 85% (консервативная оценка — не всегда перемещение даёт 100%)
- **Старые недели тянут вниз:** 13-week_num недель продолжают работать по текущей локализации
- **Fallback при отсутствии истории:** если нет 13 недель weekly_snapshots в БД, используем упрощённую модель (равномерная текущая локализация)

### 5.9 Расширение `history.py`

Новая таблица:

```sql
CREATE TABLE IF NOT EXISTS weekly_snapshots (
    cabinet TEXT NOT NULL,
    week_start DATE NOT NULL,           -- начало ISO-недели
    article TEXT NOT NULL,
    region TEXT NOT NULL,                -- 6 FO
    local_orders INTEGER NOT NULL DEFAULT 0,
    nonlocal_orders INTEGER NOT NULL DEFAULT 0,

);
```

Заполняется при каждом запуске — разбивает период на ISO-недели. Первый запуск накапливает только текущую неделю. Через 13 недель полной истории прогноз максимально точный. До этого — fallback.

def run_for_cabinet(cabinet_key: str, days: int = 30, ...) -> dict:

## 10. Критерии успеха

1. Лист «Справочник» содержит 8 тематических блоков, визуально отформатирован, объясняет всю логику WB без необходимости читать код.
2. Лист «Экономика сценариев» показывает 7 уровней локализации (30, 40, 50, 60, 70, 80, 90) + текущий anchor, с цветовой heat-map, с топ-15 артикулов и блоком окупаемости перестановок.
3. Лист «Перестановки + Roadmap» показывает понедельный прогноз на 13 недель с автоматически подсвеченными вехами (неделя пересечения 60%, неделя достижения 80%), с графиком, с детальным планом.
4. Каждая колонка во всех трёх листах имеет строку-описание и (где применимо) cell note с формулой.
5. Модули независимы: можно вызвать `scenario_engine.analyze_scenarios(...)` или `relocation_forecaster.simulate_roadmap(...)` из любого другого сервиса или тесте.
6. Коэффициенты КТР синхронизированы с официальной таблицей WB.
7. CLI-флаги позволяют запускать любую подвыборку шагов.
8. Unit-тесты покрывают все новые калькуляторы (`scenario_engine`, `relocation_forecaster`, `reference_builder`) + синхронизацию КТР.

## 11. Нерешённые вопросы (обсудить перед планом реализации)

1. **Оборот кабинета** для расчёта комиссии 0.5% — берём из `logistics_costs` + revenue data? Или из отдельного источника? Уточнить источник истины.
2. **`target_localization` после переноса** — дефолт 85%. Возможно стоит делать per-article (если артикул в регионе N получает 90% локализации после переноса — использовать 90%, если 70% — использовать 70%). Это требует более сложной модели.
3. **Cell notes vs строки-описания** — возможно строки-описания достаточно, cell notes добавляют сложности реализации (batchUpdate notes). Рассмотреть в плане реализации.
4. **Backward-compat** — старый лист «Экономика {cabinet}» удаляется или переименовывается в «Экономика сценариев {cabinet}»? Если переименовывается — старые ссылки в Google Sheets могут сломаться.
5. **Частота обновления weekly_snapshots** — если сервис запускается ежедневно, снапшоты накапливаются быстро, но каждый день «частичная» неделя. Нужна логика аккумулирования до конца ISO-недели.
