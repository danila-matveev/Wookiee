# Reviews Audit Skill — Design Spec

## Overview

Claude Code skill (`/reviews-audit`) для глубокого анализа возвратов, отзывов, вопросов и чатов с покупателями на Wildberries. Монолитный SKILL.md с 7 фазами, выполняемыми Claude последовательно. Без Python-оркестратора.

**Контекст:** WOOKIEE — бренд женского бесшовного белья, ~400 млн руб/год, WB + Ozon. Фокус v1 — Wildberries.

## Решения

| Вопрос | Решение |
|--------|---------|
| Архитектура | Монолитный SKILL.md (подход A) |
| Публикация | Markdown + Notion |
| Скоуп данных | Отзывы + Вопросы + Чаты (без claims) |
| Источник заказов/выкупов | БД через shared/data_layer |
| Notion-интеграция | Сразу в v1 (продуктовый стратег) |

## Архитектура

```
Пользователь → /reviews-audit
       ↓
  [Фаза 1] Параметры: период, фокус (модель/все)
       ↓
  [Фаза 2] Сбор данных:
       ├── WB Feedbacks API (GET /api/v1/feedbacks, пагинация по 1000)
       ├── WB Questions API (GET /api/v1/questions, пагинация)
       ├── WB Chats API (GET /api/v1/seller/chats)
       └── БД (shared/data_layer): заказы, выкупы, возвраты по артикулам
       ↓
  [Фаза 3] Маппинг на товарную матрицу:
       └── Supabase MCP → nmId → модель → артикул → цвет/размер
       ↓
  [Фаза 4] Цифровой анализ:
       ├── % возвратов к заказам/выкупам (относительные доли)
       ├── Динамика (адаптивная: год→мес, мес→нед, нед→день)
       ├── Аномалии (рост доли > пропорции к заказам)
       └── Рейтинг: зоны + расчёт «сколько 5★ нужно для целевого»
       ↓
  [Фаза 5] Текстовый анализ + оценка коммуникации бренда:
       ├── Кластеризация по темам (по тексту, не по звёздам)
       ├── Паттерны вопросов и чатов
       ├── Оценка качества ответов бренда
       └── Cross-sell возможности
       ↓
  [Фаза 6] Продуктовый стратег:
       └── Notion MCP → «Модельный ряд» → gap-анализ:
           позиционирование vs реальная обратная связь
       ↓
  [Фаза 7] Синтез + публикация:
       ├── Markdown → docs/reports/reviews-audit-{YYYY-MM-DD}.md
       └── Notion → база «Аналитические отчёты»
```

## Фаза 1: Ввод параметров

При запуске скилл спрашивает:

1. **Период**: последняя неделя / месяц / квартал / год / кастомные даты
2. **Фокус** (опционально): конкретная модель или «все»

Авто-определение гранулярности разбивки:
- Год → помесячно (12 точек)
- Квартал → помесячно (3 точки)
- Месяц → понедельно (4-5 точек)
- Неделя → подневно (7 точек)

## Фаза 2: Сбор данных

### WB API

| Эндпоинт | Данные | Ключевые поля |
|----------|--------|---------------|
| `GET /api/v1/feedbacks` | Отзывы | text, productValuation (звёзды), createdDate, answer (ответ бренда), nmId |
| `GET /api/v1/questions` | Вопросы | text, answer, createdDate, nmId |
| `GET /api/v1/seller/chats` | Чаты | messages[], createdDate, nmId |

- Базовый URL: `https://feedbacks-api.wildberries.ru`
- Авторизация: `Authorization: {WB_API_KEY}` (токен категории «Вопросы и отзывы»)
- Rate limit: 3 req/sec, burst 6
- Пагинация: по 1000 записей, выгребаем всё за период
- Используем существующий `shared/clients/wb_client.py` (retry + 60s backoff на 429)

### БД (shared/data_layer)

SQL-запрос через `shared/data_layer` для получения:
- orders_count, buyout_count, return_count — по артикулам за период
- GROUP BY с `LOWER()` (правило проекта для артикулов)

### Промежуточное хранение

Сырые данные складываем в JSON: `/tmp/reviews_audit_raw_{date}.json`

### Маппинг nmId → товарная матрица

Через Supabase MCP (project gjvwcdtfglupewcwzfhw):
- nmId → product → artikul → model_osnova
- Получаем: название модели, артикул, цвет, размер

## Фаза 3: Маппинг на товарную матрицу

Иерархия: **ModelOsnova → Model → Artikul (цвет)**

Для каждого отзыва/вопроса/чата определяем:
- К какой модели относится (через nmId → Supabase)
- К какому артикулу (модель + цвет)
- Статус модели (активна / снята / тест)

Группируем данные на трёх уровнях для последующего анализа.

## Фаза 4: Цифровой анализ

### Метрики (всё в относительных долях)

| Метрика | Формула | Уровень |
|---------|---------|---------|
| % возвратов | returns / buyouts * 100 | модель → артикул → цвет |
| % негативных отзывов | negative_reviews / total_reviews * 100 | по тексту, не по звёздам |
| Средний рейтинг | средневзвешенный по звёздам | модель → артикул |
| Gap до целевого | (4.7 - текущий) → сколько 5★ нужно | модель |
| % проблемных тем | тема_count / total_reviews * 100 | модель → тема |
| Динамика долей | Δ% между периодами | все уровни |

### Рейтинговые зоны

- **Целевой** (>=4.7): всё хорошо
- **Приемлемый** (4.5-4.6): мониторить
- **Плохой** (<4.5): требует внимания

### Расчёт «сколько 5★ нужно для целевого 4.7»

```
// Уравнение: (sum_stars + 5 * X) / (total + X) >= 4.7
need_5star = ceil((4.7 * total - sum_stars) / (5 - 4.7))
```

### Алерты (аномалии)

Алерт срабатывает только при статистической значимости (не на единичных случаях):
- Доля проблемной темы выросла >2x между периодами
- % возвратов вырос непропорционально к росту заказов
- Рейтинг упал на >=0.2 за период
- Новая тема жалоб появилась (не было раньше, теперь >1%)

### Адаптивная разбивка по времени

- Год → 12 месяцев (тренд)
- Месяц → 4-5 недель
- Неделя → 7 дней

### Drill-down

Если на уровне модели метрика в красной зоне → автоматически раскрываем до артикула → до цвета/размера. Ищем конкретный артикул-виновник.

## Фаза 5: Текстовый анализ + оценка коммуникации бренда

### Классификация по тексту, не по звёздам

5★ может быть негативным отзывом, 3★ — позитивным. Claude анализирует содержание текста.

**Тональность:**
- Позитивный (довольны продуктом)
- Нейтральный (информационный)
- Негативный (жалоба, разочарование)
- Смешанный (5★ но с претензией, или 3★ но с похвалой)

### Кластеры проблем

Предопределённые + автоматическое выявление новых:
- Размерная сетка (маломерит / большемерит)
- Качество ткани (катышки, растяжение, выцветание)
- Цвет (не соответствует фото, выцветает при стирке)
- Упаковка / доставка
- Посадка / комфорт
- Соотношение цена/качество
- + новые кластеры при обнаружении повторяющегося паттерна

**Для каждого кластера:**
- Доля от всех отзывов (%)
- Динамика доли между периодами (рост/падение)
- Топ-артикулы внутри кластера
- 3-5 показательных цитат

### Оценка коммуникации бренда

| Критерий | Что оцениваем | Как |
|----------|--------------|-----|
| Скорость ответа | Время от отзыва/вопроса до ответа бренда | createdDate vs answer.date |
| Полнота | Ответ закрывает проблему покупателя? | Claude оценивает текст ответа |
| Тон | Эмпатия, профессионализм, без шаблонности | Claude оценивает |
| Cross-sell | Рекомендация другого артикула/размера? | Claude проверяет наличие |
| Без ответа | % отзывов/вопросов без ответа бренда | answer == null |

### Анализ чатов

- Как бренд закрывает боли покупателей
- Отработка возражений и негатива
- Cross-sell рекомендации (есть ли, уместны ли)
- Общая оценка качества поддержки

### Рекомендации

- По каждому кластеру: «как отрабатывать эту боль» (шаблон ответа)
- Cross-sell возможности: «при жалобе на размер рекомендовать артикул X»
- Паттерны для производства: «голубой цвет модели Y — проверить стойкость краски»

### Batch-обработка

При >200 отзывов — порции по 100, промежуточные кластеры мержатся.

## Фаза 6: Продуктовый стратег (Notion)

### Источник данных

Notion MCP → страница WOOKIEE → база «Модельный ряд»:
- Поле `Name` — название модели
- Поле `Позиционирование` — продуктовый смысл

### Gap-анализ: задумка vs реальность

Для каждой модели с проблемами из Фаз 4-5:
- **Задумка:** что закладывали в продукт (из Notion)
- **Реальность:** что говорят покупатели (из отзывов/вопросов/чатов)
- **Gap:** конкретное расхождение → рекомендация

Пример:
- Задумка: «комфорт на каждый день, мягкая ткань, универсальный размер»
- Реальность: «ткань жёсткая после стирки, маломерит»
- Gap: несоответствие по 2 из 3 ключевых обещаний → пересмотреть ткань/лекала

## Фаза 7: Синтез + публикация

### Формат отчёта

Claude формирует формат сам, но обязательно включает:

1. **Executive Summary** — 3-5 ключевых выводов с цифрами
2. **Рейтинговая карта моделей** — таблица: модель / рейтинг / зона / trend / топ-проблема
3. **Детальный анализ проблемных моделей** — drill-down до артикула/цвета
4. **Кластеры проблем** — с долями, динамикой, цитатами
5. **Оценка коммуникации бренда** — скорость, полнота, tone, % без ответа
6. **Gap-анализ: позиционирование vs восприятие** — по данным из Notion
7. **Actionable рекомендации** — ранжированные по критичности, с «что если» сценариями
8. **Расчёт: сколько 5★ нужно** — для моделей в красной/жёлтой зоне

### Ревью-цикл

После генерации Claude сам проверяет:
- Все цифры имеют контекст (% от чего?)
- Нет алертов на единичных случаях
- Рекомендации конкретные, а не общие фразы
- Динамика учитывает пропорции к объёму заказов

### Публикация

- Markdown → `docs/reports/reviews-audit-{YYYY-MM-DD}.md`
- Notion → база «Аналитические отчёты» (существующий паттерн)

## Принципы анализа

1. **Доли, а не абсолюты.** Все метрики — относительные. Рост возвратов пропорционально заказам — норма, не алерт.
2. **Текст > звёзды.** 5★ с претензией = негативный. 3★ с похвалой = позитивный.
3. **Пороги значимости.** Не цепляться за единичные случаи. 1 жалоба из 1000 = шум.
4. **Динамика долей.** Было 1% маломерит → стало 3% = алерт. Было 1% → осталось 1% = норма.
5. **Drill-down при проблеме.** Модель → артикул → цвет. Найти конкретного виновника.
6. **Actionable рекомендации.** Не «улучшите качество», а «проверить стойкость краски голубого цвета модели Wendy на фабрике X».

## Зависимости

- `shared/clients/wb_client.py` — HTTP-клиент WB API (httpx, retry, rate-limiting)
- `shared/data_layer/` — SQL-запросы к БД подрядчика (заказы/выкупы/возвраты)
- Supabase MCP — товарная матрица (nmId → модель/артикул/цвет)
- Notion MCP — база «Модельный ряд» (позиционирование моделей)
- Notion MCP — база «Аналитические отчёты» (публикация)

## Ограничения v1

- Только Wildberries (Ozon — в v2)
- Без claims (заявок на возврат) — добавим позже
- Размеры не маппятся в текущей товарной матрице (только цвет как вариант)
