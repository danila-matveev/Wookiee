# WB Logistics Toolkit — Design Spec

**Date:** 2026-05-06  
**Status:** Approved  
**Audience:** Technical and non-technical  

---

## Зачем этот инструмент

Wildberries берёт деньги за логистику по тарифу с повышающим коэффициентом (КТР), который зависит от того, насколько ваш товар лежит «рядом» с покупателями. Чем ниже локализация — тем больше платите. Сверх того, при локализации ниже 60% WB берёт ещё и процент с продажной цены каждого товара (ИРП). В сумме это может удваивать стоимость логистики.

**Toolkit решает две задачи:**

1. **Оптимизатор локализации** — показывает текущее состояние, рассчитывает сколько вы переплачиваете, предлагает что и куда переложить, чтобы снизить расходы.

2. **Аудит переплат** — пересчитывает вашу историческую логистику по официальным тарифам WB и показывает расхождение. Используется для претензий и возврата денег через суд.

## Конфигурация

### `cabinets.yaml`
```yaml
cabinets:
  - name: ooo
    sheet_id: "1TMadxTXPYnuGTvnMZT9Ny7eL4KbvsjK2XsuRjpRmnM0"
  - name: ip
    sheet_id: "1AbCde..."
```

Токены WB хранятся только в `.env` по именам кабинетов: `WB_TOKEN_OOO`, `WB_TOKEN_IP`. `shared/config.py` подставляет нужный токен по полю `name`. `cabinets.yaml` содержит только несекретные данные — его можно коммитить.

### `warehouse_status.yaml`
```yaml
warehouses:
  - name: Новосибирск
    fd: Дальневосточный + Сибирский
    available: false
    reason: "Закрыт WB для поставок СФО (май 2026)"
  - name: Казань
    fd: Приволжский
    available: true
    note: "Дорогой склад (225%), но высокая локализация ПФО ~95%"
  - name: Коледино
    fd: Центральный
    available: true
```

Алгоритм перестановок читает этот файл и исключает недоступные склады из рекомендаций. Это **конфигурационный файл**, не аналитические данные — ручное обновление при изменениях от WB корректно. Все расчётные показатели (ДЛ, ИЛ, ИРП) по-прежнему считаются только из API.

### `.env.example`
```bash
# Токены кабинетов WB (имя совпадает с name в cabinets.yaml, uppercase)
WB_TOKEN_OOO=eyJ...
WB_TOKEN_IP=eyJ...

# ID Google Sheets для каждого кабинета
CABINET_OOO_SHEET_ID=1TMadx...
CABINET_IP_SHEET_ID=1AbCde...

# Google Service Account
GOOGLE_CREDENTIALS_PATH=credentials.json

# Supabase (история тарифов и таблица КТР/КРП)
SUPABASE_URL=https://xxx.supabase.co
SUPABASE_KEY=eyJ...
```

`credentials.json` добавлен в `.gitignore`. `check_setup.py` проверяет его наличие и предупреждает если файл случайно в staging.

```
    - warehouse_status.yaml (исключает закрытые склады)
    - лимиты перераспределения (из Supabase wb_tariffs)
    - защита доноров (ДЛ донора не ниже 70%)
      ↓
  Sheets: Перемещения, Допоставки, Сводка, Обновление
```

### Ключевые концепции расчёта ДЛ/ИЛ/ИРП

**Доля Локализации (ДЛ)** — процент заказов по артикулу, где склад отгрузки и адрес доставки в одном ФО. Считается по-артикульно, не по кабинету.

**ИЛ (КТР)** — средневзвешенный КТР по всем артикулам:
```
ИЛ = sum(КТР_артикула × заказы_артикула) / sum(заказы)
```

**ИРП (КРП)** — средневзвешенный КРП% по всем артикулам:
```
ИРП = sum(КРП%_артикула × заказы_артикула) / (заказы_РФ + заказы_СНГ)
```

При одинаковом ИЛ разные ИРП возможны — таблицы преобразования ДЛ→КТР и ДЛ→КРП независимы и имеют разную «ступенчатость».

**Таблица КТР/КРП** загружается из Supabase таблицы `wb_coeff_table` при старте. При изменении тарифов WB — обновляется в Supabase без изменения кода.

| ДЛ артикула | КТР  | КРП%  |
|-------------|------|-------|
| ≥ 95%       | 0.50 | 0     |
| 90–94.99%   | 0.60 | 0     |
| 85–89.99%   | 0.70 | 0     |
| 80–84.99%   | 0.80 | 0     |
| 75–79.99%   | 0.90 | 0     |
| 70–74.99%   | 1.00 | 0     |
| 60–69.99%   | 1.00 | 0     |
| **< 60% — ИРП-зона** |||
| 55–59.99%   | 1.05 | 2.00  |
| 50–54.99%   | 1.10 | 2.05  |
| ...         | ...  | ...   |
| 0–4.99%     | 2.20 | 2.50  |

**Цель:** ДЛ ≥ 60% по каждому артикулу → ИРП = 0. ДЛ ≥ 80% → КТР = 0.80 (скидка 20% на логистику).

### 13-недельное окно

WB считает ИЛ как скользящее среднее за 13 недель. Это означает: если сегодня переложить товар и достичь ДЛ = 100%, ИЛ не изменится мгновенно. Каждую неделю новые данные «вытесняют» старые:

```
Неделя t: ДЛ_эффективная = ((13-t) × ДЛ_до + t × ДЛ_после) / 13
```

Пример: сейчас ДЛ = 50%. Переложили в неделю 0 → ДЛ стала 85%.
- Неделя 4: ДЛ_эффективная = (9×50 + 4×85) / 13 = 60.8% — ИРП только что исчез
- Неделя 7: ДЛ_эффективная = 72.7% — подходим к КТР 1.0
- Неделя 13: ДЛ_эффективная = 85% — полный эффект

### Алгоритм перестановок

`permutation_calculator.py` — полная переработка `generate_localization_report_v3.py` под WB API:

1. Берёт данные из WB API: остатки по складам (`warehouse_remains`), заказы (`supplier/orders`)
2. Для каждого артикула считает текущую ДЛ и потенциал роста
3. Ранжирует по вкладу в улучшение ИЛ: `(КТР_текущий - КТР_целевой) × заказы`
4. Исключает склады с `available: false` из `warehouse_status.yaml`
5. Защищает доноров: склад-донор не опускается ниже 70% ДЛ
6. Жадный алгоритм распределяет объёмы с учётом дневных лимитов складов

## Воркфлоу Инструмента 2 — Аудит переплат

```
python audit/run_audit.py --cabinet ooo --from 2026-01-01 --to 2026-03-31

  WB API: reportDetailByPeriod v5 (отчёт за период)
  WB API: content (объёмы товаров)
  Supabase: wb_tariffs (исторические коэффициенты складов по датам)
      ↓
  warehouse_coef_resolver: 3-tier приоритет:
    1. Fixation (если активна на дату)
    2. Supabase wb_tariffs (историческая база)
    3. dlv_prc из отчёта (fallback)
      ↓
  logistics_overpayment: пересчёт каждой строки по официальной формуле
      ↓
  Excel (11 листов + лист Рекомендации)
```

### Лист Рекомендации (новый)

Итоговый лист для подготовки претензии или судебного иска:

- Суммарная переплата за период (₽)
- Топ-10 артикулов по сумме переплаты
- Топ-5 складов с наибольшими расхождениями
- Динамика переплат по месяцам (тренд)
- Шаблон формулировки претензии с подстановкой цифр

### ETL тарифов

```
python audit/etl/tariff_collector.py --date 2026-05-06

  WB API: /api/v1/tariffs/box (текущие коэффициенты складов)
      ↓
  Supabase: upsert в wb_tariffs (warehouse_name, date, delivery_coef_pct)
```

Запускается ежедневно (cron). Перед запуском аудита `run_audit.py` проверяет дату последнего обновления и предупреждает если тарифы старше 2 дней.

## Supabase — схема таблиц

### `wb_tariffs` — история коэффициентов складов

| Поле | Тип | Описание |
|------|-----|----------|
| id | uuid | PK |
| warehouse_name | text | Название склада |
| date | date | Дата тарифа |
| delivery_coef_pct | numeric | Коэффициент доставки % |
| box_delivery_base | numeric | Базовая ставка (₽/л) |
| box_delivery_liter | numeric | Ставка за доп. литр |
| created_at | timestamptz | Дата загрузки |

### `wb_coeff_table` — таблица КТР/КРП

| Поле | Тип | Описание |
|------|-----|----------|
| id | int | PK |
| min_loc | numeric | Минимальная ДЛ % |
| max_loc | numeric | Максимальная ДЛ % |
| ktr | numeric | Коэффициент транспортировки |
| krp_pct | numeric | КРП в % от цены |
| valid_from | date | С какой даты действует |

При старте `shared/coeff_table.py` загружает актуальную таблицу (last valid_from ≤ today).

Первоначальное наполнение: скрипт `audit/etl/import_coeff_table.py` импортирует таблицу из официальной документации WB Partners. При изменении тарифов WB — добавляется новая строка с новым `valid_from`, старые строки не трогаются (история сохраняется).

## Безопасность и надёжность

- `credentials.json` — в `.gitignore`, `check_setup.py` проверяет что он не в staging
- `.env` — в `.gitignore`
- Все денежные расчёты — через `Decimal` с `ROUND_HALF_UP`
- Перед запуском аудита — проверка даты последнего ETL
- `warehouse_status.yaml` — явные флаги `available: false` исключают закрытые склады
- `il_overrides.json` — **удалён**. ИЛ считается только из данных WB API

## Ограничения и допущения

| Ограничение | Описание |
|-------------|----------|
| Расхождение ≤ 3 п.п. | Наш расчёт ДЛ по календарным дням, WB — по ISO-неделям |
| Сценарии — упрощение | Сценарный движок применяет один КТР ко всем артикулам кабинета |
| Лаг данных WB API | orders endpoint обновляется с задержкой ~1-2 дня |
| Одобрение перестановок | Автоматического механизма нет — это расчёт, решение принимает человек |
| Сезонность | `weekly_orders_history` в forecaster зарезервирован, сезонность пока не учитывается |
