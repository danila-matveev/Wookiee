## Как это работает: четыре шага

### Шаг 1 — Сбор данных (параллельно)

Запускаются пять независимых сборщиков одновременно. Каждый обращается к своему источнику и сохраняет результат в виде файла:

```
collectors/finance.py    →  /tmp/coo_finance.json
collectors/models.py     →  /tmp/coo_models.json
collectors/logistics.py  →  /tmp/coo_logistics.json
collectors/ads.py        →  /tmp/coo_ads.json
collectors/team.py       →  /tmp/coo_team.json
```

Параллельный запуск важен: каждый сборщик независим, падение одного не останавливает остальные.

### Шаг 2 — Проверка данных

Перед публикацией каждый файл проверяется по жёстким правилам. Есть два уровня реакции:

**Стоп (публикация не происходит):**
- Заказы за неделю равны нулю — данные не пришли
- Доля рекламных расходов выше 100%
- Маржа хуже −50%

**Предупреждение (публикуем, но ставим флаг ⚠️ в ячейке):**
- Выручка по модели выросла больше чем в три раза за неделю
- Оборачиваемость по складу больше 180 дней
- Индекс локализации WB ниже 30%
- Стоимость привлечения покупателя выше 5 000 ₽
- Данные Битрикс старше 24 часов

Если сработал «Стоп» — Claude показывает, что именно сломано, и ждёт решения.

### Шаг 3 — Заполнение отчёта

Claude читает все пять файлов и заполняет разделы:

- Раздел 1: подставляет числа в таблицу P&L, две недели рядом для сравнения
- Раздел 6: по каждой из 16 моделей — выручка, доля рекламы, маржа, тренд (рост/падение к прошлой неделе)
- Раздел 7: по каждому рекламному каналу — расходы, доля, стоимость привлечения
- Раздел 8: индекс локализации, оборачиваемость по каждой модели и цвету
- Раздел 9: по каждому сотруднику — сколько задач выполнено, в работе, просрочено; предварительная оценка 🔴/🟡/🟢
- Разделы 2–3: предлагает формулировки проблем и решений на основе всех собранных данных

### Шаг 4 — Создание страницы в Notion

1. Берём шаблон Елизаветы (`35658a2bd5878028ad75f1773a0f8593`), создаём копию
2. Переименовываем: `Отчётность COO 28 апр — 4 мая 2026` (даты считаются автоматически)
3. Кладём рядом с предыдущими отчётами
4. Вставляем данные в таблицы
5. Разделы 4, 5, 10, 11, 12 остаются пустыми — с подсказкой «заполнить вручную»
6. Выводим ссылку на готовую страницу

## Откуда берём каждую цифру

### Финансы (раздел 1)

**Источник:** база данных PostgreSQL через `shared/data_layer.py`

| Показатель | Откуда |
|---|---|
| Заказы до СПП, ₽ и шт | Таблицы заказов WB + OZON |
| Выкуп %, отмены %, возвраты % | Рассчитываем из продаж и заказов |
| Выручка после СПП | Таблицы продаж с вычетом скидки покупателя |
| Комиссия, логистика, хранение, эквайринг, приёмка | Таблицы расходов площадки |
| Реклама (расходы) | Таблицы рекламных кампаний |
| Маржинальная прибыль | Выручка − все расходы площадки |
| Средний чек | Выручка ÷ количество продаж |
| Доля рекламных расходов (внутренняя / внешняя) | Расходы реклама ÷ заказы в рублях |

Период: текущая неделя (пн–вс) и предыдущая неделя рядом.

### Оборачиваемость и склад (раздел 1 и 8)

**Источник:** WB API + МойСклад API + PostgreSQL

| Показатель | Откуда |
|---|---|
| Остатки FBO, шт | WB API (склады WB) |
| Остатки FBO, ₽ | Остатки × себестоимость из базы |
| МойСклад, шт и ₽ | МойСклад API |
| Заморожено, ₽ | Остатки в пути × себестоимость |
| Оборачиваемость, дни | `get_wb_turnover_by_model()` из `data_layer/inventory.py` |
| Индекс локализации WB | `services/wb_localization/run_localization.py` |
| GMROI | (Маржа за неделю × 52) ÷ средние остатки по себестоимости |
| ROI | Маржа ÷ себестоимость остатков × 100 |

### Комплекты белья (раздел 6)

**Источник:** PostgreSQL, группировка по `LOWER(SPLIT_PART(article, '/', 1))`

16 моделей: Wendy, Vuki, Ruby, Audrey, Charlotte, Moon, Set Vuki, Joy, Set Moon, Lana, Eva, Set Ruby, Bella, Set Bella, Alice, Valery

| Маржа, % | (Выручка − расходы площадки) ÷ выручка |
| Тренд | Сравнение выручки с прошлой неделей |

## Что нужно из конфигурации (`.env`)

Все ключи уже есть в проекте, новых не требуется:
- `NOTION_TOKEN` — для создания страницы
- `WB_API_KEY_OOO`, `WB_API_KEY_IP` — для остатков и локализации
- `MOYSKLAD_TOKEN` — для остатков МойСклад
- `Bitrix_rest_api` — для задач команды
- PostgreSQL — `DB_HOST`, `DB_PORT`, `DB_USER`, `DB_PASSWORD`, `DB_NAME`

## Ограничения и риски

| Риск | Как снижаем |
|---|---|
| WB API не отвечает | Сборщик возвращает ошибку, Claude флагирует секцию, остальные данные публикуются |
| Данные Битрикс устарели | Проверяем возраст файла, перезапускаем fetch_data.py |
| Новая модель появилась в ассортименте | config.py содержит список моделей — обновить вручную при найме |
| Шаблон Notion изменился | ID шаблона в config.py — обновить вручную |
| Данные за неполную неделю (запуск в среду) | Считаем по фактическим дням, помечаем «неделя ещё не закончена» |
