# ABC-аудит товарной матрицы — Дизайн скилла

**Дата:** 11 апреля 2026
**Статус:** Draft
**Скилл:** `/abc-audit`

---

## 1. Цель

Глубокий товарный ABC-анализ бренда Wookiee (WB + OZON) с рекомендациями по каждому артикулу: наращивать, поддерживать, оптимизировать, выводить. Анализ учитывает производственные ограничения (MOQ, color_code связки в коллекциях) и строится командой субагентов с перекрёстной проверкой.

---

## 2. Вызов

```
/abc-audit              # по состоянию на сегодня
/abc-audit 2026-04-01   # по состоянию на конкретную дату
```

Пользователь задаёт только дату отсечки. Периоды анализа (30/90/180 дней) скилл определяет сам.

---

## 3. Аналитическая модель

### 3.1 Двухосевая классификация

**Ось 1 — ABC по абсолютной маржинальной прибыли (руб).**
Стандартный Парето по кумулятивной доле маржи:
- **A** = артикулы, дающие 80% суммарной маржи
- **B** = следующие 15%
- **C** = оставшиеся 5%

**Ось 2 — Годовая доходность на вложенный капитал (ROI).**
```
ROI = маржинальность% × (365 / оборачиваемость_дней)
```
Зоны:
- **Высокий ROI** (>150%) — капитал крутится быстро и прибыльно
- **Средний ROI** (50-150%) — нормальная эффективность
- **Низкий ROI** (<50%) — капитал заморожен

### 3.2 Матрица решений (3×3)

| | Высокий ROI (>150%) | Средний ROI (50-150%) | Низкий ROI (<50%) |
|---|---|---|---|
| **A** (80% маржи) | ЗВЕЗДА → наращивать производство | РАБОЧАЯ ЛОШАДКА → оптимизировать оборачиваемость | ТЯЖЕЛОВЕС → срочно работать с ценой/стоками |
| **B** (15% маржи) | ПОТЕНЦИАЛ → усилить маркетинг | СЕРЕДНЯК → мониторить | КАНДИДАТ НА ВЫВОД → снижать остатки |
| **C** (5% маржи) | НИШЕВЫЙ → оставить, не инвестировать | БАЛЛАСТ → готовить к выводу | МЁРТВЫЙ ГРУЗ → выводить немедленно |

### 3.3 Расчёт оборачиваемости

Оборачиваемость считается по **всем товарным остаткам**:
- WB склады (FBO)
- OZON склады
- МойСклад (основной склад + товары в пути)

```
оборачиваемость_дней = суммарный_сток_всех_складов / среднедневные_продажи
```

Среднедневные продажи = сумма продаж (WB + OZON) за период / количество дней.

**Привязка к MOQ:** для каждого артикула считаем:
```
месяцев_на_MOQ = 500 / (среднедневные_продажи × 30)
```

Пороги оборачиваемости по MOQ:
- **Отлично** (<1 мес): 500 шт продаётся за месяц или быстрее
- **Нормально** (1-3 мес): 500 шт за 1-3 месяца
- **Приемлемо** (3-6 мес): допустимо, но нужен мониторинг
- **Плохо** (6-12 мес): снижать остатки, работать с ценой/маркетингом
- **Лимит** (>12 мес): заказывать невыгодно — капитал замораживается на год+. Выводить или переходить на заказ раз в год.

### 3.4 Адаптивная глубина анализа

Скилл автоматически определяет горизонт по каждому артикулу:

1. **Базовый слой — 30 дней.** Основной период классификации.
2. **Контрольный слой — 90 дней.** Проверка стабильности. Если 30-дневная и 90-дневная классификации расходятся — confidence снижается.
3. **Углублённый слой — 180 дней.** Только для спорных кейсов: расхождение между 30д и 90д, или товар на границе категорий.
4. **Сезонность.** Если есть данные за аналогичный период прошлого года — сравнение для выявления сезонных паттернов (летние/зимние цвета).

**Критерий достаточности данных:** менее 30 штук продаж за период → расширить горизонт. Data Analyst принимает это решение на Stage 2.

## 4. Товарная иерархия и коллекции

### 4.1 Иерархия Supabase

```
modeli_osnova (25 шт) → tip_kollekcii (tricot / seamless_wendy / seamless_audrey)
  └── modeli (58 шт) → по юрлицам (ИП / ООО): Vuki + Vuki2, Moon + Moon2
        └── artikuly (548 шт) → модель + цвет
              └── tovary (1473 шт) → артикул + размер + баркод
```

### 4.2 Статусы (таблица `statusy`)

| ID | Статус | Обработка в анализе |
|---|---|---|
| 1 | Продается | Основной анализ: матрица ABC × ROI |
| 2 | Выводим | Отдельный анализ: поиск товаров для возврата в «Продается» |
| 6 | Новый | Информационный блок, без решений |
| 7 | Запуск | Информационный блок, без решений |
| 3 | Архив | Исключается полностью |
| 4 | Подготовка | Исключается |
| 5 | План | Исключается |

### 4.3 Color_code слои в трикотажной коллекции

В трикотажной коллекции (`tip_kollekcii = 'tricot'`) существуют **два независимых color_code слоя**:

**Слой «Комплекты белья»:**
Модели: Vuki, Vuki2, Moon, Moon2, Ruby, Joy
Color_code примеры: "2" (чёрный), "7" (бежевый), "w3" и т.д.
Производственное ограничение: если производим цвет — производим для ВСЕХ моделей в слое.

**Слой «Сеты трусов»:**
Модели: Set Vuki, Set Moon, Set Ruby
Color_code примеры: "222" (три чёрных трусов)
Производственное ограничение: аналогичное — если цвет производим, то для всех сетов.

Решения по слоям **независимы**: вывод color_code "2" у комплектов не означает вывод "222" у сетов.

### 4.4 Бесшовная и другие модели

Модели Wendy, Audrey, Eva, Charlotte, Bella, Lana, Mia и др. — **без color_code привязки**. Каждый артикул (модель + цвет) анализируется независимо. Решение по одному цвету не влияет на другие цвета той же модели.

### 4.5 MOQ

MOQ = 500 штук на цвет — универсально для всех коллекций и моделей. В будущем значение будет в Supabase; сейчас хардкод.

## 5. Архитектура субагентов

### 5.0 Stage 0: Парсинг аргументов

Вход: дата отсечки (по умолчанию — сегодня).
Вычисляем:
- `CUT_DATE` — дата отсечки
- `P30_START` = CUT_DATE - 30 дней
- `P90_START` = CUT_DATE - 90 дней
- `P180_START` = CUT_DATE - 180 дней
- `YEAR_AGO_START` / `YEAR_AGO_END` — аналогичный период прошлого года (для сезонности)

### 5.1 Stage 1: Collector (Python-скрипт, ~30 сек)

Скрипт: `scripts/abc_audit/collect_data.py`

Собирает из БД и Supabase:
- WB + OZON финансы по артикулам за 30/90/180 дней (маржа, выручка, себестоимость, логистика, хранение, комиссия, реклама)
- WB + OZON продажи по артикулам (шт + руб) за те же периоды
- Стоки: WB (FBO) + OZON + МойСклад (основной + в пути)
- Supabase: полная иерархия (modeli_osnova → modeli → artikuly → tovary), статусы, коллекции, color_code, tip_kollekcii, категории
- Данные за аналогичный период прошлого года (если есть) — для сезонности
- ДРР по артикулам (внутренняя + внешняя реклама)
- Данные по размерам (продажи по баркодам → привязка к размерам)

Выход: `/tmp/abc-audit-{CUT_DATE}.json`

Никакого AI на этом этапе — чистый SQL + Supabase API.

### 5.2 Stage 2: Data Analyst (один агент, ~5 мин)

**Роль:** Подготовить и проверить данные. Gate перед экспертным советом.

Задачи:
1. **Проверка целостности** — все артикулы из Supabase имеют данные по продажам? Пропуски в стоках? Артикулы без статуса?
2. **Контроль полноты** — сверка: Supabase артикулы vs артикулы в данных продаж. Таблица: статус → кол-во в Supabase → кол-во в анализе → исключено → причина.
3. **Определение глубины по артикулу** — если <30 продаж за 30 дней → расширить до 90д; если <30 за 90д → до 180д. Записывает `depth` для каждого артикула.
4. **Расчёт базовых метрик** по каждому артикулу:
   - Выручка (руб/период) — доля в общем объёме (%)
   - Маржинальная прибыль (руб/период) — доля в общей марже (%)
   - Маржинальность (%)
   - Оборачиваемость (дней) — по всем складам
   - ROI годовой
   - ДРР (% — внутренняя + внешняя, раздельно)
   - Выкуп % — доля выкупленных заказов (влияет на обратную логистику и реальную маржу)
   - Среднедневные продажи (шт)
   - Месяцев на MOQ (500 / среднедневные_продажи / 30)
   - Тренд: сравнение 30д vs 90д (рост/падение/стабильно)
   - Сезонность: сравнение с аналогичным периодом прошлого года (если данные есть). Если текущий период — низкий сезон для цвета, пометка `🔄 Сезонный` — защита от ошибочного вывода
5. **ABC-классификация** — Парето по марже, зоны ROI, присвоение ячейки матрицы.
6. **Агрегация по color_code** — для трикотажной коллекции: суммируем метрики по всем моделям внутри color_code (отдельно для комплектов и сетов).
7. **Quality_flags** — список проблем качества данных (на русском).

**Gate:** Если критические проблемы (>30% артикулов без данных, стоки устарели >7 дней) → СТОП.

Выход: обогащённый JSON с метриками, ABC-категориями, depth, quality_flags.

### 5.3 Stage 3: Экспертный совет (3 агента параллельно, ~8-10 мин)

Все три получают одинаковый обогащённый JSON из Stage 2.

#### Товаровед

Экспертиза: товарная иерархия, коллекции, производственные ограничения.

- Валидирует иерархию: правильно ли агрегированы color_code? Учтены ли пары ИП/ООО (Vuki+Vuki2, Moon+Moon2)?
- По каждому color_code в трикотажной (оба слоя): агрегатная оценка + drill-down по моделям с бенчмарком (медиана коллекции).
- Слабые модели внутри сильного color_code: не "выводить", а "что с ней не так?" (контент, маркетинг, цена).
- По бесшовной и другим: артикульный анализ без привязки к color_code.
- Статусы: "Продается" → основной анализ; "Выводим" → поиск кандидатов на возврат; "Новинка/Запуск" → информация.
- MOQ-анализ: при текущих продажах за сколько месяцев продадим 500 шт? Выгодно ли заказывать?
- **Возможности расширения матрицы:**
  - По цветам: модель с 2-3 цветами и сильными метриками → рекомендовать новые цвета.
  - По размерам: анализ распределения продаж по размерам. Если крайний размер (L) продаётся >50% от соседнего (M) → рекомендовать добавить XL. С оговоркой о влиянии на % выкупа.

#### Финансист

Экспертиза: юнит-экономика, ценообразование, эффективность капитала.

- Матрица 3×3 (ABC × ROI) — распределение артикулов по ячейкам, суммы маржи, количество.
- Юнит-экономика по топ-моделям: цена → себестоимость → логистика → хранение → комиссия → маржа.
- Выручка по артикулам: доля в общем объёме, сопоставление с долей маржи. Высокая выручка + низкая маржа = неэффективное ценообразование.
- Ценовые аномалии: маржинальность <15% у "Продается" → флаг.
- Выкуп %: низкий выкуп (<70%) → повышенная обратная логистика → занижение реальной маржи. Причины: размерная сетка, качество фото, несоответствие ожиданиям.
- Сравнение ИП vs ООО: одна и та же модель (Vuki vs Vuki2). **Разные стратегические цели:** ИП = высокий объём + приемлемая маржа (массовый сегмент); ООО = меньший объём + высокая маржа + лучшая упаковка (премиум). Оценка: достигает ли каждое юрлицо своей цели, а не "кто эффективнее" в целом.
- "Что если" сценарии: если поднять/снизить цену на 5-10% у конкретного товара → эффект на маржу.
- Замороженный капитал: сколько денег заморожено в стоках C-товаров с низким ROI?
- Отдельная матрица для товаров в статусе "Выводим".

#### Маркетолог

Экспертиза: реклама, конверсия, продвижение.

- ДРР по артикулам: переинвестированные (ДРР >15%) и недоинвестированные (ДРР <3% при хорошей марже).
- Связка реклама → заказы: ДРР растёт — растут ли заказы?
- Конверсионные аномалии: если есть трафик, но низкая конверсия → проблема карточки/контента.
- Рекомендации по перераспределению маркетинг-бюджета: снять с C-товаров, добавить на A-товары с потенциалом.
- Анализ "недоинвестированных звёзд": A-товары с низким ДРР — можно ли ускорить продажи рекламой?

Выход каждого: структурированные findings + recommendations + confidence (HIGH/MEDIUM/LOW).

### 5.4 Stage 4: Арбитр (один агент, ~5-8 мин)

**Роль:** CEO / совет директоров. Финальное решение по каждому артикулу.

Задачи:
1. **Поиск противоречий** между экспертами. Пример: Финансист — "выводить Audrey/beige" (низкий ROI), Маркетолог — "недоинвестирован, дать рекламу". Арбитр разбирает, принимает решение с обоснованием.
2. **Проверка коллекционных ограничений** — если Финансист рекомендует вывести цвет в трикотажной коллекции, Арбитр проверяет агрегат по color_code.
3. **Confidence** по каждому решению:
   - **HIGH** (>80%): данных достаточно (>30 продаж), 30д и 90д классификации совпадают, эксперты согласны.
   - **MEDIUM** (60-80%): есть расхождения между периодами или между экспертами.
   - **LOW** (<60%): мало данных, сильная сезонность, эксперты противоречат друг другу.
4. **Вердикт по каждому артикулу / color_code:**
   - Действие: НАРАЩИВАТЬ / ПОДДЕРЖИВАТЬ / ОПТИМИЗИРОВАТЬ / ГОТОВИТЬ К ВЫВОДУ / ВЫВОДИТЬ НЕМЕДЛЕННО
   - Обоснование: 2-3 предложения с цифрами
   - Конкретные шаги: "поднять цену на X%", "увеличить ДРР до Y%", "проверить контент"
   - Confidence + что нужно для повышения confidence
5. **Приоритизация:**
   - 🔴 P0 — действовать сегодня (OOS на ЗВЕЗДЕ, убыточный товар с большими стоками)
   - 🟡 P1 — в течение недели
   - 🟢 P2 — в течение месяца

### 5.5 Протокол перепроверки Арбитра

Арбитр выносит один из трёх вердиктов:

**APPROVE** — всё корректно → Stage 5.

**CORRECT** — мелкие несоответствия, Арбитр исправляет сам без пересчёта → Stage 5.

**REJECT** — критическая ошибка → перезапуск минимального Stage.

Правила:
- Максимум **1 REJECT** за весь цикл.
- REJECT содержит: что не так, какой Stage перезапустить, что пересчитать.
- Перезапускается минимальный необходимый Stage (один эксперт, не все три).
- После повторного прохода: если всё ещё проблема → Арбитр помечает конкретные артикулы/блоки как `⚠️ Данные ненадёжны: [причина]`, confidence = LOW, и идём дальше. Бесконечного цикла нет.

### 5.6 Stage 5: Synthesizer (один агент, ~5 мин)

Собирает всё в MD-отчёт. Чистая вёрстка по правилам форматирования (секция 7). Без собственной аналитики — только структурирование результатов Арбитра.

## 6. Структура выходного отчёта (13 секций)

### I. Паспорт отчёта

Период, каналы, полнота данных, quality_flags на русском.

Обязательная таблица контроля полноты:
```
| Статус | В Supabase | В анализе | Исключено | Причина |
|---|---|---|---|---|
| Продается | 142 | 142 | 0 | — |
| Выводим | 38 | 38 | 0 | — |
| Новинка/Запуск | 12 | 12 | 0 | Информационный блок |
| Архив | 356 | 0 | 356 | Не активны |
| Итого | 548 | 192 | 356 | |
```

### II. Топ-выводы и действия

Все значимые находки (без искусственного лимита). Каждая с цифрами и причинно-следственной цепочкой.

### III. Матрица ABC × ROI — «Продается»

Сводная таблица 3×3: количество артикулов, сумма маржи, эмодзи-маркеры. Под таблицей — расшифровка каждой ячейки со списком артикулов и ключевыми метриками.

В матрицу попадают **только артикулы в статусе "Продается"**.

### IV. Трикотажная коллекция (анализ по color_code)

Два подраздела:
- **IV.1 Комплекты белья** (Vuki, Vuki2, Moon, Moon2, Ruby, Joy) — анализ по color_code с агрегатом + drill-down по каждой модели + бенчмарк (медиана коллекции).
- **IV.2 Сеты трусов** (Set Vuki, Set Moon, Set Ruby) — отдельный color_code слой, аналогичная логика.

Для слабых моделей внутри сильного color_code — конкретные гипотезы (контент, маркетинг, цена).

### V. Бесшовная и другие модели (артикульный анализ)

Каждый артикул независимо. Группировка по моделям для удобства чтения. Ключевые метрики + действие + confidence.

### VI. Товары к выводу

Конкретный список артикулов с вердиктом ВЫВОДИТЬ НЕМЕДЛЕННО или ГОТОВИТЬ К ВЫВОДУ. Обоснование с цифрами: ROI, оборачиваемость, месяцев на MOQ, замороженный капитал. Рекомендация по стратегии вывода:
- Плавное снижение цены (5-10% в неделю) — если остатки небольшие
- Агрессивная распродажа (-20-30%) — если остатки значительные
- Продажа в убыток / безубыток — допустимо, если цель — быстро высвободить капитал для A-товаров. Расчёт: убыток от распродажи vs стоимость замороженного капитала (хранение + упущенная выгода)

**Защита от сезонной ошибки:** если артикул помечен `🔄 Сезонный` и текущий период — его низкий сезон, Арбитр не рекомендует вывод, а помечает "пересмотреть через N месяцев (начало высокого сезона)".

### VII. Товары к росту

Артикулы с вердиктом НАРАЩИВАТЬ. Текущие стоки, оборачиваемость, прогноз OOS, рекомендуемый объём заказа. Оценка эффекта в руб/мес при бесперебойных поставках.

### VIII. Выводимые товары — пересмотр

Отдельная матрица ABC × ROI для артикулов в статусе "Выводим". Поиск кандидатов на возврат в "Продается" — товары с хорошей маржинальностью и приемлемой оборачиваемостью, которые восстановили спрос.

### IX. Новинки — статус раскрутки

Информационный блок без решений. Текущие метрики, тренд за период с момента запуска. Цель: мониторинг, принятие решений после 30+ дней.

### X. Возможности роста товарной матрицы

Два типа рекомендаций:
- **По цветам:** модели с 2-3 цветами и сильными метриками → предложение добавить цвета.
- **По размерам:** анализ распределения продаж по размерам. Если крайний размер (L) продаётся >50% от соседнего (M) → рекомендация добавить XL. С оговоркой о возможном влиянии на % выкупа.

### XI. Маркетинг × товарная матрица

Перекрёстный анализ: ДРР vs ABC-категория. Переинвестированные C-товары, недоинвестированные A-товары. Рекомендации по перераспределению бюджета.

### XII. Рекомендации → Действия

Приоритизированный список всех действий:

```
| # | Приоритет | Действие | Артикул/группа | Эффект руб/мес | Срок |
|---|---|---|---|---|---|
| 1 | 🔴 Срочно | Заказать 1000 шт | Wendy/black | +180К | 1-3 дня |
| 2 | 🔴 Срочно | Снизить цену -20% | Audrey/olive | высвоб. 280К | 1-3 дня |
| 3 | 🟡 Важно | Проверить контент | Joy (все цвета) | +60-90К | неделя |
| 4 | 🟢 Рекомендация | Расширить размерный ряд +XL | Eva/black | +35К | месяц |
```

### XIII. Итог

10-20 строк: главные цифры, ключевые решения, первые 3 действия.

## 7. Правила форматирования

### 7.1 Терминология — только русский

| Запрещено | Правильно |
|---|---|
| COGS | Себестоимость |
| Margin% | Маржинальность |
| Turnover | Оборачиваемость |
| Revenue | Выручка |
| ROI | Годовая доходность (допустимо ROI в скобках) |
| DRR | ДРР |

### 7.2 Числа

| Тип | Формат | Пример |
|---|---|---|
| Рубли | Пробел-разделитель + ₽ | `1 234 567 ₽` |
| Крупные суммы | Сокращённо | `8,8М ₽`, `42,8 млн` |
| Проценты | Одна десятая | `24,1%` |
| Изменение % | Со знаком | `+16,9%`, `-3,2%` |
| Процентные пункты | п.п. | `+3,2 п.п.` |
| Штуки | Пробел-разделитель | `1 234 шт` |
| Дни | + дн | `45 дн` |
| Отсутствующие данные | Тире | `—` |

Значимые изменения (Δ >±10%) — **жирным**.

### 7.3 Таблицы

Только markdown pipe-формат. Итоговые строки — жирным. Аномалии — `⚠️`.

### 7.4 Секции и callout-блоки

- Секции: `## ▶ I. Название`
- Подсекции: `### ▶ IV.1 Название`
- Callout-блоки:
  - `> ✅ Позитивный вывод`
  - `> ❌ Негативный вывод`
  - `> ⚠️ Предупреждение`
  - `> 💡 Инсайт`
  - `> 📊 Нейтральный вывод`
- Приоритеты: 🔴 Срочно (1-3 дня) / 🟡 Важно (неделя) / 🟢 Рекомендация (месяц)

### 7.5 Причинно-следственные цепочки

Обязательны для каждой находки:
- НЕ: "Joy/чёрный — кандидат на вывод"
- ДА: "Joy/чёрный: маржинальность 18%, ROI 72% (vs медиана коллекции 176%). ДРР = 2% (другие модели 6-9%) → недоинвестирован в рекламу → низкие продажи (2 шт/д vs медиана 8) → медленная оборачиваемость (92 дн)"

### 7.6 Confidence

Каждое решение (НАРАЩИВАТЬ / ВЫВОДИТЬ / и т.д.) сопровождается:
- Уровень: HIGH / MEDIUM / LOW
- Обоснование уровня: "данных достаточно, периоды согласованы" или "мало продаж, нужен 90-дневный горизонт"

### 7.7 Данные

- ТОЛЬКО реальные данные из БД. Никаких выдуманных цифр.
- Если данных нет: "Данные за период отсутствуют" с объяснением.
- Никогда не выдумывать модели или артикулы.
- Все ячейки заполнены если данные есть.
