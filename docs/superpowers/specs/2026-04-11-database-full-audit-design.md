# Database & Sheet Full Audit — Design Spec

**Date:** 2026-04-11
**Status:** Draft
**Scope:** Полный аудит Supabase БД + Google Sheet спецификации + синк-скрипт + рекомендации

## 1. Контекст и проблема

### Что случилось
При ручной проверке БД обнаружены ошибки данных:
- В `modeli_osnova` (id=24) попала запись `компбел-ж-бесшов` — это артикул, а не модель-основа
- В `modeli` (112 записей) — мусорные/дублирующиеся записи: `RubyPT` (должна быть `Ruby P`), `Vuki animal`, `Vuki выстиранки`, `VukiN animal`, `VukiW выстиранки` — русские названия тех же моделей
- Инфраструктурные таблицы (agent_runs, report_runs и т.д.) смешаны с товарными в одной schema `public`

### Источник правды
Google Sheet `19Nbr0kD8JJlwd7OCIMbM9qxucYNjmXnWtwJUgXv0vlg` — единственный источник правды для всех товарных данных. Листы: "Все модели", "Все товары", "Все артикулы", "Аналитики цветов", "Склейки WB", "Склейки OZON", "Упаковки".

### Синк-скрипт
`scripts/sync_sheets_to_supabase.py` (900+ строк) — однонаправленный синк Sheet → Supabase. Последний запуск 2026-04-07 уже зафиксировал проблемы, но не исправил (soft-delete с manual review).

## 2. Цели аудита

1. **Найти все ошибки данных** в PIM-таблицах (modeli_osnova → modeli → artikuly → tovary) и связанных справочниках
2. **Проанализировать структуру Google Sheet** — дубли колонок, невалидные данные, предложить улучшения
3. **Проверить инфраструктурные таблицы** — используются ли, нужны ли, описание, политика retention
4. **Найти баги в sync-скрипте** — почему мусорные данные попали в БД
5. **Предложить реорганизацию schema** — PIM vs Infra

## 3. Формат результата

**Фаза 1:** Отчёт `docs/reports/database-audit-2026-04-11.md` с таблицами проблем и рекомендаций. Каждая рекомендация с уровнем уверенности (HIGH / MEDIUM / LOW). Пользователь утверждает/отклоняет каждую.

**Фаза 2:** Применение утверждённых правок (отдельная сессия).

## 4. Архитектура — команда субагентов с оркестратором

### Принцип
Все субагенты работают под контролем оркестратора. Ни один агент не действует автономно — результаты каждого проверяются Verifier-агентом перед включением в отчёт.

### Волна 1 — Сбор данных (3 агента параллельно)

#### 1.1 DB Reader
- Подключается к Supabase через psycopg2 (connection string из .env)
- Снимает snapshot всех таблиц:
  - **PIM-таблицы** (modeli_osnova, modeli, artikuly, tovary, cveta, kategorii, kollekcii, statusy, importery, fabriki, razmery, skleyki_wb, skleyki_ozon, tovary_skleyki_wb, tovary_skleyki_ozon, sertifikaty, modeli_osnova_sertifikaty, istoriya_izmeneniy): полный SELECT
  - **Infra-таблицы** (agent_runs, report_runs, orchestrator_runs, kb_chunks, content_assets, analytics_rules, notification_log, archive_records, field_definitions): COUNT, MAX(created_at), sample 5 rows
- Выход: JSON-файл `/tmp/db-audit-snapshot.json`

#### 1.2 Sheet Reader
- Читает все листы через `gws sheets get` CLI:
  - "Все модели" (A:CM, ~314 rows)
  - "Все товары" (headers + data)
  - "Все артикулы" (headers + data)
  - "Аналитики цветов" (headers + data)
  - "Склейки WB" (headers + data)
  - "Склейки OZON" (headers + data)
  - "Упаковки" (headers + data)
- Выход: JSON-файл `/tmp/sheet-audit-snapshot.json`

#### 1.3 Code Scanner
- Grep по кодовой базе для каждой таблицы — какие файлы её упоминают
- Анализ SQL-запросов в shared/data_layer/, scripts/, services/
- Выход: JSON-файл `/tmp/code-usage-snapshot.json` — маппинг `table_name → [file:line, ...]`

### Волна 2 — Анализ (3 агента параллельно, на данных из Волны 1)

**Вход:** db-audit-snapshot.json + sheet-audit-snapshot.json
**Проверки:**

| Таблица | Проверка |

| modeli_osnova | Записи, которых нет в Sheet колонке B ("Модель основа") |
| modeli_osnova | Записи с некорректными значениями (артикулы вместо моделей) |
| modeli | Дубли по LOWER(kod) |
| modeli | Русские/транслитерные варианты одной модели (Vuki animal = Vuki) |
| modeli | Некорректные коды (RubyPT → Ruby P) |
| modeli | Записи без nazvanie_en (NULL) — нормально ли это? |
| modeli | FK: все model_osnova_id ссылаются на существующие modeli_osnova |
| artikuly | FK: все model_id ссылаются на существующие modeli |
| artikuly | FK: все cvet_id ссылаются на существующие cveta |

| tovary | FK: все artikul_id ссылаются на существующие artikuly |

**Выход:** Раздел отчёта с таблицей проблем — для каждой: запись, проблема, рекомендация.

#### 2.2 Sheet Analyst
**Вход:** sheet-audit-snapshot.json
**Проверки:**

| Лист | Проверка |
|------|----------|
| Все модели | Колонки B (Модель основа) vs C (Название модели) — семантическая разница, % совпадений. Если >90% идентичны → рекомендовать удалить одну |
| Все модели | Колонка H (Артикул модели) — валидность значений |
| Все модели | Пустые ячейки в обязательных колонках (A, B, G-Статус, I-Категория) |
| Все модели | Колонки AH-AJ помечены "не нужно" — предложить удалить |
| Все модели | 91 колонка — какие реально используются sync-скриптом, какие нет |
| Все товары | Структура, пустые ячейки, дубли |
| Все артикулы | Структура, пустые ячейки, дубли |
| Общее | Иерархическая структура (модель → размеры) — удобна ли для машинного чтения |

**Выход:** Раздел отчёта с предложениями по улучшению Sheet.

#### 2.3 Infra Auditor
**Вход:** db-audit-snapshot.json + code-usage-snapshot.json
**Проверки для каждой infra-таблицы:**

| Вопрос | Как проверяет |
|--------|---------------|
| Используется ли в коде? | code-usage-snapshot.json |
| Сколько записей? | COUNT из snapshot |
| Когда последняя запись? | MAX(created_at) |
| Пишут ли в неё сейчас? | Grep по INSERT/upsert в коде |
| Читают ли из неё? | Grep по SELECT/fetch в коде |
| Нужна ли? | На основе использования + даты последней записи |

**Для каждой таблицы выдаёт:**
- Назначение (описание)
- Вердикт: KEEP / DELETE / ARCHIVE / REVIEW
- Рекомендуемая retention policy (если KEEP)
- К какой schema отнести: `pim` или `infra`

**Выход:** Раздел отчёта с таблицей infra-таблиц.

### Волна 3 — Верификация (1 агент)

#### 3.1 Verifier
**Вход:** результаты PIM Auditor + Sheet Analyst + Infra Auditor
**Действия:**

1. **Противоречия:** Проверяет, не противоречат ли выводы агентов друг другу
   - Пример: PIM Auditor говорит "модель X — мусор, удалить", а Sheet Analyst находит её в Sheet
2. **Полнота:** Все ли таблицы покрыты аудитом (сверяет со списком из DB Reader)
3. **Spot-check:** Выборочно перепроверяет 5 рекомендаций типа "удалить" через прямой SQL-запрос
4. **Классификация:** Присваивает каждой рекомендации уровень уверенности:
   - **HIGH** — однозначная ошибка, безопасно исправлять (мусор в modeli_osnova)
   - **MEDIUM** — вероятная ошибка, нужно подтверждение (RubyPT → Ruby P)
   - **LOW** — неясно, требует ручной проверки (модели с NULL в nazvanie_en)
5. **Сборка отчёта:** Объединяет все разделы в единый `docs/reports/database-audit-2026-04-11.md`

## 6. Технические детали

### Доступ к данным
- **Supabase:** psycopg2, connection string из `.env` (DB_HOST, DB_PORT, DB_USER, DB_PASS, DB_NAME)
- **Google Sheets:** `gws sheets get <spreadsheet_id> --range '<sheet>!<range>'`
- **Кодовая база:** Grep/Glob по локальным файлам

### Безопасность
- Фаза 1 — ТОЛЬКО ЧТЕНИЕ. Никаких INSERT/UPDATE/DELETE
- Все данные сохраняются в /tmp/ (временные) и docs/reports/ (отчёт)
- Фаза 2 (применение правок) — отдельная сессия после утверждения

### Ограничения
- DB Server (89.23.119.253:6433) — ТОЛЬКО ЧТЕНИЕ (это сервер подрядчика, не Supabase)
- Supabase — можно читать и писать, но в Фазе 1 только читаем
- Google Sheets — читаем через gws CLI

## 7. Что НЕ входит в scope

- Миграция schema (pim/infra) — только рекомендация, не исполнение
- Правки в Google Sheet — только рекомендации
- Рефакторинг sync-скрипта — только описание багов
- Таблицы WB/OZON аналитики (wb_daily_report, ozon_daily_report и т.д.) — они на стороннем сервере подрядчика
