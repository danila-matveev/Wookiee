# Василий — скрипт оптимизации индекса локализации WB

> Полное описание системы: как работает, какие API использует, алгоритм расчёта, архитектура, и план автоматизации перестановок.

---

## 1. Что делает скрипт

Скрипт **автоматически рассчитывает**, какие товары и в каком количестве нужно **переместить между складами Wildberries**, чтобы повысить **индекс локализации** — метрику, показывающую долю заказов, отгруженных из локального региона.

**Целевой индекс: 75%** — при достижении WB снижает логистические издержки.

### Результат работы скрипта
- Таблица перемещений: **откуда** (склад-донор) → **куда** (склад-получатель) → **сколько штук**
- Таблица допоставок: что довезти **с собственного склада** на WB
- Сводка по регионам и проблемным SKU
- Всё автоматически экспортируется в **Google Sheets**

```

shared/clients/
├── wb_client.py                     # WB API клиент
├── moysklad_client.py               # МойСклад API клиент
```

### Кабинеты
Скрипт работает для двух юрлиц (кабинетов WB):
- **IP** (ИП) — с токеном `CABINET_IP`
- **OOO** (ООО) — с токеном `CABINET_OOO`

## 3. Источники данных и API

### 3.1. WB Analytics API — Остатки на складах

| Параметр | Значение |
|----------|----------|
| **Эндпоинт** | `GET /api/v1/warehouse_remains` |
| **Base URL** | `https://seller-analytics-api.wildberries.ru` |
| **Тип** | Асинхронный отчёт (создание → опрос → скачивание) |
| **Авторизация** | `Authorization: <API_KEY>` |
| **Время генерации** | 5-15 минут |

**Как работает:**
1. `GET /api/v1/warehouse_remains?groupByBarcode=true&groupBySa=true&groupByNm=true&groupBySize=true` → получаем `taskId`
2. `GET /api/v1/warehouse_remains/tasks/{taskId}/status` → поллим каждые 15 сек (до 100 попыток)
3. После `status: "done"` ждём 60 секунд (буфер готовности)
4. `GET /api/v1/warehouse_remains/tasks/{taskId}/download` → массив товаров

**Что получаем:**
```json
{
  "vendorCode": "WKEE-HOODIE-001",
  "techSize": "M",
  "warehouses": [
    {"warehouseName": "Коледино", "quantity": 150},
    {"warehouseName": "Краснодар", "quantity": 30}
  ]
}
```

**Фильтры при обработке:**
- Склады-транзиты (`SKIP_WAREHOUSES`) — исключаются
- Только российские ФО (6 округов) — СНГ не участвует в перемещениях

### 3.2. WB Statistics API — Заказы покупателей

| Параметр | Значение |
|----------|----------|
| **Эндпоинт** | `GET /api/v1/supplier/orders` |
| **Base URL** | `https://statistics-api.wildberries.ru` |
| **Тип** | Синхронный, с пагинацией |
| **Период** | 30 дней по умолчанию |

**Что получаем для каждого заказа:**
```json
{
  "supplierArticle": "WKEE-HOODIE-001",
  "techSize": "M",
  "nmId": 12345678,
  "warehouseName": "Коледино",           // откуда отгрузили
  "oblastOkrugName": "Центральный ФО",   // куда доставили
  "oblast": "Московская область",
  "date": "2026-03-20T14:30:00",
  "isCancel": false
}
```

**Критические нюансы обработки:**
- **НЕ фильтруем по `isCancel`** — WB учитывает отменённые заказы в индексе
- Фильтруем по полю `date`, не по `lastChangeDate`
- `oblastOkrugName` — приоритетный источник ФО доставки, fallback на `oblast`

### 3.3. МойСклад API — Остатки собственного склада

| Параметр | Значение |
|----------|----------|
| **Эндпоинт** | `GET /entity/assortment` |
| **Base URL** | `https://api.moysklad.ru/api/remap/1.2` |
| **Авторизация** | Bearer token (`MOYSKLAD_TOKEN`) |

**Что получаем:** `{article: quantity}` — текущие остатки на нашем складе.

Используется для расчёта допоставок: если на складах WB нехватка, а у нас товар есть — предлагаем довезти.

**Graceful degradation:** если токен не задан — скрипт работает без допоставок.

### 3.4. Supabase — Статусы артикулов

| Параметр | Значение |
|----------|----------|
| **Функция** | `data_layer.get_artikuly_statuses()` |
| **Зачем** | Артикулы со статусом «Выводим» исключаются из допоставок |

### 3.5. Спецификации.xlsx — Справочник баркодов

| Параметр | Значение |
|----------|----------|
| **Файл** | `sku_database/Спецификации.xlsx` → лист «Все товары» |
| **Маппинг** | `(артикул, размер) → баркод` |
| **Зачем** | Баркоды добавляются в отчёт для удобства склада |

## 4. Маппинг регионов

### Склады WB → Федеральные округа

Файл `wb_localization_mappings.py` содержит **40+ маппингов** складов:

| Склад WB | Федеральный округ |
|-----------|-------------------|
| Коледино, Электросталь, Тула, Подольск | Центральный |
| Краснодар, Невинномысск | Южный + Северо-Кавказский |
| Казань | Приволжский |
| СПб Шушары | Северо-Западный |
| Екатеринбург | Уральский |
| Новосибирск, Хабаровск | Дальневосточный + Сибирский |

### Области доставки → Федеральные округа

**285+ вариантов написания** российских регионов маппятся в 6 ФО.

### Определение локальности заказа

```
Заказ ЛОКАЛЬНЫЙ, если:
  ФО склада отгрузки == ФО региона доставки

Пример: Коледино (Центральный) → Москва (Центральный) = ЛОКАЛЬНЫЙ
Пример: Коледино (Центральный) → Екатеринбург (Уральский) = НЕ ЛОКАЛЬНЫЙ
```

## 5. Алгоритм расчёта

### Шаг 1: Расчёт индекса локализации по SKU

```
Индекс = Локальные заказы / Все заказы × 100%
```

Считается для каждой комбинации `(артикул, размер)`.

### Шаг 2: Выявление проблемных SKU

Фильтр: `индекс < 75%` И `заказов > 0`

Сортировка по **impact** (влиянию):
```
impact = Всего заказов × (75% − текущий индекс)
```

Чем больше заказов и чем ниже индекс — тем выше приоритет.

### Шаг 3: Защита доноров

Перед тем как забрать товар из региона-донора, проверяем безопасность:

```python
daily_orders = region_orders_total / period_days
safe_minimum = ceil(daily_orders × safety_days)  # по умолчанию 14 дней
available_for_transfer = max(0, stock - safe_minimum)
```

**Правило:** нельзя забрать товар, если:
- После изъятия остаток будет ниже `safe_minimum` (14 дней продаж)
- Индекс донора упадёт ниже `min_donor_localization` (70%)

### Шаг 4: Матчинг донор → получатель

Жадный алгоритм (greedy matching):
1. Для каждого проблемного SKU находим **доноров** (регионы с излишками) и **получателей** (регионы с дефицитом)
2. Каждый получатель «тянет» товар у доноров, пока дефицит не закрыт или доноры не исчерпаны
3. Если у донора несколько складов в регионе — распределяем пропорционально остаткам

### Шаг 5: Допоставки

Если перемещений между складами WB недостаточно:
```
К допоставке = min(нужно, макс_по_обороту, есть_на_своём_складе)
макс_по_обороту = daily_orders × max_turnover_days  # 100 дней
```

Артикулы со статусом «Выводим» — исключаются.

### Шаг 6: Приоритизация

| Приоритет | Условие | Балл |
|-----------|---------|------|
| Критично+ | Индекс < 50% И заказов > 10 | 35 |
| Критично | Индекс < 50% | 30 |
| Важно+ | Индекс 50-75% И заказов > 10 | 25 |
| Важно | Индекс 50-75% | 20 |
| OK | Индекс ≥ 75% | 10 |

## 6. Выходные данные

### Google Sheets (автоматический экспорт)

Для каждого кабинета (IP, OOO) создаётся 5 листов:

| Лист | Содержимое |
|------|-----------|
| **Перемещения {кабинет}** | Откуда склад → Куда склад × Кол-во |
| **Допоставки {кабинет}** | Артикул → Регион → Кол-во → Есть на своём складе |
| **Сводка {кабинет}** | Общий индекс, кол-во перемещений/допоставок |
| **Регионы {кабинет}** | Индекс по каждому ФО, доля остатков/заказов |
| **Проблемные SKU {кабинет}** | Топ SKU по impact |

Плюс общие листы:
- **Обновление** — дашборд с метриками обоих кабинетов
- **История** — append-only лог всех расчётов с трендом

### Excel (локальный файл)

8 листов: Сводка, Критичные, Важные, Остальные, Все перемещения, Анализ SKU, Защита доноров, Допоставки, Регионы, Динамика.

Сохраняется: `Отчеты готовые/Отчет_локализация_v3_{кабинет}_{дата}.xlsx`

### SQLite (аудит)

`services/wb_localization/data/vasily.db` — таблица `reports`:
- cabinet, timestamp, overall_index, movements, supplies, region/problem JSON
- Сравнение с предыдущим расчётом (регионы улучшились / ухудшились)

## 7. Способы запуска

### CLI
```bash
# Оба кабинета, 30 дней
python -m services.wb_localization.run_localization

# Один кабинет, другой период
python -m services.wb_localization.run_localization --cabinet ip --days 14

# Кастомные параметры
python -m services.wb_localization.run_localization --safety-days 7 --min-donor-localization 65

# Dry-run (только загрузка и сводка, без отчёта)
python -m services.wb_localization.run_localization --dry-run
```

### HTTP API (vasily_api)
```
POST /run      — запуск фонового расчёта (требует X-API-Key)
GET  /status   — статус: idle / running / done / error
GET  /health   — Docker healthcheck
```

Используется **Google Apps Script** — кнопка в Google Sheets вызывает `/run`.

## 8. Анализ WB API: возможности автоматизации перестановок

### 8.1. Что доступно в WB API

#### Управление складами продавца (FBS)

| Метод | Эндпоинт | Описание |

|-------|----------|----------|

| PATCH | `/api/v3/supplies/{id}/deliver` | Закрыть и отправить |

#### Коробки в поставке
| Метод | Эндпоинт | Описание |

| POST | `/api/v3/supplies/{id}/trbx` | Создать коробку |

| Метод | Эндпоинт | Описание |
|-------|----------|----------|

| Метод | Эндпоинт | Описание |

#### Пропуска на склад
| Метод | Эндпоинт | Описание |
|-------|----------|----------|
| POST | `/api/v3/passes` | Создать пропуск (ФИО, номер авто, склад) |
| GET | `/api/v3/passes` | Список пропусков |

### 8.2. Что НЕ доступно в WB API

> **Перераспределение товара между складами WB (FBO → FBO) через API НЕ ПОДДЕРЖИВАЕТСЯ.**

Это подтверждено:
- В [WB Developer Forum](https://dev.wildberries.ru/forum/1506) — нет публичного API для перераспределения
- Функция управляется только через **Seller Portal UI** (раздел «Отчёт по остаткам на складах»)
- Требует включения в **Тарифном конструкторе** (комиссия 0.5% оборота)
- Лимит: 5,000 единиц/день на склад
- Нельзя выбрать конкретный размер для мульти-размерных товаров

### 8.3. Возможные пути автоматизации

#### Вариант A: Автоматизация через поставки FBW (допоставки)

Можно автоматизировать **допоставки с собственного склада на WB** через официальный API:

```
1. GET /api/v1/acceptance/coefficients
   → Найти склады WB с открытой приёмкой (coefficient ≤ 1, allowUnload = true)

2. GET /api/v1/warehouses
   → Определить доступные типы упаковки для конкретных баркодов

3. Создать поставку через FBS workflow:
   POST /api/v3/supplies → supplyId
   POST /api/v3/supplies/{id}/trbx → создать коробки
   PATCH /api/v3/supplies/{id}/trbx/{trbxId} → добавить товары
   POST /api/v3/supplies/{id}/trbx/stickers → получить стикеры
   GET /api/v3/supplies/{id}/barcode → QR поставки
   PATCH /api/v3/supplies/{id}/deliver → отправить

4. POST /api/v3/passes → создать пропуск на склад
```

**Статус:** Полностью автоматизируемо через API.

#### Вариант B: Автоматизация перераспределения через Seller Portal (headless browser)

Поскольку API для перестановок нет, единственный путь — **автоматизация UI**:

1. Авторизация в Seller Portal через headless browser (Playwright/Selenium)
2. Навигация в раздел перераспределения
3. Заполнение форм на основе данных скрипта
4. Подтверждение заявок

**Риски:**
- Нарушение Terms of Service WB
- Нестабильность при изменении UI
- Необходимость хранения авторизационных данных
- Блокировка аккаунта

#### Вариант C: Гибридный подход (рекомендуемый)

1. **Скрипт рассчитывает** оптимальные перемещения (уже работает)
2. **Google Sheets** показывает готовые заявки с приоритетами
3. **Оператор** вручную создаёт заявки на перераспределение в Seller Portal по данным из таблицы
4. **Допоставки** автоматизируются через WB API (Вариант A)

Это минимизирует риски и максимизирует автоматизацию там, где API позволяет.
