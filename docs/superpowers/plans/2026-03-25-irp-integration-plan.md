# План интеграции ИРП в скрипт перестановок

## Контекст

Скрипт перестановок (`services/wb_localization/`) оптимизирует **индекс локализации (ИЛ)** — перемещает товары между складами WB, чтобы заказы обслуживались локально. С 23.03.2026 WB добавил **Индекс Распределения Продаж (ИРП)** — процент от цены товара, который взимается при низкой локализации артикула.

**Ключевой инсайт**: ИРП создаёт финансовый обрыв на границе 60% локализации. Выше 60% — КРП = 0 (бесплатно). Ниже — от 2.00% до 2.50% от цены каждого заказа. Текущий скрипт оптимизирует под порог 75% (TARGET_LOCALIZATION_INDEX), но **реальный финансовый эффект максимален при переходе через границу 60%**.

Текущие значения: ИЛ = 0.98, ИРП = 0.42%. Расчёт calc_irp.py сходится с WB Partners.

---

## Что меняется

### 1. Приоритизация перемещений

**Сейчас**: `impact = orders × (75 - loc_index)` — чисто по "расстоянию до цели 75%".

**Нужно**: двухуровневая приоритизация:
- **Уровень 1 (ИРП-зона)**: артикулы с локализацией < 60% → у них КРП > 0, каждый заказ стоит денег. Приоритет = **финансовый impаct в рублях**.
- **Уровень 2 (ИЛ-зона)**: артикулы с локализацией 60-75% → КРП = 0, но КТР > 1.0 (наценка на логистику). Приоритет = старая формула impact.
- **Уровень 3 (OK)**: > 75% → никаких действий.

Финансовый impact (₽) для ИРП-зоны:

```
impact_rub = КРП% × средняя_цена_товара × ожидаемые_заказы_за_период
```

### 3. Источник цен

Для расчёта финансового impact в рублях нужна цена товара. Источники:
- **WB Prices API** (`/api/v1/info`) — розничная цена до скидки WB (именно она умножается на ИРП)
- **reportDetailByPeriod** → поле `retail_price` (но это за прошлый период)
- **Supabase / abc_date** → средняя цена из финансовых данных

Рекомендация: использовать WB Prices API, т.к. ИРП считается от **текущей розничной цены до скидки WB**.

### Шаг 4: Новая приоритизация

Заменить `add_priority()`:

```python
# Текущая логика (только по локализации):
# Критично+ = loc < 50% & orders >= 10  → балл 35
# Критично  = loc < 50%                 → балл 30
# Важно+    = loc < 75% & orders >= 10  → балл 25
# Важно     = loc < 75%                 → балл 20
# OK        = loc >= 75%                → балл 10

# Новая логика (ИРП + ИЛ):
# ИРП-Критично = loc < 60% & impact_rub > 5000₽/мес  → балл 50
# ИРП-Важно    = loc < 60% & impact_rub > 0           → балл 40
# ИЛ-Критично  = 60% <= loc < 75% & orders >= 10      → балл 25
# ИЛ-Важно     = 60% <= loc < 75%                     → балл 20
# OK           = loc >= 75%                            → балл 10
```

Ключевое: **ИРП-зона всегда выше по приоритету**, потому что каждый заказ стоит реальных денег (2-2.5% от цены).

## Важные ограничения

- **Товары-исключения**: WB упоминает товары, у которых КРП всегда = 0%. Список не публичен. Если расхождение projected vs WB > 0.1%, нужно будет добавить исключения вручную.
- **Цены**: ИРП считается от цены **до скидки WB** (retail_price в API), не от final_price.
- **Период**: WB считает ИРП за 13 недель (91 день), а наш скрипт перестановок работает с `--days 30`. Для расчёта ИРП нужно загружать заказы за 91 день отдельно.
- **КТР < 60%**: значения КТР ниже 60% локализации экстраполированы (WB не показывает). Если найдём расхождение — подкорректируем.
