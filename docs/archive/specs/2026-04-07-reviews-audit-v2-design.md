# Reviews Audit Skill v2 — Design Spec

## Overview

Claude Code skill (`/reviews-audit`) v2 — глубокий продуктовый анализ отзывов, вопросов и возвратов на Wildberries. Монолитный SKILL.md с 8 фазами, LLM-кластеризация текстов, продуктовый анализ по каждой активной модели, rich-публикация в Notion.

**Контекст:** WOOKIEE — бренд женского бесшовного белья, ~400 млн руб/год, WB + Ozon. Фокус v2 — Wildberries.

## Решения

| Вопрос | Решение |
|--------|---------|
| Архитектура | Монолитный SKILL.md + субагенты |
| Кластеризация | LLM (все отзывы с текстом, порции по 200) |
| Статус моделей | Supabase MCP (фильтр выводимых) |
| Позиционирование | Notion MCP (база «Модельный ряд») |
| Публикация Notion | Через Notion MCP напрямую (enhanced markdown) |
| Публикация MD | plain markdown в git (`docs/reports/`) |
| Скоуп данных | Отзывы + Вопросы (чаты — API 404, отложены) |
| Источник заказов/возвратов | БД через shared/data_layer |
| Кабинеты | Оба (IP + OOO), дедупликация |

## Отличия от v1

| Аспект | v1 | v2 |
|--------|----|----|
| Кластеризация | keyword-based (6 кластеров) | LLM (все найденные кластеры) |
| Модели | Только проблемные детально | Карточка КАЖДОЙ активной модели |
| Преимущества | Не анализировались | Все кластеры преимуществ |
| Фильтр моделей | Нет | Supabase: исключить «выводим» |
| Возвраты | Общая таблица | Помесячная динамика + drill-down до артикула |
| Notion | sync_report_to_notion (plain) | Notion MCP (rich: цвета, callout, toggle) |
| Ответы бренда | Ошибочный % «без ответа» | Убран — все отзывы отвечены |
| iscancel | Баг (text vs integer) | Исправлен: cast to text |
| Адаптивность | Один режим | 4 режима: год/квартал/месяц/неделя |

## Архитектура

```
Пользователь → /reviews-audit
       ↓
  [Фаза 1] Параметры: период, фокус
       ↓
  [Фаза 2] Сбор данных: collect_data.py v2
       ├── WB Feedbacks API (оба кабинета, дедупликация)
       ├── WB Questions API
       └── БД: заказы/выкупы/возвраты по артикулам + помесячно
       ↓
  [Фаза 3] Фильтрация и маппинг:
       ├── Supabase MCP → nmId → модель/артикул/цвет
       ├── Supabase MCP → статус модели (исключить «выводим»)
       └── Группировка: model_osnova → artikul → цвет
       ↓
  [Фаза 4] Цифровой анализ (Claude):
       ├── Сводная карточка по каждой модели
       ├── Помесячная динамика (рейтинг, отзывы, % негатива)
       ├── Возвраты + drill-down до артикула при аномалиях
       └── Алерты (статистически значимые, >5 случаев)
       ↓
  [Фаза 5] LLM-кластеризация текстов (субагент):
       ├── Все отзывы с текстом (~5-6K за год) порциями по 200
       ├── ВСЕ проблемы + ВСЕ преимущества (не топ-3)
       ├── Частотность + помесячная динамика долей
       ├── Цитаты по каждому кластеру
       └── Разбивка per_model для Фазы 6
       ↓
  [Фаза 6] Продуктовый анализ по моделям (субагенты параллельно):
       ├── Карточка каждой активной модели
       ├── Рекомендации: производство, карточка, cross-sell
       └── Анализ вопросов по модели
       ↓
  [Фаза 7] Gap-анализ (Notion MCP):
       └── Позиционирование из «Модельный ряд» vs реальное восприятие
       ↓
  [Фаза 8] Синтез + публикация:
       ├── MD-файл → docs/reports/ (plain markdown для git)
       └── Notion → enhanced markdown через MCP (rich-форматирование)
```

## Фаза 1: Ввод параметров

При запуске скилл спрашивает:

1. **Период**: последняя неделя / месяц / квартал / год / кастомные даты
2. **Фокус** (опционально): конкретная модель или «все»

Авто-определение гранулярности:
- Год → помесячно (12 точек)
- Квартал → помесячно (3 точки)
- Месяц → понедельно (4-5 точек)
- Неделя → подневно (7 точек)

Авто-определение глубины анализа:

| Период | Карточки моделей | LLM-анализ | Gap-анализ |
|--------|-----------------|-----------|-----------|
| Год | Все активные | Все тексты | Да |
| Квартал | Все активные | Все тексты | Да |
| Месяц | Только с алертами + сводная | Все тексты | Только проблемные |
| Неделя | Только алерты | Все тексты | Нет |

## Фаза 2: Сбор данных

### collect_data.py v2

**CLI:**
```bash
python3 scripts/reviews_audit/collect_data.py \
  --date-from "2025-04-07" \
  --date-to "2026-04-07" \
  --cabinet both \
  --output /tmp/reviews_audit_data.json
```

**Параметр `--cabinet`:** `ip` | `ooo` | `both` (дефолт `both`)

При `both`: собирает с обоих кабинетов, дедуплицирует по `id` отзыва/вопроса (WB API отдаёт одинаковые данные по бренду, но дедупликация на случай расхождений).

### WB API

| Эндпоинт | Данные |
|----------|--------|
| `GET /api/v1/feedbacks` | Отзывы: text, pros, cons, productValuation, createdDate, answer, productDetails (nmId, supplierArticle), color |
| `GET /api/v1/questions` | Вопросы: text, answer, createdDate, productDetails |

- Базовый URL: `https://feedbacks-api.wildberries.ru`
- Авторизация: `Authorization: {WB_API_KEY_IP}` / `{WB_API_KEY_OOO}`
- Пагинация: take=5000, skip-based
- Rate limit: 0.1s между страницами, 60s backoff на 429

### БД (shared/data_layer)

Три SQL-запроса:

**1. Заказы/возвраты по модели (существующий, исправленный):**
```sql
SELECT
    CASE WHEN date >= %s THEN 'current' ELSE 'previous' END as period,
    {get_osnova_sql("SPLIT_PART(supplierarticle, '/', 1)")} as model,
    COUNT(*) as orders_count,
    SUM(CASE WHEN iscancel::text IN ('0', 'false') OR iscancel IS NULL THEN 1 ELSE 0 END) as buyout_count,
    SUM(CASE WHEN iscancel::text IN ('1', 'true') THEN 1 ELSE 0 END) as return_count
FROM orders
WHERE date >= %s AND date < %s
GROUP BY 1, 2
ORDER BY 1, 3 DESC;
```

**2. Заказы/возвраты по артикулу (новый):**
```sql
SELECT
    {get_osnova_sql("SPLIT_PART(supplierarticle, '/', 1)")} as model,
    LOWER(supplierarticle) as artikul,
    COUNT(*) as orders_count,
    SUM(CASE WHEN iscancel::text IN ('0', 'false') OR iscancel IS NULL THEN 1 ELSE 0 END) as buyout_count,
    SUM(CASE WHEN iscancel::text IN ('1', 'true') THEN 1 ELSE 0 END) as return_count
FROM orders
WHERE date >= %s AND date < %s
GROUP BY 1, 2
ORDER BY 1, 3 DESC;
```

**3. Помесячная разбивка (новый):**
```sql
SELECT
    DATE_TRUNC('month', date)::date as month,
    {get_osnova_sql("SPLIT_PART(supplierarticle, '/', 1)")} as model,
    COUNT(*) as orders_count,
    SUM(CASE WHEN iscancel::text IN ('0', 'false') OR iscancel IS NULL THEN 1 ELSE 0 END) as buyout_count,
    SUM(CASE WHEN iscancel::text IN ('1', 'true') THEN 1 ELSE 0 END) as return_count
FROM orders
WHERE date >= %s AND date < %s
GROUP BY 1, 2
ORDER BY 1, 2;
```

## Фаза 3: Фильтрация и маппинг

### 3.1 Маппинг nmId → товарная матрица

Через Supabase MCP:

```sql
SELECT p.nm_id, a.artikul, a.cvet_id, c.name as cvet_name,
       m.name as model_name, mo.name as model_osnova_name,
       mo.id as model_osnova_id
FROM products p
JOIN artikuls a ON a.id = p.artikul_id
JOIN models m ON m.id = a.model_id
JOIN model_osnovas mo ON mo.id = m.model_osnova_id
LEFT JOIN cvets c ON c.id = a.cvet_id
WHERE p.nm_id IN ({{nm_ids}});
```

Результат: `nmId → {model_osnova, model, artikul, cvet}`.

Если nmId не найден — маппинг через `supplierArticle` из feedbacks (fallback, как в v1).

### 3.2 Статусы моделей

Через Supabase MCP:

```sql
SELECT name, status FROM model_osnovas;
```

Модели со статусом, обозначающим выводимость, исключаются из анализа. В отчёте одной строкой: «Исключены выводимые модели: X, Y, Z».

### 3.3 Группировка

Три уровня агрегации:
- **Уровень 1:** model_osnova (Ruby, Wendy, Joy...)
- **Уровень 2:** artikul (Ruby/розовый, Ruby/чёрный...)
- **Уровень 3:** конкретный nmId (для drill-down)

## Фаза 4: Цифровой анализ

Claude считает метрики из собранных данных по каждой активной модели.

### 4.1 Сводная карточка модели

| Метрика | Формула |
|---------|---------|
| Рейтинг | средневзвешенный по productValuation |
| Зона | >=4.7 целевой / 4.5-4.6 приемлемый / <4.5 плохой |
| Распределение звёзд | кол-во 1★ / 2★ / 3★ / 4★ / 5★ |
| Нужно 5★ | ceil((4.7 * total - sum_stars) / 0.3) — только если <4.7 |
| Заказы | из orders_by_model |
| Возвраты % | returns / orders * 100 |
| Отзывов | count feedbacks для модели |
| Вопросов | count questions для модели |

### 4.2 Помесячная динамика

Для каждой модели — таблица по месяцам:
- Рейтинг
- Кол-во отзывов
- % негативных (1-3★) к общему числу
- % возвратов (из orders_monthly)
- Визуальный тренд: растёт / стабилен / падает

### 4.3 Drill-down по артикулам

Если у модели:
- Рейтинг <4.5 (плохой) → разбить по артикулам/цветам
- % возвратов аномально высок → найти конкретный артикул-виновник
- Алерт сработал → drill-down до причины

Используем `orders_by_artikul` для поиска конкретного артикула с проблемой.

### 4.4 Алерты

Срабатывают ТОЛЬКО при >5 случаях:
- Рейтинг модели упал >=0.2 за последние 3 месяца
- % возвратов модели вырос непропорционально к росту заказов
- Доля негативных отзывов модели выросла >2x между кварталами
- Конкретный артикул/цвет имеет рейтинг на 0.5+ ниже модели

## Фаза 5: LLM-кластеризация текстов

### Субагент-кластеризатор

Получает все отзывы с текстом (text, pros, cons — хотя бы одно непустое). Обрабатывает порциями по 200.

### Задача на каждую порцию

Для каждого отзыва определить:
- **Тональность** (по тексту, не по звёздам): позитивный / негативный / смешанный / нейтральный
- **Проблемы** (если есть): список конкретных проблем своими словами
- **Преимущества** (если есть): список конкретных похвал
- **Модель** (из productDetails.supplierArticle)
- **Месяц** (из createdDate)

### Агрегация

После всех порций субагент мержит и формирует:

**Кластеры проблем (ВСЕ найденные):**
- Название кластера
- Количество упоминаний
- % от всех отзывов с текстом
- Помесячная динамика (кол-во по месяцам)
- Топ-модели внутри кластера с количеством
- 3-5 показательных цитат

**Кластеры преимуществ (ВСЕ найденные):**
- Аналогичная структура

**Разбивка per_model:**
Для каждой модели — её проблемы и преимущества с частотностью. Это входные данные для Фазы 6.

### Выход

JSON-файл `/tmp/reviews_audit_clusters.json`:
```json
{
  "problems": [
    {
      "cluster": "Размерная сетка — маломерит",
      "count": 187,
      "pct": 3.3,
      "monthly": {"2025-04": 12, "2025-05": 15, ...},
      "top_models": [{"model": "wendy", "count": 42}, ...],
      "quotes": ["цитата 1", "цитата 2", ...]
    }
  ],
  "advantages": [...],
  "per_model": {
    "ruby": {
      "problems": [{"cluster": "...", "count": N, "pct": N, "quotes": [...]}],
      "advantages": [{"cluster": "...", "count": N, "pct": N, "quotes": [...]}]
    }
  }
}
```

## Фаза 6: Продуктовый анализ по моделям

### Параллельные субагенты

По одному субагенту на каждую активную модель (8-12 моделей). Запускаются параллельно.

### Входные данные субагента

- Метрики из Фазы 4 (рейтинг, динамика, возвраты, drill-down)
- Кластеры из Фазы 5 (`per_model[model]`)
- Все тексты вопросов по этой модели

### Карточка модели (выход субагента)

```
## {Model Name}

### Сводка
Рейтинг: X.XX (зона) | Отзывов: N | Вопросов: N
Заказов: N | Возвратов: X.X% | Тренд рейтинга: [растёт/стабилен/падает]

### Динамика рейтинга
[таблица по месяцам: рейтинг, кол-во отзывов, % негативных, % возвратов]

### Распределение звёзд
[1★: N | 2★: N | 3★: N | 4★: N | 5★: N]
[если <4.7: «Нужно X пятизвёздочных отзывов для 4.7»]

### Возвраты
[% возвратов по месяцам]
[если аномалия: drill-down по артикулам — какой артикул/цвет виновник]

### Что ценят покупатели
[ВСЕ кластеры преимуществ с частотностью и цитатами]
1. {Преимущество} ({N} упоминаний, {X}%)
   «цитата»
...

### Проблемы
[ВСЕ кластеры проблем с частотностью, динамикой и цитатами]
1. {Проблема} ({N} упоминаний, {X}%)
   Динамика: [описание тренда]
   Артикулы: [если drill-down нужен]
   «цитата»
...

### Анализ вопросов
[О чём спрашивают покупатели: темы + частотность]

### Рекомендации
- Производство: [конкретные дефекты → конкретные действия]
- Карточка товара: [что добавить/исправить в описании]
- Cross-sell: [какие модели рекомендовать покупателям этой модели]
```

## Фаза 7: Gap-анализ

### Источник

Notion MCP → страница WOOKIEE → база «Модельный ряд»:
- Поле `Name` — название модели
- Поле `Позиционирование` — продуктовый смысл

### Анализ

Только для моделей с проблемами из Фаз 4-6:

| Модель | Задумка (Notion) | Реальность (отзывы) | Gap | Рекомендация |
|--------|-----------------|-------------------|-----|-------------|
| Ruby | «комфорт, мягкая ткань» | Хвалят комфорт, ругают швы | Швы ≠ качество ткани | Проверить пошив |
| Lana | «универсальный размер» | «маломерит, верх/низ не совпадают» | Размерная сетка | Скорректировать лекала |

## Фаза 8: Синтез + публикация

### Самопроверка

Перед финализацией:
- [ ] Все цифры имеют контекст (% от чего? из какого периода?)
- [ ] Нет алертов на единичных случаях (<5 случаев = не алерт)
- [ ] Рекомендации конкретные, а не «улучшите качество»
- [ ] Динамика учитывает пропорции к объёму заказов
- [ ] Тональность определена по тексту, не по звёздам
- [ ] Выводимые модели исключены
- [ ] Каждая активная модель имеет карточку (для года/квартала)

### Структура отчёта

```
I. Executive Summary
   [3-5 callout-блоков с ключевыми выводами и цифрами]

II. Сводная таблица всех активных моделей
   [цветная таблица: модель / рейтинг / зона / тренд / возвраты / топ-проблема]
   [green_bg = целевой, yellow_bg = приемлемый, red_bg = плохой]
   [Строка: «Исключены выводимые: X, Y, Z»]

III. Карточки моделей
   [toggle-секция на каждую активную модель — из Фазы 6]

IV. Кластеры проблем (бренд в целом)
   [ВСЕ кластеры с частотностью, помесячной динамикой, цитатами]
   [Алерты — callout red_bg]

V. Кластеры преимуществ (бренд в целом)
   [ВСЕ кластеры — для маркетинга и карточек товаров]

VI. Gap-анализ: позиционирование vs восприятие
   [таблица: модель / задумка / реальность / gap / рекомендация]

VII. Actionable-рекомендации
   [КРИТИЧНО → ВАЖНО → УЛУЧШЕНИЯ]
   [каждая с «что если» сценарием в рублях/процентах]

VIII. Методология
   [источники данных, ограничения, период]
```

### Публикация

**MD-файл:**
```
docs/reports/reviews-audit-{YYYY-MM-DD}.md
```
Plain markdown для git-истории и grep.

**Notion:**
Через Notion MCP (`notion-create-page`) в базу «Аналитические отчёты».

Enhanced markdown по образцу Q4 vs Q1 отчёта:
- `color="blue_bg"` для шапок таблиц
- `color="green_bg"` / `color="red_bg"` для подсветки ячеек
- Callout-блоки с иконками для ключевых выводов
- Toggle-секции для карточек моделей (раскрываемые)
- Горизонтальные разделители между секциями

### Адаптация по периоду

| Период | Глубина | Карточки моделей | LLM-анализ | Gap-анализ |
|--------|---------|-----------------|-----------|-----------|
| Год | Полная | Все активные | Все тексты (~5-6K) | Да |
| Квартал | Полная | Все активные | Все тексты (~2-3K) | Да |
| Месяц | Средняя | Только с алертами + сводная | Все тексты (~1-2K) | Только проблемные |
| Неделя | Лёгкая | Только алерты | Все тексты (~200-500) | Нет |

## Принципы анализа

1. **Доли, а не абсолюты.** Все метрики относительные. Рост возвратов пропорционально заказам — норма.
2. **Текст > звёзды.** 5★ с претензией = негативный. 3★ с похвалой = позитивный.
3. **Пороги значимости.** <5 случаев = шум. Не алертить.
4. **Динамика долей.** Было 1% → стало 3% = алерт. Было 1% → осталось 1% = норма.
5. **Drill-down при проблеме.** Модель → артикул → цвет. Найти виновника.
6. **Actionable.** Не «улучшите качество», а «проверить стойкость краски голубого Wendy, партия Q1 2026».
7. **GROUP BY с LOWER().** При группировке по артикулам/моделям — всегда LOWER().
8. **Выводимые — не анализировать.** Модели со статусом «выводим» исключаются.
