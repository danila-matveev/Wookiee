# Аудит логистики ООО — итоговые deliverables

**Date:** 2026-04-08
**Status:** Design approved
**Context:** Перерасчёт аудита логистики ООО Wookiee с 5 исправлениями, сверенный с расчётом финансистов. Расхождение 94,062 руб за период 05.01–01.02 полностью объяснено и воспроизведено (delta 0.95 руб на 1,479 строках).

---

## Цель

1. Зафиксировать результаты аудита и причины расхождений
2. Получить от финансистов подтверждение корректности алгоритма на периоде 01.01–23.03
3. После подтверждения — расширить до 31.03 и автоматизировать для любых кабинетов
4. Документировать алгоритм для масштабирования

## Подход

**Двухэтапный (C):**
- Этап 1 (сейчас): полный пакет на данных до 23.03 → отправляем финансистам
- Этап 2 (после их OK): запуск runner.py за 01.01–31.03 с WB API → финальный Excel

## Deliverable 1: Итоги аудита + постмортем (Notion)

**Аудитория:** Внутренняя команда + финансисты
**Формат:** Notion page

### Структура

1. **Резюме**
   - Таблица: старый расчёт → новый расчёт → разница, по периодам
   - Ключевая цифра: 174,499 → 24,373 руб (полный период 01.01–23.03)
   - Валидация: сверка 05.01–01.02 с файлом финансистов → delta 0.95 руб

2. **5 ошибок — разбор каждой**

   Для каждой ошибки:
   - Что было неправильно
   - Почему так получилось (root cause)
   - Как исправлено
   - Вклад в рублях
   - Как избежать в будущем

   | # | Ошибка | Root cause | Вклад (руб) |
   |---|---|---|---|
   | 1 | Обратная логистика в расчёте | Не было whitelist фильтра по bonus_type_name | 0 (fixed-rate rows = 0) |
   | 2 | Калиброванный тариф вместо периодного | tariff_calibrator.py reverse-calculated ~33.6 вместо period-based 46/32 | +86,601 |
   | 3 | Коэффициенты складов из Supabase вместо файла тарифов | Supabase ETL мог отставать, файл тарифов точнее | -3,399 |
   | 4 | ИЛ ≈ 1.0 вместо значений из дашборда WB (0.98–1.11) | calculate_weekly_il() считал из заказов, не из дашборда | -271,959 |
   | 5 | Отрицательные переплаты в итогах | Не было фильтра overpayment >= 0 | +38,630 |

3. **Валидация**
   - Таблица СВОД: наш старый vs наш новый vs финансисты — для каждого report_id за 05.01–01.02
   - Построчная сверка: 1,479 строк, max delta < 1 руб

4. **Риски автоматизации**

   | Риск | Описание | Митигация |
   |---|---|---|
   | ИЛ без API | В WB нет API для ИЛ, только дашборд | il_overrides.json — ручной ввод раз в неделю |
   | ETL тарифов | Новые склады / задержка ETL | tariff_collector.py --backfill + файл тарифов как fallback |
   | Смена тарифов WB | WB меняет base tariffs без предупреждения | Мониторинг tariff_periods.py + алерт при обнаружении нового периода |
   | Объёмы товаров | Расхождение карточка vs реальный | dimensions_checker.py + сверка с отчётом по остаткам |
   | IRP-формула | С 23.03.2026 новая формула с per-SKU localization | Отдельная валидация на данных после 23.03 |
   | Фиксация коэффициентов | Expired fixation → dlv_prc может содержать старое значение | 3-tier resolution: fixation → tariff file → dlv_prc |

5. **Рекомендации для масштабирования**
   - Чеклист для запуска на новом кабинете
   - Необходимые данные: API key, ИЛ из дашборда, файл тарифов складов
   - Частота запуска: еженедельно после закрытия отчётного периода

## Deliverable 4: Алгоритм расчёта (MD-документация)

**Аудитория:** Внутренняя команда + будущие разработчики
**Формат:** Markdown
**Файл:** `docs/logistics-audit-algorithm-v2.md`

### Структура

1. **Формула**
   ```
   cost = (base_1l + max(0, vol - 1) * extra_l) * coef * IL
   overpayment = delivery_rub - cost
   ```

2. **Параметры** — таблица: параметр, источник, приоритет, fallback

3. **Тарифные периоды** — таблица из tariff_periods.py

4. **Sub-liter тиры** — таблица для vol < 1L, order_date >= 22.09.2025

5. **3-tier коэффициент склада:**
   - Tier 1: Fixation active (fix_to > order_dt) → dlv_prc
   - Tier 2: Tariff file (closest date <= order_dt, по складу)
   - Tier 3: dlv_prc fallback

6. **ИЛ (Индекс Локализации):**
   - Источник: WB Partners → Тарифы → Индекс локализации → История
   - Применение: по дате заказа → неделя (понедельник) → ИЛ
   - Приоритет: il_overrides.json > calculate_weekly_il()

7. **Фильтрация строк:**
   - Вход: supplier_oper_name == "Логистика"
   - Whitelist: bonus_type_name in {К клиенту при продаже, К клиенту при отмене}
   - Итоги: только overpayment > 0

8. **Блок-схема обработки строки**
   ```
   Raw row → filter logistics → filter forward → get volume →
   → resolve coef (3-tier) → get tariffs (period) →
   → get IL (week lookup) → calculate cost → calculate overpay →
   → if overpay > 0: include in Переплата
   ```

9. **Входные данные:**
   - WB API reportDetailByPeriod v5 (или Excel "Детализация")
   - Файл тарифов складов (Тарифы на логискику.xlsx)
   - Габариты из карточек товаров (WB API или Excel)
   - ИЛ из дашборда WB (ручной ввод в il_overrides.json)

10. **Известные ограничения**
    - ИЛ: нет API, ручной ввод
    - IRP-формула: с 23.03.2026, требует отдельной валидации
    - Монопалеты: не покрыты (только короб)
    - DBS: не покрыт

## Deliverable 5: Верификация

**3 уровня проверки:**

### Уровень 1: Построчная сверка с файлом финансистов
- Вход: их "Переплата по логистике" (1,479 строк за 05.01–01.02)
- Алгоритм: для каждой строки — наш расчёт vs их расчёт
- Target: max |delta| < 1 руб по строке
- Результат: таблица с delta по каждой строке

### Уровень 2: Сверка СВОД по отчётам
- Вход: 8 report_id за 05.01–01.02 (наш СВОД vs их СВОД)
- Target: |delta| < 5 руб по каждому отчёту
- Результат: таблица сравнения

### Уровень 3: Кросс-валидация
- Фисанов: подтверждение что остаток < 1 руб (уже сходится: 0.96 руб)
- Полный период: итоговая сумма positive overpay < 5% от total delivery
