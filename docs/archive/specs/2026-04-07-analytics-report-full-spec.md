**Дата:** 2026-04-07
**Статус:** Финальный spec для реализации
**Цель:** Единый Claude Code скилл `/analytics-report` — замена Oleg v2

## 1. Обзор

Скилл `/analytics-report` анализирует ВСЕ аспекты бизнеса Wookiee за любой период: финансы, маркетинг (внутренний + внешний), трафик, воронку, остатки, цены, план-факт. Запускается вручную, позже автоматизируется через cron.

**Вызов:**
```
/analytics-report 2026-04-05                    → дневной
/analytics-report 2026-03-30 2026-04-05          → недельный
/analytics-report 2026-03-01 2026-03-31          → месячный
```

**Тестовый период:** 30.03 — 05.04.2026

## 3. Архитектура: команда субагентов

### 3.1 Субагенты-аналитики (параллельно)

#### Субагент 1: Финансовый аналитик
**Роль:** Полная финансовая воронка на всех уровнях

**Анализирует:**
- Выручка до/после СПП → Комиссия → Логистика → Хранение → Себестоимость → Реклама (внутр/внешн) → НДС → Штрафы → Удержания → **Маржа**
- ВСЕ метрики в **₽ + доля от выручки (%) + изменение vs пред. период**
- 3 уровня: бренд → канал (WB/OZON) → модель
- Для месячного: + артикулы внутри моделей (топ/анти-топ по маржинальности)
- Группировка моделей по статусам: "Продается" / "Выводим" / "Архив" / "Новинки"
- Сведение ΔМаржи: таблица какая статья расходов как повлияла на изменение маржи

**Данные:** `get_wb_finance`, `get_ozon_finance`, `get_wb_by_model`, `get_ozon_by_model`, `get_wb_by_article`, `get_ozon_by_article`, SKU-статусы из Supabase

**Правила:**
- Для "Выводим" — приоритет скорости распродажи, можно жертвовать маржой
- Для "Продается" — целевая маржа ≥ 20-25%
- WB маржа: `SUM(marga) - SUM(nds) - SUM(reclama_vn) - COALESCE(SUM(reclama_vn_vk), 0) - COALESCE(SUM(reclama_vn_creators), 0)`
- OZON маржа: `SUM(marga) - SUM(nds)`
- Единая маржа (single margin) — включает ВСЮ рекламу, никаких M-1/M-2

**Шаблон отчёта (из Oleg v2):**

Паспорт отчёта → Топ-выводы (3-5 ключевых находок с ₽ эффектом) → План-факт → Ключевые изменения (таблица 15 строк) → Сведение ΔМаржи → По маркетплейсам (WB/OZON) → По моделям → Драйверы/Антидрайверы → Итог

---

#### Субагент 2: Аналитик внутренней рекламы МП
**Роль:** Эффективность рекламы WB и OZON

**Анализирует:**
- ДРР с ОБЯЗАТЕЛЬНОЙ разбивкой: внутренняя (МП) + внешняя отдельно
- ROI по моделям: ROMI, CPC, CTR, CPO, CPM
- Связка реклама → заказы: ad↑ AND orders↑ = эффективно, ad↑ orders flat = неэффективно
- Бюджет vs факт по кампаниям
- Дневная динамика рекламы (показы, клики, CTR, расход, заказы, CPO)
- **Чёрные дыры рекламы:** модели с ROMI < 100%
- Матрица эффективности: Growth (ROMI>1000%) / Harvest (60-300%) / Optimize (<100%) / Cut (отрицат.)
- Средний чек и связь с ДРР
- Прогноз на следующий период

**Данные:** `wb_adv`, `adv_budget`, `ozon_adv_api`, `adv_stats_daily`, `get_wb_model_ad_roi`, `get_ozon_model_ad_roi`, `get_wb_ad_daily_series`, `get_ozon_ad_daily_series`, `get_wb_campaign_stats`, `get_wb_ad_budget_utilization`

**Бенчмарки из Oleg v2:**
| Метрика | Отлично | Норма | Плохо |
|---------|---------|-------|-------|
| CTR % | > 5% таргетированный | 2-3% | < 1% нецелевой |
| CPC ₽ | < 5₽ | 5-10₽ | > 15₽ |
| CPM ₽ | < 100₽ | 100-300₽ | > 500₽ |
| CR % | > 5% | 2-5% | < 1% |

**Двойной KPI рекламы (из Oleg v2):**
1. **ДРР продаж** = ad / sales_revenue × 100 — влияет на текущую маржу
2. **ДРР заказов (CPO)** = ad / orders_revenue × 100 — отражает будущую эффективность
- Высокий ДРР продаж НО нормальный ДРР заказов → инвестиция, НЕ отключать
- Высокий ДРР заказов → реклама неэффективна, снижать
- Оба высокие → СТОП и диагностика

#### Субагент 4: Аналитик трафика и воронки
**Роль:** Полная воронка от показов до выкупа

**Анализирует:**
- Воронка: показы → карточка → корзина → заказ → выкуп
- CR на каждом шаге + динамика + бенчмарки
- Органический vs платный трафик (НЕ суммировать — несравнимые метрики!)
- ASCII-визуализация воронки (из Oleg v2 marketer)
- Органика vs Платное: доли, динамика, конверсии
- По моделям: какие модели ad-dependent, какие органические
- ⚠️ Выкупы — ЛАГОВЫЙ показатель (3-21 дн)
- ⚠️ Трафик: ~20% расхождение с PowerBI — использовать как ТРЕНД, не абсолют

**Данные:** `content_analysis` (WB), `wb_adv`, `get_wb_organic_vs_paid_funnel`, `get_wb_traffic_by_model`, `get_ozon_traffic`

**Диагностика аномалий воронки (из Oleg v2):**
| Симптом | Проверить |
|---------|----------|
| CR карточка→корзина < 5% | Контент/фото, цена, конкуренты |
| CR корзина→заказ < 15% | Ценовая конкурентоспособность |
| Переходы ↓ >20% | Позиции, рекламный бюджет, сезонность |
| ДРР ↑ >3пп | Стоимость клика, конверсия рекламы |
| Органика ↓ >10пп | Позиции, новые конкуренты |
| Выкупы ↓ (заказы стабильны) | Качество товара, размеры |
| CAC > прибыль/продажу | Реклама убивает маржу |
| ROMI < 100% | Убыточная реклама |

#### Субагент 5: Аналитик остатков и оборачиваемости
**Роль:** Складские остатки, оборачиваемость, рекомендации по запасам

**Анализирует:**
- Остатки: WB FBO + OZON FBO + МойСклад (собственный склад Москва) + в пути
- Оборачиваемость = stock / daily_sales (дней запаса)
- Учёт скорости ЗАКАЗОВ, не только продаж
- Рекомендации по каждой модели: пополнить / ок / распродать / FREEZE / ликвидация

**Данные:** `get_wb_turnover_by_model`, `get_ozon_turnover_by_model`, `get_moysklad_stock_by_model`, `get_wb_avg_stock`, `get_ozon_avg_stock`

**Пороги риска (из Oleg v2):**
| Статус | Оборачиваемость | Действие |
|--------|-----------------|----------|
| ДЕФИЦИТ | < 14 дней | Пополнить WB FBO (МойСклад: X шт) |
| OK | 14-60 дней | Без действий |
| ВНИМАНИЕ | 60-90 дней | Мониторинг |
| OVERSTOCK | 90-250 дней | Снизить цену / FREEZE отгрузки |
| МЁРТВЫЙ СТОК | > 250 дней | Маркдаун / ликвидация |

**Ограничения по складу и цене (из Oleg v2):**
| Статус запаса | Ценовое ограничение |
|--------------|---------------------|
| critical_low (<1 нед) | НЕ снижать цену |
| low (<2 нед) | НЕ снижать цену |
| healthy (2-8 нед) | Без ограничений |
| overstocked (>8 нед) | Предпочтительно снижение |
| severely_overstocked (>16 нед) | Принудительное снижение |

#### Субагент 6: Аналитик цен и ценовых гипотез
**Роль:** Ценовая динамика, СПП, рекомендации по ценам

**Анализирует:**
- Динамика цен по моделям/артикулам
- СПП: НЕ управляемый, но влияет на спрос (рост СПП → рост заказов)
- Гипотезы "что если": эффект изменения цены на маржу в ₽
- Годовой ROI = margin% × (365 / turnover_days)
- Цена и маркетинг — ЕДИНСТВЕННЫЕ инструменты управления
- Комиссия — НЕ управляемая

**Данные:** `get_wb_price_changes`, `get_wb_spp_history_by_model`, `get_wb_price_margin_by_model_period`, `get_ozon_price_changes`

**Ценовая логика (из Oleg v2):**
1. Overstock + нормальная маржа (>15%) → снизить цену (ускорить продажи)
2. Дефицит + нормальная маржа → повысить цену (спрос > предложение)
3. Низкая маржа (<15%) → НЕ снижать (даже при слабых продажах)
4. Мёртвый сток (>250д) → маркдаун / ликвидация
5. Мало данных → держать, наблюдать

**Confidence discipline (из Oleg v2):**
- HIGH confidence (|r| > 0.5, > 30 дней данных) → можно рекомендовать CUT
- MEDIUM → только HOLD или тест 1 SKU на 2 недели
- LOW → всегда HOLD

**Годовой ROI (из Oleg v2):**
| Категория | ROI | Интерпретация |
|-----------|-----|---------------|
| roi_leader | > 500% | Отлично: высокая маржа + быстрая оборачиваемость |
| healthy | 200-500% | Норма: стабильный бизнес |
| underperformer | 50-200% | Проблема: медленная оборачиваемость или низкая маржа |
| deadstock_risk | < 50% | Критично: деньги заморожены |

#### Субагент 7: Аналитик план-факт
**Роль:** Сопоставление факта с планом, рекомендации по выполнению

**Анализирует:**
- План поартикульно WB + OZON (из Google Sheets)
- Факт MTD, план MTD (пропорциональный), выполнение MTD %, прогноз, прогноз vs план %
- Все метрики: заказы (шт/₽), продажи (шт/₽), выручка, маржа, реклама (внутр/внешн)
- Статусы: ✅ >105% | ⚠️ 95-105% | ❌ <95%
- Рекомендации действий для выполнения/перевыполнения

**Данные:** Google Sheets `1Dsz7s_mZ0wUhviGFho89lyhtjce1V0Cmv_RPL1aLxnk` (gid=1957404595) + `get_plan_vs_fact` из data_layer

**ОБЯЗАТЕЛЕН** в дневном, недельном И месячном отчёте.

#### Субагент 8: Детектор аномалий
**Роль:** Поиск значимых отклонений, проверка на ошибки данных

**Анализирует:**
- Если доля метрики изменилась > 3 п.п. → флаг
- Если себестоимость резко меняется в долях (20% → 25%) — вероятно ошибка
- Все значимые изменения по ₽ эквиваленту
- Отдельный анализ логистики: тарифы, индекс локализации
- Data quality: retention == deduction (дубликаты), fan-out JOIN

**Данные:** Результаты всех остальных субагентов + `validate_wb_data_quality`

**Severity (из Oleg v2):**
- high: delta > 5 п.п. ИЛИ > 100К₽
- medium: delta 3-5 п.п.

**Глубокая диагностика модели при значительном изменении (±10%+ заказов) — из Oleg v2:**
1. Цена до/после СПП — изменения за последние 3 дня
2. Наличие на складе (OOS) — выпали ли ключевые размеры?
3. Органический трафик — показы/переходы/конверсия карточки
4. OZON отдельно — оборачиваемость, хранение, наличие по цветам/размерам

### 3.2 Субагенты-верификаторы (последовательно после аналитиков)

#### Субагент 9: Эксперт по маркетплейсам (WB-специалист)
**Роль:** Проверка корректности интерпретации данных

**Проверяет:**
- СПП не управляемый → нет рекомендаций по СПП
- Выкупы лаговые → не используются как причина в дневном анализе
- Комиссия фиксирована → нет рекомендаций по комиссии
- Рекомендации реалистичны для маркетплейсов
- Специфика WB "склейки" — заказы Wendy могут быть от связанных артикулов
- Корректность формулировок по внешней рекламе (системность каналов)

#### Субагент 10: Финансовый директор (CFO)
**Роль:** Главный верификатор, сведение всего воедино

**Проверяет (из Oleg v2 validator):**
1. **Точность цифр:** числа в тексте = данные (±1% допуск)
2. **Покрытие:** все warning/critical сигналы покрыты рекомендациями
3. **Направление:** действие логически соответствует проблеме
4. **KB-правила:** нет конфликтов с правилами аналитики
5. **Реалистичность:** ожидаемый эффект достоверен
- Может ОТКЛОНИТЬ и запросить пересчёт (макс 1 retry)

#### Субагент 11: Data Quality Critic
**Роль:** Проверка данных, арифметика, консистентность

**Проверяет (из Oleg v2 data-quality-critic):**
1. **Арифметика:** сумма моделей = итого бренда (±1%)
2. **ДРР:** обязательно с разбивкой внутр/внешн
3. **СПП:** средневзвешенный при объединении каналов (НЕ среднее арифметическое)
4. **GROUP BY LOWER():** нет дублей моделей с разным регистром
5. **Единая маржа:** включает ВСЮ рекламу, нет M-1/M-2
6. **Выкуп %:** не используется как причина в дневном анализе
7. **Доли складываются:** комиссия + логистика + ... + маржа ≈ 100% (±2%)
8. **Нет артефактов:** "0%", "—", "NaN", "null" вместо данных

## 4. Структура отчёта

### 4.1 Обязательные секции (для ВСЕХ глубин)

**0. Паспорт отчёта**
Таблица: Период | Сравнение | Тип | Полнота данных | Лаг

**1. Топ-выводы и действия**
3-5 ключевых находок с ₽ эффектом. Формат: [₽ эффект] Что → Гипотеза → Действие

**2. План-факт** (ОБЯЗАТЕЛЕН ВСЕГДА)
Таблица: Метрика | План (месяц) | Факт MTD | План MTD | Выполнение MTD % | Прогноз | Статус
Метрики: заказы шт, заказы ₽, выручка, маржа, реклама внутр, реклама внешн

**3. Ключевые изменения (Бренд)**
ПОЛНАЯ таблица (15 строк, без сокращений):
Маржинальная прибыль, Маржинальность %, Продажи шт/руб, Заказы руб/шт, Реклама внутр/внешн, ДРР от заказов/продаж, Средний чек заказов/продаж, Оборачиваемость, Годовой ROI %, СПП %

**4. Цены, СПП, ценовые гипотезы**
- Динамика СПП по каналам
- Средние цены + прогноз
- Рекомендации по ценам (конкретные модели с ₽ эффектом)

**5. Сведение ΔМаржи (Reconciliation)**
Таблица: Статья | Текущий | Прошлый | Δ | Влияние на маржу (₽)
Строки: Выручка, Себестоимость/ед, Комиссия до СПП, Логистика/ед, Хранение/ед, Реклама внутр, Реклама внешн, НДС, Прочие, Невязка

**6. По маркетплейсам**

## 6. Правила аналитики (полный свод)

### Жёсткие правила (НАРУШЕНИЕ = ОШИБКА)
1. **GROUP BY модели** = `LOWER(SPLIT_PART(article, '/', 1))` — ВСЕГДА
2. **Процентные метрики** при объединении каналов = ТОЛЬКО средневзвешенные: `sum(X) / sum(Y) × 100`
3. **ДРР** = ВСЕГДА с разбивкой: внутренняя (МП) + внешняя (блогеры, ВК) ОТДЕЛЬНО
4. **Единая маржа** = включает ВСЮ рекламу (internal + external). Никаких M-1/M-2
5. **WB маржа** = `SUM(marga) - SUM(nds) - SUM(reclama_vn) - COALESCE(SUM(reclama_vn_vk), 0) - COALESCE(SUM(reclama_vn_creators), 0)`
6. **OZON маржа** = `SUM(marga) - SUM(nds)`
7. **Органик + Paid** = НЕ суммировать (несравнимые метрики)
8. **Числа** — ТОЛЬКО из данных, никогда не придумывать
9. **Факт vs Гипотеза** — ВСЕГДА разделять
10. **Все рекомендации** = только через ЦЕНУ или МАРКЕТИНГ (единственные инструменты)
11. **Формат чисел** — `1 234 567 ₽`, проценты `24.1%`, доли `8.0% → 11.2% (+3.2 п.п.)`
12. **Даты в тексте** — "17 марта 2026". В таблицах "17.03.2026". ЗАПРЕЩЁН формат YYYY-MM-DD

### Лаговые/неуправляемые показатели
- **Выкуп %** — лаг 3-21 дней. ЗАПРЕЩЁН как причина в дневном анализе. В недельном — с оговоркой. В месячном — полностью надёжен.
- **СПП** — НЕ управляемый. Рост СПП → рост спроса → рост заказов. Влияет на конечную цену покупателя.
- **Комиссия** — НЕ управляемая. Фиксируем, но не рекомендуем действия.

### Аналитические принципы (из Oleg v2 playbook)
- **Системное мышление:** изменение одного фактора каскадирует (цена↓ → конверсия↑ → заказы↑ → РИСК: исчерпание запасов → падение органических позиций)
- **Нелинейность:** не шаблон "маржа↓ → повысить цену". Может быть выгоднее снизить цену ради оборачиваемости
- **Evidence-based:** каждая гипотеза начинается с факт-триггера из текущих данных
- **Эффект в деньгах:** "Ожидаемый эффект" = влияние на маржу в ₽ по конкретной модели
- **Последствия 2-го порядка:** каждое действие имеет обратную сторону — заполнять "Риски"
- **Приоритизация по ₽:** все рекомендации отсортированы по рублёвому эффекту (не %)
- **Trade-off прозрачность:** показывать ОБЕ стороны (объём vs маржа%, рост vs эффективность)
- **Тон:** сухой финансовый директор. Без оценочных суждений ("к сожалению", "радует"). Факты + ₽ эффект.

### Data Quality
- retention == deduction на одних датах = дубликат, коррекция маржи
- Fan-out JOIN nomenclature = рекламный расход ×4, использовать SELECT DISTINCT nmid
- Трафик: ~20% gap с PowerBI — использовать как ТРЕНД
- OZON search_stat organic_data = 0 — органика недоступна
