# Modular Analytics System v2 — Design Spec

**Дата:** 2026-04-08
**Статус:** Утверждён для реализации
**Предшественник:** v1 `/analytics-report` (12 субагентов, поверхностный анализ)

---

## 1. Обзор

Модульная система аналитики: 3 глубоких модуля (финансы, маркетинг, воронка) + мета-оркестратор + аналитический движок.

**Вызовы:**
```
/analytics-report 2026-03-30 2026-04-05    → все модули + сводка
/finance-report 2026-03-30 2026-04-05      → только финансы (deep)
/marketing-report 2026-03-30 2026-04-05    → только маркетинг (deep)  
/funnel-report 2026-03-30 2026-04-05       → только воронка WB (deep)
```

Период: 1 дата = день, ≤14 дней = неделя, >14 = месяц.

## 1.2. Иерархия анализа

Каждый модуль анализирует данные на ВСЕХ уровнях иерархии:

```
Бренд (WB + OZON суммарно)
├── Канал: WB
│   ├── Статус: Продается (целевая маржа ≥20-25%)
│   │   ├── Модель: Wendy
│   │   │   ├── Артикул: wendy/dark_beige → ЛОКОМОТИВ (топ по заказам)
│   │   │   ├── Артикул: wendy/fig → ПРОБЛЕМА (падение -48%)
│   │   │   └── ...
│   │   ├── Модель: Ruby
│   │   └── ...
│   ├── Статус: Выводим (приоритет: скорость распродажи)
│   │   ├── Модель: Other
│   │   └── Модель: Set Wendy
│   ├── Статус: Архив
│   └── Статус: Запуск (инвестиционная фаза)
│       └── Модель: Charlotte
├── Канал: OZON
│   └── (аналогичная структура)
```

**Обязательные разрезы в каждом отчёте:**
1. **Бренд** — сводные цифры WB+OZON
2. **По каналам** — WB и OZON отдельно с полным P&L
3. **По статусам** — Продается / Выводим / Архив / Запуск (суммарные метрики по группе)
4. **По моделям** — ВСЕ модели из Supabase `get_model_statuses_mapped()`
5. **По артикулам** — значимые артикулы (топ/анти-топ по Δ)
6. **Локомотивы** — артикулы с наибольшим вкладом в маржу (топ-3 на модель)

**Что анализировать на каждом уровне:**
- Выручка, маржа, маржинальность%, заказы шт/₽, продажи шт/₽
- Структура затрат (доли от выручки): COGS%, комиссия%, логистика%, хранение%, реклама%, НДС%, штрафы%
- ДРР (internal + external отдельно)
- Оборачиваемость, годовой ROI
- Юнит-экономика: маржа/ед, логистика/ед, COGS/ед, CAC

**Ключевая цель анализа:** повышение маржинальной прибыли без потери оборачиваемости. Два рычага: ЦЕНА (вверх/вниз) и МАРКЕТИНГ (бюджет/эффективность).

## 1.3. Analytics Knowledge Base (для агентов)

Единая база знаний — файл `references/analytics-kb.md`, который подключается ко ВСЕМ субагентам. Содержит:

### Жёсткие правила (нарушение = ошибка)
1. GROUP BY модели = `LOWER(SPLIT_PART(article, '/', 1))`
2. Процентные метрики при объединении каналов = ТОЛЬКО средневзвешенные
3. ДРР = ВСЕГДА с разбивкой: внутренняя + внешняя ОТДЕЛЬНО
4. Единая маржа: WB = `marga - nds - reclama_vn - reclama_vn_vk - reclama_vn_creators`; OZON = `marga - nds`
5. Органик + Paid = НЕ суммировать
6. Формат: `1 234 567 ₽`, `24,1%`, `8,0% → 11,2% (+3,2 п.п.)`
7. Даты: "17 марта 2026" в тексте, "17.03.2026" в таблицах

### Особенности WB
- **Склейки:** заказы модели могут включать связанные артикулы
- **Воронка content_analysis:** расхождение ~20% с PowerBI (использовать как тренд)
- **Реклама:** wb_adv содержит показы, клики, расход — fan-out при JOIN с nomenclature (×4)
- **СПП:** не управляемый, но влияет на спрос. Рост СПП → рост заказов
- **Комиссия:** фиксирована по категории, не управляема
- **retention == deduction:** дубликат в abc_date, использовать только одно поле

### Особенности OZON
- **Органика:** `search_stat organic_data = 0` — органический трафик через API недоступен
- **Комиссии:** 34 типа в `ozon_services`, объединять через abc_date
- **Реклама:** OZON adv уже включена в marga через abc_date

### Лаговые показатели
- **Выкуп %:** лаг 3-21 день. ЗАПРЕЩЁН как причина в дневном анализе. В недельном — с оговоркой.
- **Блогеры:** ROI с лагом 3-7 дней. Дневной ДРР блогера ЗАПРЕЩЁН.
- **Цена заказов vs продаж:** разница = будущий рост/падение выручки (лаг выкупа)

### Бенчмарки
| Метрика | Отлично | Норма | Плохо |
|---|---|---|---|
| CTR (WB) | >5% | 2-3% | <1% |
| CPC (WB) | <5₽ | 5-10₽ | >15₽ |
| CPM (WB) | <100₽ | 100-300₽ | >500₽ |
| CR (WB) | >5% | 2-5% | <1% |
| Маржинальность | >25% | 15-25% | <15% |
| Оборачиваемость | <60дн | 60-120дн | >120дн |

### Юнит-экономика по статусам
| Статус | Целевая маржа | Стратегия | Ценовой рычаг |
|---|---|---|---|
| Продается | ≥20-25% | Рост оборота + маржи | Да |
| Выводим | Любая | Максимальная скорость распродажи | Снижать агрессивно |
| Архив | — | Минимальная активность | Маркдаун |
| Запуск | ≥15% (инвестиция) | Набор скорости | Осторожно |

### Ценовая логика (decision tree)
1. OVERSTOCK + маржа >15% → снизить цену (ускорить оборот)
2. DEFICIT + маржа ок → повысить цену (спрос > предложение)
3. Маржа <15% → НЕ снижать (даже при слабых продажах)
4. Мёртвый сток >250дн → маркдаун / ликвидация
5. DEFICIT + Выводим → не пополнять, дождаться вымывания

### Матрица эффективности рекламы
| Категория | ROMI | Действие |
|---|---|---|
| Growth | >1000% | Масштабировать бюджет |
| Harvest | 300-1000% | Поддерживать |
| Optimize | 100-300% | Оптимизировать ставки/ключи |
| Cut | <100% | Отключить |

### Диагностика аномалий воронки
| Симптом | Проверить |
|---|---|
| CR карточка→корзина <5% | Контент/фото, цена, конкуренты |
| CR корзина→заказ <15% | Ценовая конкурентоспособность, наличие размеров |
| Переходы ↓ >20% | Позиции, рекламный бюджет, сезонность |
| Заказы ↓ при CR стабильном | Падение трафика (органик или реклама) |
| Маржа ↓ при выручке стабильной | Рост затратных статей (COGS, логистика, реклама) |

### Запрещённые формулировки
- "Внешняя реклама прекратилась" → "Сегодня без внешней рекламы"
- "Экономия на внешней рекламе" → нельзя (несистемный расход)
- "ДРР блогера 45%" → "Размещение X₽. ДРР считать рано (лаг 3-7 дн)"
- "Повысить/снизить СПП" → СПП не управляемый
- "Договориться о комиссии" → комиссия фиксирована

## 2. Архитектура — 3 слоя

```
Слой 1: ДАННЫЕ
  collect_all.py → JSON (~500KB)
  (Python, ThreadPoolExecutor, уже реализован)

Слой 2: АНАЛИТИЧЕСКИЙ ДВИЖОК (НОВЫЙ)
  Wave A: Детекция — "что изменилось?" (субагенты находят значимые Δ)
  Wave B: Диагностика — "почему?" (субагенты ищут причины)
  Wave C: Гипотезы — "что будет / что делать?" (субагенты формулируют действия)

Слой 3: МОДУЛИ ОТЧЁТОВ
  Каждый модуль берёт результаты движка + raw data → генерирует глубокий отчёт → Notion
```

### Поток данных

```
/analytics-report 2026-03-30 2026-04-05
  │
  ├─ Слой 1: collect_all.py → /tmp/analytics-data.json
  │
  ├─ Слой 2: Аналитический движок (3 волны субагентов)
  │   ├─ Wave A: 3 детектора (фин + маркет + воронка) → findings_raw
  │   ├─ Wave B: 1 диагност (cross-domain причины) → diagnostics
  │   └─ Wave C: 1 стратег (гипотезы + действия) → hypotheses
  │
  └─ Слой 3: 3 модуля параллельно
      ├─ finance-report → Notion #1
      ├─ marketing-report → Notion #2
      └─ funnel-report → Notion #3
      
  Chat: Executive Summary (5-7 строк + ссылки на 3 Notion-страницы)
```

При вызове одного модуля (напр. `/finance-report`):
```
collect_all.py → Wave A (только финансовый детектор) → Wave B → Wave C → finance-report → Notion
```

## 3. Слой 2: Аналитический движок

### Wave A: Детекция — "ЧТО изменилось?"

3 субагента-детектора (параллельно), каждый сфокусирован на своей области:

**Финансовый детектор:**
- Сканирует все финансовые метрики (выручка, маржа, COGS, логистика, хранение, комиссия, НДС, реклама)
- На уровнях: бренд → канал → модель → артикул
- Порог значимости: Δ доли >3 п.п. ИЛИ Δ в ₽ >50К₽ (неделя) / >10К₽ (день)
- Выход: список `{метрика, уровень, объект, текущий, прошлый, Δ, Δ%, severity}`

**Маркетинговый детектор:**
- Сканирует: ДРР (internal+external), ROMI, CPO, CTR, CPC по моделям и кампаниям
- Внешний маркетинг: блогеры (из Google Sheets), ВК, Яндекс ЯПС/АДС/АДБ
- Порог: Δ ДРР >2 п.п. ИЛИ ROMI изменился на >30% ИЛИ CPO вырос >20%
- Выход: список значимых изменений + матрица Growth/Harvest/Optimize/Cut

**Воронковый детектор:**
- Сканирует: CR каждого шага (показы→карточка→корзина→заказ→выкуп) по КАЖДОЙ модели
- Список моделей: из Supabase (get_model_statuses_mapped)
- Порог: Δ CR >1 п.п. ИЛИ Δ заказов >15%
- Значимые артикулы: рост/падение переходов >30% или заказов >30%
- Выход: per-model findings + per-article flags

### Wave B: Диагностика — "ПОЧЕМУ это произошло?"

1 субагент-диагност получает ВСЕ findings из Wave A и ищет **cross-domain причины**:

Логика:
- Маржа модели упала → Проверить: цена изменилась? COGS вырос? Реклама выросла? Логистика?
- Заказы модели упали → Проверить: переходы упали? CR упал? OOS (нет размеров)? Конкуренты?
- ДРР вырос → Проверить: бюджет увеличен? CPO вырос? Заказы от рекламы упали?
- CR корзина→заказ упал → Проверить: цена выросла? Размеры закончились? Доставка замедлилась?

**Расчёт эффекта в ₽:**
Для каждого значимого изменения: "если восстановить метрику до прошлого уровня → +X₽ маржи"

Выход: `{finding_id, причина (1-2 предложения), confidence, ₽_эффект, связанные_findings}`

### Wave C: Стратег — "ЧТО ДЕЛАТЬ?"

1 субагент-стратег формулирует действия на основе диагностики:

Формат каждой гипотезы:
```
Приоритет: P0/P1/P2/P3
Объект: модель/канал
Факт: что изменилось (из Wave A)
Причина: почему (из Wave B)
Действие: что сделать
Метрика контроля: что мониторить
База → Цель: текущее значение → целевое
Эффект: +X₽ маржи/мес
Окно: срок проверки
Риски: что может пойти не так
```

## 4. Слой 3: Модули отчётов

### Модуль 1: `/finance-report` (deep)

**Эталон:** Фин анализ за 17 марта 2026 (Олег v2)

**I. Паспорт отчёта**
Таблица: период, сравнение, тип, полнота данных, лаг, невязка

**II. Топ-выводы и действия**
Таблица: ₽ эффект | Что → Гипотеза → Действие (из Wave C, отфильтровано по финансам)

**III. План-факт**
Таблица: метрика | план (месяц) | факт MTD | план MTD | выполнение MTD% | прогноз | статус
Данные плана из Google Sheets `1Dsz7s_mZ0wUhviGFho89lyhtjce1V0Cmv_RPL1aLxnk`

**IV. Ключевые изменения (бренд)**
15+ строк: маржа, маржин-ть, продажи шт/₽, заказы шт/₽, реклама внутр/внешн, ДРР заказов/продаж, ср.чек, оборачиваемость, ROI год, СПП, **переходы в карточку, корзина, CR переход→корзина, CR корзина→заказ**
+ Интерпретация текстом (3-5 строк)

**V. Цены, СПП, стратегия**
Таблица 5а: динамика СПП по каналам
Таблица 5б: средние цены + прогноз 3-7 дней
Интерпретация + рекомендации

**VI. Сведение ΔМаржи (Reconciliation)**
Waterfall-таблица: фактор | текущий | прошлый | Δ | влияние на маржу ₽
Строки: выручка, COGS/ед, комиссия, логистика/ед, хранение/ед, реклама внутр, реклама внешн, прочие, НДС, невязка
+ Интерпретация (главный позитив/негатив)

**VII. Wildberries (deep)**
- 7.1 Объём и прибыльность WB (таблица P&L)
- 7.2 Модельная декомпозиция WB — **ВСЕ модели**, таблица:
  Модель | Маржа₽ | МаржаΔ% | Маржин-ть% | Остаток FBO | Свой склад (МойСклад) | В пути | Итого остаток | Оборач.(FBO) | Оборач.(все) | ROI год% | ДРР продаж% | Комментарий
- 7.3 Воронка WB (объём: показы/переходы/корзина/заказы + эффективность: CTR/CR каждого шага)
- 7.4 **Факторное разложение заказов WB** (за счёт показов / CTR / конверсии — 3 фактора с вкладом в шт)
- 7.5 Структура затрат WB (% от выручки до СПП + Δ п.п. по каждой статье)
- 7.6 Реклама WB (итоги внутр/внешн + детальная статистика: показы/клики/CTR/CPC/CPM/CPO/расход)

**VIII. OZON (deep)**
Аналогичная структура (без факторного разложения — нет данных по органической воронке)

**IX. Драйверы WB**
Таблица: модель | доля продаж | Δмаржа | маржа(тек) | маржин-ть% | Δмаржин-ть | Δпродажи₽ | Δзаказы₽ | ДРР продаж | ДРР заказов | внутр.рекл | внешн.рекл | комментарий

**X. Антидрайверы WB**
Аналогичная таблица

**XI. Драйверы/Антидрайверы OZON**
Аналогичные таблицы

**XII. Гипотезы → Действия**
Таблица: приоритет | объект | гипотеза | действие | метрика контроля | база | цель | эффект | окно | риски
(из Wave C, отфильтровано по финансам)

**XIII. Advisor**
🔴 Критичные / 🟡 Внимание / 🟢 Позитивные — с confidence%

**XIV. Итог**
Что изменилось / Почему / Что двигало / Что делать (1-3 действия)

**Краткий блок маркетинга в финансовом отчёте:**
В секции VII.6 и VIII: основные цифры рекламы (расход, ДРР, CPO, Δ) + ссылка "подробнее в маркетинговом отчёте"

### Модуль 2: `/marketing-report` (deep)

**Эталон:** Маркетинговый анализ за 16-22 марта 2026 (Олег v2)

## 8. Расширяемость

Добавить новый модуль (напр. `/logistics-report`, `/pricing-report`):
1. Создать `.claude/skills/{name}/SKILL.md` + промпты
2. Добавить детектор в Wave A (если нужны новые данные)
3. Зарегистрировать в мета-оркестраторе
4. Коллектор: добавить новый модуль в `collectors/` если нужны новые данные
