# Еженедельный отчёт "Анализ логистических расходов WB" — спецификация

> Дата: 25.03.2026. Статус: Draft.

## Проблема

С 23.03.2026 WB ввёл ИРП (Индекс Распределения Продаж) — процент от цены товара, который взимается при низкой локализации (<60%). Вместе с ИЛ (Индекс Локализации) это создаёт двойную наценку на логистику для артикулов с плохим распределением по складам.

Сейчас нет регулярного отчёта, который показывает:
- Сколько мы переплачиваем за логистику из-за ИРП и ИЛ
- Как меняется ситуация неделя к неделе
- Какие конкретные модели дают наибольшую переплату и какую экономию даст их оптимизация

## Решение

Еженедельный Notion-отчёт "Анализ логистических расходов" (понедельник, 13:00 МСК) + лист "ИРП-анализ" в Google Sheets.

Отчёт встраивается в существующий pipeline скрипта локализации (подход A): `run_service_report()` → форматирование в MD → доставка через `agents/v3/delivery/` (Notion + Telegram).

## Scope

| В scope | Вне scope |
|---------|-----------|
| Обновление КТР-коэффициентов (27.03.2026) | Историческая база тарифов WB в Supabase (отдельная задача) |
| Расширение History для ИРП-динамики | Автоматизация перестановок через WB API |
| Google Sheets лист "ИРП-анализ" | Литраж из WB Content API (пока средний тариф) |
| Notion-отчёт с динамикой и экономией | |
| Telegram-summary | |

### Файл

### Предусловие

### Новая функция `_write_irp_analysis(spreadsheet, cabinet, result, meta)`

**Шапка (строки 1-6):**
| Метрика | Значение |
|---------|----------|
| ИЛ текущий | 0.98 |
| ИРП текущий | 0.42% |
| ИРП-переплата | 45 200 ₽/мес |
| SKU в ИРП-зоне | 12 |
| SKU в ИЛ-зоне | 30 |

**Таблица (строка 8+):**

Данные из `result['_sku_stats']`, сгруппированные по модели (артикул без размера), отфильтрованные по `Зона != 'OK'` или `КТР > 1.0`.

| Модель | Индекс% | КТР | КРП% | Зона | Ср. цена | Заказов | ИРП-нагрузка ₽/мес | Потенц. экономия |
|--------|---------|-----|------|------|----------|---------|---------------------|-------------------|

**Группировка по модели:**
- Артикул без размера = модель (например, `Joy/shinny_pink`)
- Суммируем заказы, ИРП-нагрузку по всем размерам
- Средневзвешенный индекс, КТР, КРП
- Средняя цена

**Потенциальная экономия:**
- Если артикул в ИРП-зоне (<60%): экономия = текущая ИРП-нагрузка (при достижении 60% КРП → 0)
- Если артикул в ИЛ-зоне (60-74%): экономия = (КТР - 0.9) × базовая_логистика × заказы/мес (при достижении 75%)

Сортировка по убыванию потенциальной экономии.

**Вызов:** из `export_to_sheets()` после `_write_top_problems()`.

## 4. Notion-отчёт "Анализ логистических расходов"

### 4a. Форматировщик

**Новый файл:** `services/wb_localization/report_md.py`

Детерминистический рендеринг (без LLM). Две функции:

#### `format_localization_weekly_md(results: list[dict], period_days: int) -> str`

Принимает список результатов (по одному на кабинет, из `run_service_report()`). Возвращает Markdown.

**Структура отчёта:**

```markdown
# Анализ логистических расходов WB

> Период индексов: {91 дн, DD.MM — DD.MM}. Динамика: неделя DD.MM — DD.MM.

## Сводка по кабинетам

| Кабинет | ИЛ | ИРП | Индекс лок. | SKU в ИРП-зоне | Переплата ₽/мес |
|---------|-----|------|------------|----------------|-----------------|
| ИП      | 0.98 | 0.42% | 73.5% | 12 | 45 200 |
| ООО     | 1.01 | 0.15% | 81.2% | 5  | 12 100 |

Переплата = ИРП-нагрузка (КРП% × цена × заказы) + ИЛ-наценка ((КТР-1) × тариф × заказы).

## Динамика за неделю

### {кабинет}
- Индекс локализации: 73.5% → 74.1% (+0.6 п.п.) ✓
- ИЛ: 0.98 → 0.97 (улучшение)
- ИРП: 0.42% → 0.38% (−0.04 п.п.)
- Переплата: 45 200 → 38 900 ₽/мес (экономия 6 300 ₽/мес)
- SKU в ИРП-зоне: 12 → 10 (−2)
- Улучшенные регионы: ЦФО, ПФО
- Ухудшенные регионы: —

## Зональная разбивка

| Зона | Описание | SKU ИП | SKU ООО | Доля заказов |
|------|----------|--------|---------|-------------|
| ИРП-зона | <60%, КРП > 0 | 12 | 5 | 18% |
| ИЛ-зона | 60-74%, КТР > 1 | 30 | 15 | 25% |
| OK | ≥75% | 150 | 120 | 57% |

## Топ моделей по переплате

| # | Модель | Кабинет | Лок% | КТР | КРП% | Заказов/мес | Логистика ₽/мес | Переплата ₽/мес | Экономия при оптимизации |
|---|--------|---------|------|-----|------|-------------|-----------------|-----------------|--------------------------|
| 1 | Joy/shinny_pink | ООО | 41% | 1.30 | 2.10% | 170 | 28 500 | 12 800 | 12 800 (при 60%: КРП→0) |

Экономия при оптимизации:
- ИРП-зона → 60%: убирается КРП полностью
- ИЛ-зона → 75%: КТР снижается до 0.90

## Регионы

| Регион | Лок% ИП | Лок% ООО | Доля заказов | Рекомендация |
|--------|---------|----------|-------------|--------------|

## Рекомендации

Автоматически генерируются на основе данных:
- Если есть артикулы близко к порогу 60%: "X моделей в 5 п.п. от порога 60% — приоритет перестановок"
- Если регион с низкой локализацией: "Регион {X}: локализация {Y}%, {Z}% заказов — дефицит остатков"
- Общая экономия: "При оптимизации топ-10 моделей экономия составит ~{X} ₽/мес"
```

<b>Логистические расходы WB</b> (неделя DD.MM—DD.MM)

### 4b. Scheduler job

**Файл:** `agents/v3/scheduler.py`

Новая async-функция `_job_localization_weekly()`:

```
1. Проверить state.is_delivered("localization_weekly", date_to) → skip если уже доставлен
2. Для каждого кабинета (последовательно, пауза 60с):
   a. run_service_report(cabinet, days=91, history_store=history)
   b. Собрать результат
3. Загрузить текущие тарифы через WB API `/api/v1/tariffs/box` (утилита в `shared/clients/wb_client.py` → `get_box_tariffs()`)
4. format_localization_weekly_md(results, period_days=91)
5. format_localization_tg_summary(results)
6. Обернуть в envelope: {"status": "success", "report": {"detailed_report": md, "telegram_summary": tg}}
7. _deliver(envelope, "localization_weekly", date_from, date_to)
8. state.mark_delivered("localization_weekly", date_to)
```

**Расписание:** Monday 13:00 MSK (после основных отчётов v3)

**Обработка ошибок:** try/except с логированием. Если один кабинет упал — продолжить со вторым, отчёт с пометкой "данные по кабинету X недоступны".

## Расчёт переплаты — формулы

### ИРП-переплата (для артикулов с КРП > 0)

```
ИРП_переплата = КРП% / 100 × Цена_товара × Заказов_в_мес
```

### ИЛ-наценка (для артикулов с КТР > 1.0)

```
ИЛ_наценка = (КТР - 1.0) × Базовая_логистика × Заказов_в_мес
```

Где `Базовая_логистика` — средний тариф логистики. В первой версии: загружается через WB Tariffs API при запуске отчёта. Когда появится историческая база тарифов в Supabase — переключится на неё.

### Потенциальная экономия

- **ИРП-зона → 60%:** экономия = вся ИРП_переплата (КРП → 0%)
- **ИЛ-зона → 75%:** экономия = (КТР - 0.90) × базовая_логистика × заказов/мес

### Группировка по модели

Модель = артикул без размера (split по `/` или общий prefix).
Суммируем заказы и финансовые метрики по всем размерам одной модели.
