# WB Logistics Audit Service — Спецификация v3

> Сервис аудита логистических затрат Wildberries: автоматическая выгрузка данных через API, расчёт переплат по формуле из оферты ВБ, формирование итогового Excel-документа для юристов.

## 1. Цель и результат

**Вход:** API-ключ продавца WB + КТР (ручной ввод, для периода до 23.03.2026) + период (даты)

**Выход:** Excel-документ (11 листов), аналогичный файлу подрядчика (`ИП Фисанов. Проверка логистики.xlsx`)

**Что показывает:**
- Сколько WB списал за логистику
- Сколько должен был списать по формуле из оферты
- Разница = переплата (или недоплата)
- Разбивка по складам, артикулам, типам доставки, неделям

## 2. Формула расчёта (из оферты ВБ, подтверждена на реальных данных)

### 2.1 Основная формула (ЗАВИСИТ ОТ ДАТЫ)

**ВАЖНО:** С 23 марта 2026 WB изменил формулу расчёта стоимости логистики. Сервис должен использовать правильную формулу в зависимости от даты заказа (`order_dt`).

#### Формула ДО 23.03.2026 (старая — КТР)

```
Стоимость логистики = (стоимость_1л + стоимость_доп_литров) × коэф_логистики × КТР

Где:
- стоимость_1л = boxDeliveryBase (базовый тариф за первый литр)
- стоимость_доп_литров = max(0, объём - 1) × boxDeliveryLiter
- коэф_логистики = dlv_prc из отчёта реализации (если фиксация активна)
                   ИЛИ коэффициент склада из таблицы тарифов (если фиксация истекла)
- КТР = ручной ввод (индивидуален для продавца, из ЛК)

Переплата = delivery_rub (факт WB) - Стоимость_логистики (расчёт)
```

#### Формула С 23.03.2026 (новая — ИЛ + ИРП)

```
Стоимость логистики = (стоимость_1л + стоимость_доп_литров) × коэф_логистики × ИЛ
                     + Цена_товара × ИРП

Где:
- стоимость_1л, стоимость_доп_литров, коэф_логистики — те же
- ИЛ = КТР из таблицы коэффициентов WB (зависит от % локализации артикула)
- ИРП = КРП% из той же таблицы (> 0 только если локализация < 60%)
- Цена_товара = розничная цена до скидки WB

Таблица коэффициентов (из services/wb_localization/irp_coefficients.py):
  95-100% локализации → КТР=0.50, КРП=0.00%
  90-95%  → КТР=0.60, КРП=0.00%
  75-80%  → КТР=0.90, КРП=0.00%  (целевой)
  60-65%  → КТР=1.00, КРП=0.00%  (граница ИЛ/ИРП)
  55-60%  → КТР=1.10, КРП=2.00%  (ИРП-зона — начинаются надбавки)
  25-30%  → КТР=1.70, КРП=2.20%
  0-5%    → КТР=2.20, КРП=2.50%  (максимальные надбавки)

Переплата = delivery_rub (факт WB) - Стоимость_логистики (расчёт)
```

#### Определение формулы для строки

```python
from datetime import date

FORMULA_CHANGE_DATE = date(2026, 3, 23)

def calculate_logistics_cost(
    row, volume, coef, base_1l, extra_l,
    ktr_manual,                    # Ручной КТР (для старой формулы)
    sku_localization_pct,          # % локализации SKU (для новой формулы)
    retail_price,                  # Розничная цена (для ИРП)
):
    """Расчёт стоимости логистики с учётом даты смены формулы."""
    # Базовая стоимость логистики
    if volume > 1:
        base_cost = (base_1l + (volume - 1) * extra_l) * coef
    else:
        base_cost = base_1l * coef

    if row.order_dt < FORMULA_CHANGE_DATE:
        # Старая формула: × КТР (ручной)
        return base_cost * ktr_manual
    else:
        # Новая формула: × ИЛ + Цена × ИРП
        from services.wb_localization.irp_coefficients import get_ktr_krp
        il, irp_pct = get_ktr_krp(sku_localization_pct)
        return base_cost * il + retail_price * (irp_pct / 100)
```

#### Источники данных для ИЛ/ИРП

| Данные | Источник | Как получить |
|---|---|---|
| % локализации по SKU | Заказы WB (supplier/orders) | Рассчитывается: локальные / всего заказов × 100 |
| КТР (= ИЛ) | `irp_coefficients.get_ktr_krp()` | По % локализации из таблицы |
| КРП (= ИРП) | `irp_coefficients.get_ktr_krp()` | По % локализации из таблицы |
| Розничная цена | Content API / prices API | Для расчёта ИРП-компоненты |
| Общий ИЛ кабинета | ЛК → Тарифы (0.98 у ООО) | Видно на скриншоте: "Индекс локализации 0,98" |
| Общий ИРП кабинета | ЛК → Тарифы (0.42% у ООО) | "Индекс распределения продаж 0,42%" |

**Существующие скрипты (можно переиспользовать):**
- `services/wb_localization/irp_coefficients.py` — таблица КТР/КРП, `get_ktr_krp()`, `calc_financial_impact()`
- `services/wb_localization/generate_localization_report_v3.py` — `calculate_sku_stats()` считает per-SKU локализацию

### 2.2 Определение коэффициента логистики

```python
def get_logistics_coef(row, tariff_table):
    """
    Из PDF: 'В отчёте всегда показывается зафиксированный коэффициент,
    даже если срок его действия уже истёк.'

    Если order_dt <= fix_tariff_date_to → используем dlv_prc
    Если order_dt > fix_tariff_date_to → используем коэф. склада из таблицы тарифов
    Если dlv_prc == 0 → не логистика, пропускаем
    """
    if row.dlv_prc == 0:
        return None  # не логистическая операция

    if row.order_dt <= row.fix_tariff_date_to:
        return row.dlv_prc  # фиксация ещё действует
    else:
        return tariff_table.get_warehouse_coef(row.office_name, row.order_dt)
```

### 2.3 Определение базового тарифа (КРИТИЧНО)

**Проблема:** API `/api/v1/tariffs/box` возвращает ТЕКУЩИЕ тарифы. Для ретро-расчёта нужны тарифы на дату фиксации.

**Обнаружено на реальных данных ООО Wookiee (март 2026):**

| Объём товара | Обратный расчёт базы | Текущий тариф API | Совпадение |
|---|---|---|---|
| > 1L (2.9L) | ~46₽/1л + ~14₽/доп.л | 46₽ + 14₽ | **99.7%** |
| ≤ 1L (0.9L) | ~31.7₽/1л | 46₽ | **НЕТ** (69%) |
| Гибрид: 33₽ для ≤1L | — | — | **99.3%** |

**Вывод:** Для точного расчёта необходим **обратный расчёт базового тарифа** по первым строкам отчёта:

```python
def calibrate_base_tariff(logistics_rows, cards_dims):
    """
    Обратный расчёт: base = delivery_rub / (dlv_prc * KTR)
    для строк с volume <= 1L (где формула = base * coef * KTR).

    Группируем по dlv_prc, берём медиану delivery_rub/dlv_prc.
    """
    bases = []
    for row in logistics_rows:
        vol = cards_dims.get(row.nm_id, {}).get('volume', 0)
        if 0 < vol <= 1 and row.dlv_prc > 0:
            bases.append(row.delivery_rub / row.dlv_prc)

    if bases:
        return statistics.median(bases)  # ≈ base * KTR
    return None
```

### 5.3 Конкретный пример расчёта строки

```
nm_id=257131227 (wendy/brown), Коледино
─────────────────────────────────────────
Из отчёта реализации:
  delivery_rub = 147.23₽ (WB списал)
  dlv_prc = 1.95
  fix_tariff: 2025-12-11 → 2026-02-09
  order_dt: 2026-02-28 (ПОСЛЕ окончания фиксации!)
  → Используем коэф. склада Коледино = 1.95 (совпадает)

Из карточки (Content API):
  dimensions: 33×22×4 см → объём = 2.904л

Из тарифов (tariffs/box):
  boxDeliveryBase = 46₽ (базовый, до коэф.)
  boxDeliveryLiter = 14₽

Расчёт:
  volume > 1 → (46 + (2.904 - 1) × 14) × 1.95
             = (46 + 26.656) × 1.95
             = 72.656 × 1.95
             = 141.68₽ (расчёт без КТР)

  delivery_rub / расчёт = 147.23 / 141.68 = 1.039
  → Подразумеваемый КТР ≈ 1.04

  При КТР = 1.04:
  Расчётная стоимость = 141.68 × 1.04 = 147.35₽
  Переплата = 147.23 - 147.35 = -0.12₽ (нет переплаты)
```

## 6. Алгоритм работы сервиса

### Шаг 1: Выгрузка данных (параллельно где возможно)

```
1. reportDetailByPeriod v5 (пагинация по rrd_id, до 100K строк/запрос)
2. tariffs/box на каждую уникальную дату из fix_tariff_date_from
3. Content API — все карточки с dimensions (пагинация по cursor)
4. warehouse_remains (async: create → poll → download)
5. measurement-penalties (если есть штрафы)
6. deductions, antifraud-details, goods-labeling (штрафы/удержания)
```

### Шаг 2: Калибровка базового тарифа

```python
# Обратный расчёт: для строк с volume <= 1 (формула = base * dlv_prc * KTR)
# delivery_rub / dlv_prc = base * KTR
# Медиана этих значений = base * KTR
median_base_ktr = median([row.delivery_rub / row.dlv_prc
                          for row in logistics if volume <= 1 and dlv_prc > 0])

# Если КТР известен:
estimated_base = median_base_ktr / KTR

# Если КТР неизвестен (= 1):
estimated_base = median_base_ktr  # нижняя оценка переплаты
```

### Шаг 3: Расчёт переплаты по каждой строке

```python
from datetime import date
from services.wb_localization.irp_coefficients import get_ktr_krp

FORMULA_CHANGE_DATE = date(2026, 3, 23)

for row in logistics_rows:
    # 1. Определить объём из карточки
    volume = cards[row.nm_id].volume_liters

    # 2. Определить коэффициент (фикс. или складской)
    if row.order_dt <= row.fix_tariff_date_to:
        coef = row.dlv_prc
    else:
        coef = tariff_table[row.office_name].coef

    # 3. Пропустить фикс. ставки (50₽)
    if row.bonus_type_name in ('От клиента при отмене', 'От клиента при возврате'):
        row.calculated_cost = row.delivery_rub  # не пересчитывается
        row.overpayment = 0
        continue

    # 4. Базовая стоимость логистики (без множителей)
    if volume > 1:
        base_cost = (base_1l + (volume - 1) * extra_l) * coef
    else:
        base_cost = base_1l * coef

    # 5. Применить формулу В ЗАВИСИМОСТИ ОТ ДАТЫ
    if row.order_dt < FORMULA_CHANGE_DATE:
        # Старая формула: × КТР (ручной ввод)
        cost = base_cost * ktr_manual
    else:
        # Новая формула (с 23.03.2026): × ИЛ + Цена × ИРП
        sku_loc_pct = sku_localization.get(row.nm_id, 75.0)  # % локализации SKU
        il, irp_pct = get_ktr_krp(sku_loc_pct)
        retail_price = prices.get(row.nm_id, 0)
        cost = base_cost * il + retail_price * (irp_pct / 100)

    row.calculated_cost = round(cost, 2)
    row.overpayment = round(row.delivery_rub - cost, 2)
```
