> Дата: 2026-04-02
> Статус: DRAFT
> Автор: Claude Code brainstorming session

## 1. Что строим

Claude Code skill `/monthly-plan`, который генерирует ежемесячный операционный бизнес-план бренда Wookiee на следующий месяц. Многоволновая агентская система с верификацией и обратной связью.

**Не является частью Oleg.** Отдельная автономная система, использующая `shared/data_layer` для доступа к данным.

### Входы
- Месяц планирования
- Стратегический контекст от пользователя (5 фиксированных вопросов + контекстные)
- Данные из 4 источников: PostgreSQL WB, PostgreSQL OZON, Supabase, Google Sheets

### Выход
- 12-раздельный документ (A-L) — MD файл
- Публикация в Notion (database `30158a2b-d587-8091-bfc3-000b83c6b747`)
- Публикация в Google Sheets (KPI_PlanFact)

### 2.2 Принцип разделения

- **Python (collector)** — детерминистический сбор данных. Вызывает `shared/data_layer`, пакует в JSON. Быстро (10-15 сек), воспроизводимо, без LLM.
- **Claude Code (skill + субагенты)** — анализ, рекомендации, кросс-верификация, синтез. То, где нужен reasoning.
- **Субагенты имеют доступ к Bash** — могут дозапрашивать данные через `python -c "from shared.data_layer..."` если базовых данных недостаточно для глубокого анализа.

## 3. Flow — стадии выполнения

### Stage 0: Context Collection (интерактивный)

Skill задаёт 5 фиксированных вопросов пользователю:

1. **Месяц планирования** — "На какой месяц строим план?" (default: следующий от текущей даты)
2. **Модельный ряд** — "Какие модели активны? Есть изменения статусов?" (показывает текущие статусы из Supabase как подсказку)
3. **Новые запуски** — "Планируются новинки? На каком этапе производства?"
4. **Бюджетные ограничения** — "Есть лимиты по рекламе, производству, закупкам?"
5. **Стратегический контекст** — "Что ещё учесть? (сезонность, акции WB/OZON, планы команды)"

Ответы сохраняются как `user_context` и передаются каждому субагенту во всех волнах.

### Stage 1: Data Collection (Python)

```bash
python scripts/monthly_plan/collect_all.py --month 2026-05 --output /tmp/monthly-plan-data.json
```

Собирает 7 блоков параллельно:

| # | Блок | Функции shared/data_layer | Что в JSON |
|---|------|--------------------------|-----------|
| 1 | P&L Total | `finance.get_wb_finance()`, `finance.get_ozon_finance()` | Полная воронка WB/OZON/ИТОГО, текущий + предыдущий месяц |
| 2 | P&L Models | `finance.get_wb_by_model()`, `finance.get_ozon_by_model()`, `sku_mapping.get_model_statuses_mapped()` | Разбивка по моделям с статусами (активные/выводимые) |
| 3 | Pricing | `pricing_article.get_wb_price_margin_daily_by_article()`, `pricing_article.get_ozon_price_margin_daily_by_article()` | Дневные цены/объёмы за 3 месяца по артикулам |
| 4 | Advertising | `finance.get_wb_by_model()` (adv fields) + Google Sheets внешняя реклама | Реклама внутренняя + внешняя по моделям и каналам |
| 5 | Inventory | `inventory.get_wb_avg_stock()`, `inventory.get_ozon_avg_stock()`, `inventory.get_moysklad_stock_by_model()`, `inventory.get_wb_turnover_by_model()` | Остатки FBO + МС, оборачиваемость, дней стока |
| 6 | ABC | `planning.get_active_models_with_abc()` + Google Sheets план финансиста | ABC-классы по артикулам, план vs факт |
| 7 | Traffic | `traffic.get_wb_traffic_by_model()`, `funnel_seo.get_wb_article_funnel()` | Рекламный трафик, органическая воронка, SEO позиции |

**Параметры дат** (вычисляются автоматически из `--month`):

```python
CURRENT_MONTH_START = "2026-04-01"      # база для плана
CURRENT_MONTH_END   = "2026-05-01"      # exclusive
PREV_MONTH_START    = "2026-03-01"      # для м/м сравнения
ELASTICITY_START    = "2026-01-01"      # 3 месяца для E
STOCK_WINDOW_START  = "2026-04-25"      # последняя неделя для остатков
```

**Quality flags** — collector добавляет в JSON известные проблемы:
- `fan_out_bug`: true (get_wb_model_ad_roi завышает ad_spend, используем workaround)
- `db_vs_sheets_gap`: разница DB/Sheets по внешней рекламе
- `ozon_no_external_ads`: OZON внешняя реклама = 0 в БД
- `traffic_powerbi_gap`: content_analysis ≠ PowerBI (~20%)
- Для каждой модели: `data_months` (сколько месяцев данных есть для эластичности)

### Stage 1.5: Triage (Claude Code)

Claude Code читает JSON и автоматически выявляет аномалии:

**Триггеры контекстных вопросов:**
- Маржа модели Δ >5пп м/м → "Модель X: М-2 снизилась с Y% до Z%. Знаешь причину?"
- DRR модели > break-even DRR → "Модель X: DRR Y% при break-even Z%. Продолжаем рекламу?"
- Остаток <14 дней → "Модель X: N дней стока на площадке Y. Есть поставка в пути?"
- Остаток >90 дней → "Модель X: overstock N дней. Планируем скидки?"
- План финансиста vs факт >20% → "Модель X: отклонение N% от плана. Корректируем?"
- Новая модель (<3 мес данных) → "Модель X: мало данных для эластичности. Какие ценовые планы?"

Задаёт **только релевантные** вопросы (0-5 штук). Если аномалий нет — пропускает стадию.

Ответы добавляются в `user_context`.

### Stage 2: Multi-Wave Analysis

#### Волна 1 — Аналитики (5 параллельных субагентов)

Каждый субагент получает:
- Свой основной slice данных из JSON (детальные данные)
- Сводку по остальным блокам (summary, не raw данные — чтобы не раздувать контекст, особенно pricing daily_data)
- `user_context` (все ответы пользователя)
- `quality_flags` (ограничения данных)
- **Доступ к Bash** — может дозапрашивать данные через data_layer

**Субагент 1: P&L Analyst**
- Основные данные: `pnl_total`, `pnl_models`
- Задача: полная финансовая воронка (разделы A, B, C итогового документа)
- Глубина анализа:
  - Если маржа модели аномальна → дозапрашивает дневную разбивку для локализации даты
  - Если расхождение WB/OZON по модели → проверяет по артикулам
  - Строит два сценария на следующий месяц (оптимальный / агрессивный)
  - Отдельно считает активные и выводимые модели
- Выход: структурированные данные для разделов A, B, C + проекции

**Субагент 2: Pricing Analyst**
- Основные данные: `pricing` (эластичность по артикулам), `pnl_models` (маржа)
- Задача: ценовые гипотезы с confidence (раздел I + часть D)
- Глубина анализа:
  - Считает E и r для каждого артикула внутри модели
  - Volume-weighted агрегация на уровень модели
  - Классификация: неэластичная (|E|<1), эластичная (|E|>2), средняя
  - Для моделей с LOW confidence (|r|<0.3 или <30 дней) → рекомендация "держать"
  - Может запросить конкретные артикулы для углублённого анализа
  - Расчёт эффекта в рублях для каждой ценовой гипотезы
- Выход: таблица рекомендаций по моделям с E, r, confidence, ожидаемый эффект

**Субагент 3: Ad Budget Analyst**
- Основные данные: `advertising`, `pnl_models` (для break-even)
- Задача: рекламная эффективность + бюджет (разделы H, E + часть D)
- Глубина анализа:
  - ROAS и DRR с разбивкой внутренняя/внешняя по каждой модели
  - Break-even DRR = Маржа-2% (если DRR > BE → модель убыточна по рекламе)
  - Дозапрашивает Google Sheets для детализации по каналам (блогеры, WOW, ЯПС, посевы, АДС)
  - Два сценария бюджета: оптимальный (сохранение ROAS) и агрессивный (масштабирование)
  - Анализ связки реклама→заказы (эффективность)
- Выход: ROAS/DRR таблица + два сценария бюджета + рекомендации по перераспределению

**Субагент 4: Inventory & ABC Analyst**
- Основные данные: `inventory`, `abc`, `pnl_models` (для sales velocity)
- Задача: остатки, оборачиваемость, ABC, сверка с финансистом (разделы F, G)
- Глубина анализа:
  - Оценка риска по каждой модели: ДЕФИЦИТ (<14д) / OK / OVERSTOCK (>90д) / МЁРТВЫЙ СТОК (>250д)
  - Если дефицит → проверяет МойСклад (есть ли резерв)
  - Если overstock → считает сколько дней до нормализации при скидке -20%
  - ABC-классификация: A-артикул в статусе "Выводим" → флаг
  - Сверка план финансиста vs факт + тренд → достижимость плана
- Выход: таблица рисков + ABC + план vs факт + рекомендации по поставкам/ликвидации

**Субагент 5: Traffic & Funnel Analyst**
- Основные данные: `traffic`, `pnl_models` (для контекста)
- Задача: трафик, конверсия, SEO (часть раздела D + данные для других аналитиков)
- Глубина анализа:
  - Воронка по моделям: показы → клики → корзина → заказы (organic)
  - CR по этапам воронки для каждой модели
  - Рекламный vs органический трафик — доля organic
  - SEO позиции — тренд, критические потери
  - Эффект склеек WB (Wendy → Audrey/Lana attribution)
  - Ограничение: только WB, данные ≠ PowerBI (~20%), использовать как тренд
- Выход: воронка по моделям + CR анализ + SEO highlights

#### Волна 2 — Критики (2 параллельных субагента)

Получают ВСЕ выходы аналитиков (Волна 1) + `user_context` + `quality_flags`.

**Critic 1: Data Quality Auditor**
- Проверяет:
  - Арифметика: сумма моделей = Total (расхождение >1% = ошибка)
  - DRR посчитан корректно (с разбивкой внутренняя/внешняя)
  - СПП при объединении каналов — средневзвешенный (не среднее арифметическое)
  - GROUP BY с LOWER() соблюдён (нет дублей моделей с разным регистром)
  - Эластичность не завышена (модельный уровень vs артикульный)
  - Даты: exclusive end (`date < first_day_next_month`)
  - Маржа: оба показателя (М-1 и М-2) присутствуют
  - Выкуп % не используется как причина дневных изменений
- Выход: список ошибок с severity (critical/warning) + что исправить

**Critic 2: Strategy Auditor**
- Проверяет:
  - Противоречия между рекомендациями аналитиков:
    - Pricing рекомендует снизить цену + Inventory показывает дефицит → конфликт
    - Ad Analyst увеличивает бюджет модели с DRR > break-even → ошибка
    - P&L показывает рост + Inventory показывает overstock → несогласованность
  - Полнота: все активные модели покрыты рекомендациями
  - Реалистичность: сценарии бюджета vs текущий тренд
  - Confidence: LOW confidence рекомендации не должны быть агрессивными
- Выход: список противоречий + предложения по разрешению

#### Волна 3 — Корректор (1 субагент)

Получает: все выходы аналитиков + все замечания критиков + доступ к Bash.

Задача — **исправить, не решать**:
1. Арифметические ошибки → пересчитать через data_layer
2. Фактические противоречия → дозапросить данные, разрешить
3. Стратегические противоречия → пометить как "REQUIRES_CFO_DECISION" (не его компетенция)

Выход: исправленные findings аналитиков + список вопросов для Финдиректора.

#### Волна 4 — Финдиректор (1 субагент)

Получает: исправленные findings (после Корректора) + замечания критиков + user_context.

**Три вердикта (Проход 1):**

- **APPROVE** — данные корректны, рекомендации обоснованы → переход к синтезу
- **CORRECT** — мелкие правки, Финдиректор вносит inline → переход к синтезу
- **REJECT** — критические проблемы → указывает какие аналитики ошиблись и что исправить → Проход 2

**Задачи Финдиректора:**
1. Валидация каждой рекомендации аналитиков
2. Арбитраж стратегических противоречий (помеченных как REQUIRES_CFO_DECISION)
3. Расстановка приоритетов: что делать в первую/вторую/третью/четвёртую неделю
4. Финальный вердикт по каждой модели: план действий с обоснованием

**При REJECT — Проход 2 (максимум один):**

```
Проход 2: только проблемные аналитики (по указанию CFO)
  → Data Quality Critic (проверка исправленных цифр)
  → Corrector (если Critic нашёл остаточные ошибки)
  → CFO (обязан принять, может добавить inline-правки)
```

Strategy Critic не перезапускается в Проходе 2 — стратегические противоречия уже разрешены CFO в Проходе 1.

**Жёсткое ограничение: максимум 2 прохода.** На втором проходе CFO обязан принять результат. Может добавить inline-коррекции, но не может отклонить снова. Это предотвращает бесконечные циклы.

### Stage 3: Synthesis (отдельный субагент)

Получает:
- Финальный выход Финдиректора (решения, приоритеты, вердикты)
- Исправленные findings аналитиков (после Корректора) — для таблиц и данных
- user_context + quality_flags — для раздела L

Synthesizer — это отдельный субагент (не основной контекст skill'а), чтобы работать с чистым контекстом и не терять качество на длинном документе.

Генерирует 12-раздельный документ:

| Раздел | Содержание | Источник данных |
|--------|-----------|----------------|
| A | P&L Brand Total (WB/OZON/ИТОГО, факт + план) | P&L Analyst |
| B | P&L по моделям — Активные | P&L Analyst |
| C | P&L по моделям — Выводимые | P&L Analyst + Inventory Analyst |
| D | Гипотезы (ценовые + рекламные + остатки, с confidence) | Pricing + Ad + Inventory Analysts |
| E | Бюджет рекламы (два сценария) | Ad Analyst |
| F | Остатки и оборачиваемость | Inventory Analyst |
| G | ABC-анализ + сверка с финансистом | Inventory Analyst |
| H | Рекламная эффективность (ROAS, DRR, break-even) | Ad Analyst |
| I | Ценовая эластичность (volume-weighted) | Pricing Analyst |
| J | Понедельный план (4 недели с KPI и alarm-триггерами) | CFO (приоритеты) + все аналитики |
| K | Результаты верификации (кросс-проверки + критика) | Critics + Corrector + CFO вердикт |
| L | Контекст и ограничения | user_context + quality_flags |

### Stage 4: Publication

1. **MD файл** → `docs/plans/YYYY-MM-business-plan-final.md`
2. **Notion** → новая страница в database `30158a2b-d587-8091-bfc3-000b83c6b747`
3. **Google Sheets** → лист "📋 План {Месяц} {Год}" в KPI_PlanFact

| Google Sheets (план) | gws CLI / gspread | `1Dsz7s_mZ0wUhviGFho89lyhtjce1V0Cmv_RPL1aLxnk` | План финансиста: WB!A1:Z50, OZON!A1:Z30 |
| Google Sheets (KPI) | gws CLI / gspread | `1GRCGSAJESSDvAhoVMmXljXy-qErMKt-n45PV96YBiVY` | KPI цели: Sheet1!A1:Z20 |
| Google Sheets (внешн.рекл.) | gws CLI / gspread | `1PvsgAkb2K84ss4iTD25yoD0pxSZYiVgcqetVUtCBvGg` | "Итог {Месяц}!A1:Q40" |

### 4.3 Правила качества данных (зашиты в collector)

1. GROUP BY модели: `LOWER(SPLIT_PART(article, '/', 1))` — всегда
2. Даты: `date < первый_день_следующего_месяца` (exclusive end)
3. СПП при объединении: средневзвешенный `sum(spp)/sum(rev)*100`
4. Маппинг моделей: через `shared/model_mapping.py` → `map_to_osnova()`
5. ROAS: через `get_wb_by_model()` (workaround fan-out бага), НЕ через `get_wb_model_ad_roi()`
6. Маржа: всегда оба показателя (М-1 и М-2)
7. OZON Маржа = М-1 только (нет данных для М-2)
8. Все запросы через `shared/data_layer/` — никакого raw SQL

### 4.4 Известные проблемы данных

| # | Проблема | Severity | Workaround в collector |
|---|---------|----------|----------------------|
| 1 | Fan-out `get_wb_model_ad_roi()` (+40%) | Critical | ROAS через `get_wb_by_model()` |
| 2 | DB vs Sheets внешняя реклама (73%) | Critical | Собираем оба источника, помечаем расхождение |
| 3 | PowerBI маржа расхождение (+812К) | Warning | Стандартизация дат, пометка в quality_flags |
| 4 | OZON нет внешней рекламы | Warning | OZON М-2 = М-1, пометка |
| 5 | Traffic ≠ PowerBI (~20%) | Warning | Только как тренд, не абсолют |
| 6 | E ненадёжен для моделей <3 мес | Warning | confidence=LOW, рекомендация "держать" |
| 7 | Склейки WB (attribution) | Info | Пометка для аналитиков |
| 8 | Себестоимость Eva подозрительна | Warning | Пометка, все метрики Eva ненадёжны |

## 5. Субагенты — детали

### 5.1 Общий контракт

Каждый субагент:
- Получает данные в промпте (JSON slice + полный JSON для контекста)
- Имеет доступ к Bash для дозапросов (`python -c "from shared.data_layer..."`)
- Знает quality_flags и учитывает их в анализе
- Возвращает структурированный markdown с findings

### 5.2 Промпты — что содержат

Каждый промпт в `prompts/analysts/` включает:
1. **Роль** — кто этот аналитик
2. **Контекст бренда** — базовые бизнес-правила Wookiee
3. **Данные** — JSON slice (инжектится при запуске)
4. **User context** — ответы пользователя (инжектится при запуске)
5. **Quality flags** — известные ограничения (инжектится при запуске)
6. **Задание** — что именно проанализировать, какие разделы подготовить
7. **Формат выхода** — структура markdown с обязательными секциями
8. **Правила анализа** — domain-specific правила (из TECHNICAL-SPEC и AGENTS.md)
9. **Инструкция по дозапросам** — когда и как дозапрашивать данные через Bash

### 5.3 CFO — особый промпт

CFO получает дополнительно:
- Все замечания критиков (с severity)
- Все REQUIRES_CFO_DECISION вопросы от Корректора
- Явную инструкцию по трём вердиктам (APPROVE / CORRECT / REJECT)
- Формат для REJECT: `{ "verdict": "REJECT", "rerun_analysts": ["pricing", "inventory"], "feedback": "..." }`

## 6. Ограничения и safety

### 6.1 Максимум проходов

**Жёсткое ограничение: 2 прохода.** Зашито в SKILL.md. Если CFO отклонил на первом проходе — запускается Проход 2. На втором проходе CFO обязан принять (APPROVE или CORRECT, без REJECT).

### 6.2 Таймауты субагентов

- Каждый аналитик: без жёсткого таймаута (пусть работает глубоко)
- Критики: без таймаута
- Общий таймаут на весь pipeline: нет (подписка Max, оптимизируем качество)

### 6.3 Воспроизводимость

- Collector output (JSON) сохраняется в `/tmp/monthly-plan-data-YYYY-MM.json`
- При повторном запуске с `--cached` можно использовать сохранённый JSON
- Промпты субагентов детерминистичны при одних и тех же данных

## 7. Расширяемость

### Новый источник данных
1. Добавить функцию в `scripts/monthly_plan/collectors/`
2. Добавить блок в JSON-схему
3. Обновить промпт релевантного аналитика
4. Готово

### Новый аналитик
1. Добавить промпт в `prompts/analysts/`
2. Добавить запуск в SKILL.md (Волна 1)
3. Обновить промпты критиков (чтобы проверяли нового)
4. Обновить промпт синтезатора (какие разделы из нового)

### Примеры расширений (v2+)
- Данные по производству → Inventory Analyst или новый Production Analyst
- Данные по блогерам (детальные) → Ad Budget Analyst
- SMM метрики → Traffic Analyst
- Данные конкурентов → новый Market Analyst
- Прогнозная модель с сезонностью → новый Forecasting Analyst
- Ретроспектива предыдущего плана (что сбылось) → Stage 0 расширение
