## Задача

Построить полный операционный бизнес-план бренда на следующий месяц. На входе — текущий месяц как база. На выходе — P&L с разбивкой по моделям, ценовые и рекламные гипотезы, план остатков, понедельный план действий.

| `orders` | Заказы: дата, артикул, количество, цена до/после СПП, баркод | Объёмы продаж, средние цены, ценовая эластичность |

| `nomenclature` | Справочник номенклатуры: nmid → vendorcode (артикул) | Маппинг рекламных данных на артикулы |
| `stocks` | Ежедневные остатки FBO по складам | Текущие остатки, оборачиваемость |
| `ms_stocks` | Остатки МойСклад (центральный склад) | Резервы для пополнения, транзит |
| `kz_off` | SEO позиции по ключевым запросам | Позиции в поиске (информационно) |
| `plan_article` | План продаж по артикулам (от финансиста) | Сверка план vs факт |
| `adv_budget` | Бюджеты рекламных кампаний | Утилизация бюджета |

**Ключевое поле**: артикул (`vendorcode`). Формат: `model/color` (напр. `wendy/black`).

**GROUP BY модели**: `LOWER(SPLIT_PART(article, '/', 1))` — ВСЕГДА с LOWER(), иначе "Wendy" и "wendy" разъедутся.

**Формула маржи WB** (из `_sql_fragments.py`):
```sql
-- Маржа-1 (операционная, совпадает с PowerBI):
SUM(marga) - SUM(nds)

-- Маржа-2 (чистая, с вычетом внешней рекламы):
SUM(marga) - SUM(nds) - SUM(reclama_vn) - COALESCE(SUM(reclama_vn_vk), 0) - COALESCE(SUM(reclama_vn_creators), 0)
```

### 2.2 Ценовая эластичность

```python
from shared.data_layer.pricing_article import (
    get_wb_price_margin_daily_by_article,   # → дневная цена+маржа по артикулу
    get_ozon_price_margin_daily_by_article,  # → OZON аналог
)
```

**Параметры**: `(start_date, end_date, article=None, model=None)` — для эластичности нужен окно 3 месяца (напр. `'2026-01-01'`, `'2026-04-30'`)

**Что считаем** (поверх сырых данных):
1. Для каждого артикула внутри модели: средневзвешенная цена по дням с продажами
2. **E** (эластичность) = %Δ объём / %Δ цена — корреляция между ценой и количеством
3. **r** (Пирсон) — сила корреляции (чем ближе к ±1, тем надёжнее E)
4. **Volume-weighted E** по модели = средневзвешенная E артикулов, вес = доля в продажах модели

**Критично**: модельная E (без разбивки по артикулам) завышает эластичность в 2-13 раз. Всегда считать на артикульном уровне, потом агрегировать с весами.

**Паттерн "двух кластеров"**: SKU на полной цене (2,700-3,100₽) — неэластичны, SKU на скидке (800-1,800₽) — эластичны. Агрегирование маскирует этот эффект.

### 2.3 Рекламная эффективность

```python
from shared.data_layer.advertising import (
    get_wb_model_ad_roi,           # → ROAS, ROMI по моделям WB ⚠️ FAN-OUT БАГ
    get_ozon_model_ad_roi,         # → ROAS OZON
    get_wb_external_ad_breakdown,  # → внутренняя vs внешняя (блогеры/ВК/creators)
)
```

**Что считаем**:
- **ROAS** = revenue / ad_spend
- **DRR внутренний** = ad_internal / revenue_before_spp × 100
- **DRR внешний** = ad_external / revenue_before_spp × 100
- **DRR общий** = (ad_internal + ad_external) / revenue_before_spp × 100
- **Break-even DRR** = Маржа-2% модели (если DRR > break-even → убыток)
- **ROMI** = (маржа-2 - ad_total) / ad_total × 100

**⚠️ БАГ**: `get_wb_model_ad_roi()` завышает ad_spend на ~40% из-за fan-out при JOIN `nomenclature` → `wb_adv` (дубли nmid). **Workaround**: ROAS считать через `get_wb_by_model()` (поля adv_internal + adv_external).

**Google Sheets для внешней рекламы**: детальная разбивка расходов по каналам (блогеры, WOW, ЯПС, посевы, АДС) — берётся из таблицы внешней рекламы (`1PvsgAkb2K84ss4iTD25yoD0pxSZYiVgcqetVUtCBvGg`).

**⚠️ Расхождение**: DB показывает 919К внешней рекламы, Sheets показывает 530К (73%). Причина не выяснена.

## 3. Анализы (что делаем с данными)

### 3.1 P&L Total — полная воронка бренда

**Вход**: finance.py (WB + OZON)
**Выход**: таблица WB | OZON | ИТОГО по 20+ показателям

Строим полную цепочку:
```
Заказы (шт, ₽) → Продажи → Revenue до СПП → Revenue после СПП
→ минус: Себестоимость, Логистика, Хранение, Комиссия, НДС
→ Маржа-1 (до внешней рекламы)
→ минус: Реклама внутренняя, Реклама внешняя
→ Маржа-2 (после всей рекламы)
→ минус: Возвраты, Штрафы, Удержания
```

Плюс: сравнение с предыдущим месяцем (Δ% по каждому показателю).

**СПП при объединении каналов**: средневзвешенный `sum(spp_amount) / sum(revenue_before_spp) * 100`.

---

### 3.2 P&L по моделям — активные vs выводимые

**Вход**: finance.py + sku_mapping.py
**Выход**: две таблицы — активные модели (12-15 строк) и выводимые (2-5 строк)

Для каждой активной модели:
- Revenue WB + OZON отдельно и суммарно
- Реклама: внутренняя + внешняя
- Маржа-1, Маржа-1%, Маржа-2, Маржа-2%
- DRR общий
- ROAS

Для выводимых: Revenue, Маржа, Остатки WB + МС, рекомендация по ликвидации.

---

### 3.3 Ценовые гипотезы (артикульный уровень)

**Вход**: pricing_article.py (3 месяца данных)
**Выход**: для каждой модели — рекомендация по цене с confidence

Алгоритм:
1. Загружаем дневные данные цена/количество по каждому артикулу
2. Считаем E и r для каждого артикула
3. Volume-weighted агрегация на уровень модели
4. Классификация: неэластичная (|E|<1) → можно повышать; эластичная (|E|>2) → не трогать; средняя → тестировать
5. Проверка r: при |r|<0.3 — данные ненадёжны, рекомендация LOW confidence

**Confidence**:
- HIGH: |r| > 0.5, достаточно данных (>30 дней с вариацией цены)
- MED: 0.3 < |r| < 0.5 или 15-30 дней данных
- LOW: |r| < 0.3 или <15 дней или новая модель (<3 мес)

---

### 3.4 Рекламный бюджет (два сценария)

**Вход**: advertising.py + finance.py + Google Sheets (внешняя реклама)
**Выход**: таблица бюджета по моделям × каналам, два сценария

Каналы рекламы:
- **Внутренняя** (МП WB/OZON) — автоматическое продвижение
- **Блогеры** (собственные) — reclama_vn в БД
- **WOW** — рекламная интеграция WB
- **ЯПС** — Яндекс.Поисковые Сети
- **Посевы** — размещения в тематических каналах
- **АДС** — ВК реклама

Для каждой модели считаем break-even DRR = Маржа-2%. Если текущий DRR > break-even → модель убыточна по рекламе.

**Сценарий A** (оптимальный): минимальный бюджет с сохранением ROAS.
**Сценарий B** (агрессивный): план финансиста, масштабирование на основе результатов первых 2 недель.

---

### 3.5 Остатки — оценка рисков

**Вход**: inventory.py
**Выход**: таблица модель × площадка → дней стока → риск

Для каждой модели:
1. Текущий остаток FBO (WB, OZON)
2. Остаток МойСклад (резерв)
3. Среднедневные продажи текущего месяца
4. Дней хватит = остаток / ср.дневные продажи
5. Оборачиваемость FBO
6. Риск: ДЕФИЦИТ / OK / OVERSTOCK / МЁРТВЫЙ СТОК

**Действия**:
- ДЕФИЦИТ (<14д) → СРОЧНО пополнить (проверить МС)
- OVERSTOCK (>90д) → НЕ пополнять, рассмотреть скидки
- МЁРТВЫЙ СТОК (>250д) → агрессивные скидки -30-50%

---

### 3.6 ABC + сверка с планом финансиста

**Вход**: planning.py + Google Sheets (план финансиста)
**Выход**: ABC-классификация + таблица план vs факт

1. ABC-классификация артикулов внутри каждой модели
2. Загрузка плана финансиста из Sheets
3. Сверка: план финансиста (руб) vs факт текущего месяца (руб) → расхождение
4. Выявление структурных рисков: модели с отклонением >20%

## 4. Кросс-верификация (7 проверок)

После сбора всех 6 блоков данных — перекрёстные проверки:

| # | Проверка | Что сверяем | Что ищем |
|---|---------|------------|---------|
| 1 | P&L Consistency | Сумма моделей vs Total | Расхождение >1% = ошибка данных |
| 2 | Цена × Остатки | Если снижаем цену → хватит ли стока при росте объёма? | Дефицит при ценовой акции |
| 3 | Реклама × P&L | DRR vs break-even DRR | Кто рекламируется в убыток |
| 4 | Рост-модели | ROAS + остатки + ABC подтверждают рост? | Ложные сигналы роста |
| 5 | Убыточные модели | DRR × маржинальность = ROMI | Модели с ROMI < 0 |
| 6 | Замедление | Снижение продаж + большие остатки | Overstock при падении |
| 7 | Финансист vs Факт | План финансиста vs факт + тренд | Достижимость плана |

## 8. Правила качества данных

1. **GROUP BY модели**: `LOWER(SPLIT_PART(article, '/', 1))` — ВСЕГДА
2. **Даты**: `date < первый_день_следующего_месяца` (НЕ `<=`)
3. **СПП при объединении WB+OZON**: средневзвешенный `sum(spp)/sum(rev)*100`
4. **ДРР**: ВСЕГДА с разбивкой внутренняя/внешняя
5. **Выкуп %**: лаговый (3-21 день), только информационный
6. **Маппинг моделей**: `shared/model_mapping.py` → `map_to_osnova()`
7. **DB-запросы**: ТОЛЬКО через `shared/data_layer/`. Никакого raw SQL
8. **Маржа**: ВСЕГДА показывать оба показателя (М-1 и М-2)
