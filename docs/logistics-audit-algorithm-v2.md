# Алгоритм аудита логистики WB v2

> Спецификация расчётного алгоритма сервиса `services/logistics_audit/`.
> Версия: v2 (актуальна на апрель 2026).

---

## 1. Формула расчёта

### До 23.03.2026 (старая формула, КТР)

```
calculated_cost = (base_1l + max(0, volume - 1) * extra_l) * coef * IL
overpayment = delivery_rub - calculated_cost
```

| Переменная     | Описание                                                |
|----------------|---------------------------------------------------------|
| `base_1l`      | Базовая ставка за первый литр (из тарифного периода)    |
| `extra_l`      | Ставка за каждый дополнительный литр                    |
| `volume`       | Объём товара в литрах (из карточки WB)                  |
| `coef`         | Коэффициент склада (3-tier resolution)                  |
| `IL`           | Индекс Локализации (КТР) — недельный коэффициент       |
| `delivery_rub` | Фактическая сумма, списанная WB                        |

При `volume <= 1`: `calculated_cost = base_1l * coef * IL`.

### С 23.03.2026 (новая формула, ИЛ + ИРП)

```
calculated_cost = (base_1l + max(0, volume - 1) * extra_l) * coef * КТР + retail_price * (КРП / 100)
overpayment = delivery_rub - calculated_cost
```

| Переменная          | Описание                                                      |
|---------------------|---------------------------------------------------------------|
| `КТР`               | Коэффициент транспортного расстояния (по % локализации SKU)  |
| `КРП`               | Коэффициент распределения продаж, % от цены товара           |
| `retail_price`       | Розничная цена товара (после скидки продавца)                |
| `sku_localization_%` | Доля локализованных заказов конкретного артикула              |

КТР и КРП определяются по таблице `irp_coefficients.py` на основе % локализации SKU.

**Исходный код:** `services/logistics_audit/calculators/logistics_overpayment.py`

## 3. Тарифные периоды

Базовые ставки зависят от даты заказа. При активной фиксации (`fix_tariff_date_to > order_dt`) используется `fix_tariff_date_from` как reference date.

**Исходный код:** `services/logistics_audit/calculators/tariff_periods.py` — `TARIFF_PERIODS`

| Дата начала  | base_1l (₽) | extra_l (₽) |
|-------------|-------------|-------------|
| 22.09.2025  | 46.00       | 14.00       |
| 28.02.2025  | 38.00       | 9.50        |
| 11.12.2024  | 35.00       | 8.50        |
| 14.08.2024  | 33.00       | 8.00        |

Поиск: первый период, где `ref_date >= start_date` (отсортировано от новых к старым).

Если дата раньше всех периодов — используются ставки самого старого периода (33.00 / 8.00).

## 4. Sub-liter тиры

Применяются когда **одновременно**:
- `order_date >= 22.09.2025`
- `0 < volume < 1.0` литра

**Исходный код:** `services/logistics_audit/calculators/tariff_periods.py` — `SUB_LITER_TIERS`

| Макс. объём (л) | base_1l (₽) | extra_l (₽) |
|-----------------|-------------|-------------|
| 0.200           | 23.00       | 0.00        |
| 0.400           | 26.00       | 0.00        |
| 0.600           | 29.00       | 0.00        |
| 0.800           | 30.00       | 0.00        |
| 1.000           | 32.00       | 0.00        |

Поиск: первый тир, где `volume <= max_vol`. Extra_l всегда 0 (объём < 1 л).

Проверка sub-liter использует `order_date`, а не `ref_date` (фиксация не влияет).

## 5. 3-tier коэффициент склада

Коэффициент склада определяется с приоритетом из трёх источников.

**Исходный код:** `services/logistics_audit/calculators/warehouse_coef_resolver.py`

### Tier 1: Фиксация тарифа (fixation)

**Условие:** `fixed_coef > 0` AND `fixation_end > order_date`

Если у продавца активна фиксация тарифа на дату заказа, используется зафиксированный коэффициент из `dlv_prc` отчёта.

- `source = "fixation"`, `verified = True`

### Tier 2: Supabase wb_tariffs (исторические данные)

**Условие:** Tier 1 не сработал, в таблице `wb_tariffs` есть данные по складу.

Берётся ближайшая дата `<= order_date` из таблицы `wb_tariffs` для данного склада.

- Коэффициент хранится в процентах → делится на 100
- В ETL также сохраняется `storage_coef`, но в текущей формуле аудита он не используется
- `source = "supabase"`, `verified = True`

### Tier 3: dlv_prc fallback

**Условие:** Tier 1 и Tier 2 не сработали.

Используется `dlv_prc` из отчёта WB как есть.

- `source = "dlv_prc"`, `verified = False` (помечается как непроверенный)

Если `dlv_prc = 0` — коэффициент 0, строка не рассчитывается (overpayment = None).

## 6. ИЛ (Индекс Локализации)

### Что это

ИЛ (он же КТР до 23.03.2026) — множитель к стоимости логистики, зависящий от доли локализованных заказов продавца (отношение: заказы с ФО склада = ФО доставки / все заказы).

### Расчёт

1. Получаем заказы через WB Supplier Orders API за период аудита (с буфером -7 дней)
2. Для каждого заказа определяем ФО склада и ФО доставки (через `wb_localization_mappings`)
3. Если ФО совпадает — заказ "локализованный"
4. Группируем по неделям (понедельник — ключ)
5. `localization_% = local_orders / total_orders * 100`
6. По таблице `COEFF_TABLE` определяем КТР

### Применение

ИЛ применяется по `order_dt`: определяем понедельник недели заказа → берём ИЛ этой недели.

### Приоритет источников

| Приоритет | Источник                     | Описание                                              |
|-----------|------------------------------|-------------------------------------------------------|
| 1 (выше) | `il_overrides.json`          | Ручные значения из WB Partners dashboard              |
| 2         | `calculate_weekly_il()`      | Расчёт по заказам через WB Orders API                 |
| fallback  | `config.ktr` или `1.0`      | Если нет данных по неделе                             |

## 7. Фильтрация строк

### Шаг 1: Отбор логистических строк

```
supplier_oper_name == "Логистика"
```

Из всех строк `reportDetailByPeriod v5` отбираются только строки с типом операции "Логистика".

### Шаг 2: Классификация по типу доставки

**Прямые доставки (аудитируемые):**
```python
FORWARD_DELIVERY_TYPES = {"К клиенту при продаже", "К клиенту при отмене"}
```

Только строки с `bonus_type_name` из этого набора проходят расчёт переплаты.

**Фиксированные ставки (не аудитируемые):**
```python
FIXED_RATE_TYPES = {"От клиента при отмене", "От клиента при возврате"}
```

Для этих строк `overpayment = 0` (стоимость фиксирована WB, пересчёт невозможен).

### Шаг 3: Фильтрация в итогах

В сводных таблицах учитываются только строки с `overpayment > 0` (WB списал больше расчётного).

Строки с `overpayment <= 0` (WB списал корректно или меньше) не включаются в сумму переплаты.

## 8. Блок-схема обработки строки

```
reportDetailByPeriod v5 row
        │
        ▼
┌─────────────────────────────┐
│ supplier_oper_name == "Логистика"? │
└──────────────┬──────────────┘
               │ Нет → ПРОПУСК
               ▼ Да
┌─────────────────────────────┐
│ bonus_type_name ∈ FIXED_RATE_TYPES? │
└──────────────┬──────────────┘
               │ Да → overpayment = 0, ГОТОВО
               ▼ Нет
┌─────────────────────────────┐
│ bonus_type_name ∈ FORWARD_DELIVERY_TYPES? │
└──────────────┬──────────────┘
               │ Нет → ПРОПУСК (result = None)
               ▼ Да
┌─────────────────────────────┐
│ Определить volume по nm_id  │
│ (из WB Content API)         │
└──────────────┬──────────────┘
               ▼
┌─────────────────────────────┐
│ 3-tier resolve coef         │
│ Tier1: fixation active?     │
│ Tier2: supabase tariffs?    │
│ Tier3: dlv_prc fallback     │
└──────────────┬──────────────┘
               │ coef == 0 → ПРОПУСК (result = None)
               ▼
┌─────────────────────────────┐
│ Определить base_1l, extra_l │
│ ├─ sub-liter? (>=22.09.2025 │
│ │   AND vol < 1L)           │
│ └─ стандартный период       │
└──────────────┬──────────────┘
               ▼
┌─────────────────────────────┐
│ Определить IL по неделе     │
│ (il_overrides > calculated  │
│  > config.ktr > 1.0)        │
└──────────────┬──────────────┘
               ▼
┌─────────────────────────────┐
│ order_dt >= 23.03.2026?     │
└──────────────┬──────────────┘
       Нет     │     Да
       │       │      │
       ▼       │      ▼
   Старая      │  Новая формула
   формула     │  cost = base * coef * КТР
   cost =      │       + price * КРП%
   base *      │
   coef * IL   │
       │       │      │
       └───────┴──────┘
               ▼
┌─────────────────────────────┐
│ overpayment = delivery_rub  │
│              - calculated   │
└──────────────┬──────────────┘
               ▼
```

## 10. Известные ограничения

1. **Нет API для ИЛ.** WB не предоставляет API для получения Индекса Локализации. Значения берутся из ручного ввода (`il_overrides.json`) или рассчитываются по заказам (менее точно). Требуется регулярная ручная сверка с WB Partners dashboard.

2. **Формула ИРП с 23.03.2026.** Новая формула (ИЛ + ИРП) применяется к заказам с `order_dt >= 2026-03-23`. Требует per-SKU локализации и розничных цен. Если данные недоступны — используется старая формула (только КТР).

3. **Нет монопалет.** Аудит рассчитывает только коробочную логистику. Палетные отправки не аудитируются (тарифы палет загружаются для справки, но в расчёте не участвуют).

4. **Нет DBS (Delivery by Seller).** Аудит покрывает только FBS/FBO модели. Доставка силами продавца не аудитируется.

5. **Sub-liter тиры только с 22.09.2025.** Для заказов до этой даты все товары рассчитываются по стандартным тарифным периодам, независимо от объёма.

6. **Коэффициент склада из dlv_prc не верифицирован.** Tier 3 (fallback) помечается как `verified = False`. Этот коэффициент может отличаться от реального тарифа склада.

7. **Таблица КТР/КРП фиксирована в коде.** Таблица из `irp_coefficients.py` актуальна на 27.03.2026. При обновлении WB — требуется ручное обновление.
