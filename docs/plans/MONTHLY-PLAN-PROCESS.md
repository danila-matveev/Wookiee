# Ежемесячный бизнес-план Wookiee — процесс и спецификация

> Документ для создания скилла/агентов, которые автоматизируют этот процесс.

---

## 1. Что это такое

Ежемесячный процесс построения операционного бизнес-плана на следующий месяц для бренда Wookiee (женское бельё/спорт, WB + OZON). Включает:

- Полный P&L (WB+OZON) за текущий месяц как базу
- Модельную разбивку с классификацией (активные / выводимые)
- Ценовые гипотезы на основе эластичности (артикульный уровень)
- Рекламный бюджет с двумя сценариями
- Остатки и оборачиваемость с оценкой рисков
- Сверку с планом финансиста из Google Sheets
- Понедельный план действий

## 2. Итоги апрельского пилота (что получилось)

### Что сработало хорошо
- **Multi-agent верификация**: 6 параллельных субагентов + 7 кросс-проверок — поймали 9 критических ошибок
- **Два типа маржи**: Маржа-1 (как PowerBI) и Маржа-2 (наша) — устранило путаницу с финансистом
- **Артикульная эластичность**: модельный уровень завышал E в 2-13 раз (Wendy: -5.57 → -0.82, Joy: -6.25 → -1.40)
- **Сверка с финансистом**: план 40.1М vs факт 41.65М — обнаружили структурные риски (Vuki -23%, Lana +149%)
- **Volume-weighting**: корректный пересчёт E с учётом объёмов продаж по SKU
- **Break-even DRR**: для каждой модели рассчитан предельный DRR (Eva DRR 19.3% vs BE 2.9% → убыток)

### Что заняло много ручной работы
- Несколько итераций из-за расхождений с PowerBI (WB-only vs WB+OZON, формула маржи)
- Ручная корректировка после экспертной панели (Charlotte OZON не запускаем, Joy повышение отменено)
- Публикация в 3 канала (MD файл, Notion, Google Sheets) — каждый раз вручную

│   ├── inventory.py   — остатки, оборачиваемость, МойСклад

│   ├── sku_mapping.py — статусы артикулов (Продаётся/Выводим/Архив)
│   └── traffic.py     — трафик по моделям

├── Google Sheets:
│   ├── План финансиста: 1Dsz7s_mZ0wUhviGFho89lyhtjce1V0Cmv_RPL1aLxnk

### Фаза 1: Параллельная верификация (6 субагентов)

| # | Субагент | Данные | Выход |
|---|----------|--------|-------|
| 1 | **P&L Total** | finance.py (WB+OZON) | Полная воронка: заказы→revenue→маржа-1→маржа-2, сравнение м/м |
| 2 | **P&L по моделям** | finance.py + sku_mapping.py | Разбивка активные/выводимые, WB+OZON по каждой модели |
| 3 | **Ценовая эластичность** | pricing_article.py | E и r на артикульном уровне, volume-weighted |
| 4 | **Рекламная эффективность** | advertising.py + Sheets | ROAS, DRR (внутр/внешн), break-even DRR |
| 5 | **Остатки** | inventory.py | Дней стока, оборачиваемость, дефицит/overstock |
| 6 | **ABC + финансист** | planning.py + Sheets | ABC-классы, сверка план vs факт |

### Фаза 2: Кросс-верификация (7 проверок)

1. P&L Consistency: сумма моделей = итого (<1%)
2. Цена × Остатки: хватит ли стока при изменении цены?
3. Реклама × P&L: break-even DRR, кто в убытке?
4. Рост-модели: ROAS + остатки + ABC подтверждают?
5. Убыточные модели: DRR × маржинальность = ROMI
6. Замедление: overstock при снижении продаж?
7. Финансист vs Факт: достижим ли план?

### Фаза 3: Генерация плана

Структура итогового документа (12 разделов A-L):
```
A. P&L Brand Total (WB/OZON/ИТОГО, текущий vs предыдущий, план)
B. P&L по моделям — Активные
C. P&L по моделям — Выводимые
D. Гипотезы (ценовые + рекламные + остатки, с confidence)
E. Бюджет рекламы (два сценария)
F. Остатки и оборачиваемость
G. ABC-анализ и сверка с финансистом
H. Рекламная эффективность (ROAS, DRR, break-even)
I. Ценовая эластичность (volume-weighted)
J. Понедельный план (4 недели с KPI и alarm-триггерами)
K. Результаты верификации
L. Контекст и ограничения
```

### Фаза 4: Публикация (3 канала)

## 4. ПРОБЛЕМНЫЕ ЗОНЫ (критично для автоматизации)

### 🔴 Критические проблемы

**4.1 Fan-out баг в `get_wb_model_ad_roi()`**
- Завышает ad_spend на **39.86%** из-за дублей nmid при JOIN
- Файл: `shared/data_layer/advertising.py:244`
- **Статус: НЕ ИСПРАВЛЕН**
- Workaround: ROAS считается через `get_wb_by_model()` (adv_internal + adv_external)
- **Нужно**: исправить JOIN или заменить функцию

**4.2 Расхождение DB vs Google Sheets по внешней рекламе**
- DB: 919К vs Sheets: 530К (расхождение **73%**)
- Причина не выяснена — возможно разная атрибуция по месяцам
- **Нужно**: выяснить источник правды, привести в соответствие

**4.3 Расхождение с PowerBI по марже**
- Маржа-1 из DB 9.31М vs PowerBI ~8.5М (+812К)
- Причины: 31 день vs 30 дней, возможные фильтры PowerBI
- **Нужно**: стандартизировать дату-конец (`date < первый_день_следующего_месяца`)

### 🟡 Слабые места

**4.4 OZON данные неполные**
- OZON не имеет `reclama_vn_vk` и `reclama_vn_creators` → Маржа OZON = Маржа-1, нельзя посчитать Маржу-2
- Внешняя реклама OZON = 0 в DB (но реально есть, просто не трекается)
- Нет данных по трафику OZON в `traffic.py` (только WB content_analysis)
- **Влияние**: OZON-модели выглядят прибыльнее, чем есть

**4.5 Трафик content_analysis — расхождение ~20% с PowerBI**
- `traffic.py` → таблица `content_analysis`
- Фильтры не выяснены, документировано в `docs/database/DATA_QUALITY_NOTES.md`
- **Влияние**: CR и воронка — только как индикатор тренда, не абсолютные значения

**4.6 Эластичность — ограничения метода**
- Нужно минимум 30-60 дней данных с вариацией цены
- Charlotte (запуск 9 марта) — недостаточно данных, E ненадёжен
- Новые модели первые 2-3 месяца — E= LOW confidence
- Seasonality не учитывается (Q1 vs Q2 может дать разную E)
- **Влияние**: для новых моделей ценовые гипотезы = "держать" по умолчанию

**4.7 Атрибуция WB-склеек**
- Внешний трафик на Wendy → покупки Audrey/Lana/Charlotte через склейки
- В данных это невидимо — Wendy выглядит менее эффективной, Audrey/Lana — более
- **Влияние**: ROAS Wendy занижен, ROAS Audrey/Lana завышен. Ручная экспертная корректировка.

**4.8 Себестоимость Eva**
- Подозрение на ошибку (М-1 16.4% при М-2 2.9%)
- Источник себестоимости — МойСклад, но значение может быть неверным
- **Влияние**: все метрики Eva ненадёжны до пересчёта

### 🟢 Отсутствующие данные (хотелки)

**4.9 Нет данных по воронке OZON**
- `funnel_seo.py` — только WB. OZON воронка (показы→клики→корзина→заказ) отсутствует
- **Влияние**: нельзя оптимизировать CR для OZON

**4.10 Нет данных по конкурентам**
- Цены конкурентов, доли рынка, позиции в поиске — не трекаются
- **Влияние**: ценовые рекомендации в вакууме, без контекста рынка

**4.11 Нет прогнозной модели**
- Текущий подход: тренд Q1 → линейная экстраполяция
- Нет учёта сезонности, праздников, маркетинговых акций WB
- **Влияние**: план может быть неточным в сезонные месяцы (июнь, ноябрь)

## 6. Формула маржи (критичная справка)

```sql
-- Маржа-1 (операционная, как PowerBI): marga - nds
-- Маржа-2 (чистая, наша):
WB:   marga - nds - reclama_vn - COALESCE(reclama_vn_vk, 0) - COALESCE(reclama_vn_creators, 0)
OZON: marga - nds  -- (нет reclama_vn_vk/creators)
```

В итоговом документе **ВСЕГДА показывать оба** показателя.

## 7. Правила качества данных

1. **GROUP BY по модели**: `LOWER(SPLIT_PART(article, '/', 1))` — ВСЕГДА
2. **Даты**: `date < первый_день_следующего_месяца` (НЕ `<=`)
3. **СПП при объединении WB+OZON**: средневзвешенный `sum(spp)/sum(rev)*100`
4. **ДРР**: ВСЕГДА с разбивкой внутренняя/внешняя
5. **Выкуп %**: лаговый (3-21 день), только информационный
6. **Маппинг моделей**: через `shared/model_mapping.py` → `map_to_osnova()`
7. **DB-запросы**: ТОЛЬКО через `shared/data_layer/`. Никакого raw SQL.
