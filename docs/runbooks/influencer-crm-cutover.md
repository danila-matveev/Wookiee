# Influencer CRM — cutover playbook (Sheets → CRM)

Пошаговый сценарий перевода команды с Google Sheets на новый CRM UI.
Документ читается и выполняется человеком: автоматизации нет, каждый пункт нужно проверить руками и пометить.

---

## Когда применять

Применять, когда ETL стабильно отрабатывает 48 часов подряд без ошибок,
дашборд `/ops` зелёный по всем секциям и команда прошла короткий онбординг по новому UI.
Если хотя бы одно условие не выполнено — откладывать, не запускать cutover.

---

## Pre-cutover чек-лист

Все 8 пунктов должны быть пройдены за один проход (не больше суток между пунктами).
Каждая проверка — максимум 30 секунд: одна команда, один экран.

- [ ] **1. `/ops/health` — нет сбоев за 24ч.** `curl -H "X-API-Key: $CRM_API_KEY" https://<bff-host>/ops/health | jq '.etl_last_24h'`. Ожидание: `failed = 0`, `success >= 4`.
- [ ] **2. Свежесть MV.** Из того же ответа: `.mv_age_seconds < 600` (10 минут). Если больше — pg_cron `crm_v_blogger_totals_refresh` не отрабатывает, чинить до cutover.
- [ ] **3. Incremental ETL — без аномалий.** `python -m services.influencer_crm.scripts.etl_runner` (без флагов = `--incremental`) на тихой базе. В логах все листы должны показать `0 new rows` или единичные. Большие дельты = либо кто-то редактирует Sheets, либо ETL ловит фантомные изменения.
- [ ] **4. pg_cron jobs активны.** SQL: `SELECT jobname, active FROM cron.job WHERE jobname LIKE 'crm_%';`. Ожидание: 3 строки, у всех `active = true` (`crm_v_blogger_totals_refresh`, `crm_audit_log_retention`, `crm_metrics_snapshots_retention`).
- [ ] **5. Бэкап Sheets-экспорта сделан.** Файл → Скачать → `.xlsx` для каждой вкладки (bloggers, integrations, briefs, contracts). Сложить в shared-папку с датой. Это страховка на случай rollback.
- [ ] **6. Команде разослан анонс.** В рабочем чате: дата/время freeze, ссылка на CRM, мини-инструкция (3-5 пунктов: где логин, где список блогеров, как создать интеграцию). Подтверждение прочтения от каждого участника.
- [ ] **7. UI протестирован минимум 5 пользователями.** Каждый прошёл базовый сценарий: открыл список блогеров, создал тестовую интеграцию, прикрепил промокод. Ошибок и блокеров нет.
- [ ] **8. Доступы (X-API-Key) розданы.** У каждого пользователя CRM есть рабочий ключ. Проверка — пользователь сам открывает `/bloggers` и видит данные. Не на словах: реальный заход с его машины.

## Cutover sequence

Идти строго по порядку. Не перепрыгивать шаги.

### Шаг 1. Объявление и freeze writes на Sheets

В рабочем чате: "**С XX:00 freeze writes на Sheets, переходим на CRM. Не редактировать таблицы до отдельного сообщения.**"
Зафиксировать время. С этого момента источник истины переезжает в CRM.

### Шаг 2. Финальный полный ETL

```bash
python -m services.influencer_crm.scripts.etl_runner --full
```

Ждать завершения. Exit code 0 = ок. Проверить запись в `crm.etl_runs`:

```sql
SELECT started_at, status, duration_ms, mode, error_message
FROM crm.etl_runs
WHERE agent = 'crm-sheets-etl'
ORDER BY started_at DESC LIMIT 1;
```

Статус должен быть `success`. Если `failed` — STOP, разбираться, не идти дальше.

### Шаг 3. Diff-проверка: Sheets vs CRM

Сверить количество строк. SQL:

```sql
SELECT COUNT(*) FROM crm.bloggers;
SELECT COUNT(*) FROM crm.integrations;
SELECT COUNT(*) FROM crm.content_briefs;
SELECT COUNT(*) FROM crm.contracts;
```

Сверить с pivot-ами/счётчиками в Sheets (нижняя строка каждого листа). Допустимое расхождение — 0.
Если расхождение есть — записать какие листы и сколько, разобрать причину перед шагом 4.

### Шаг 4. Переключение команды на UI

В чат: "**CRM открыт, переходите по ссылке: https://<crm-host>/. Sheets с этого момента read-only.**"
Приложить мини-инструкцию (1 экран): где блогеры, где интеграции, куда писать при ошибке.
Дежурный человек 2 часа онлайн отвечает на вопросы.

### Шаг 5. Sheets → READ-ONLY

На каждый лист (bloggers, integrations, briefs, contracts) поставить header:

```
READ-ONLY — переход на CRM завершён DD-MM-YYYY. Все изменения только в CRM.
```

Дополнительно — снять права на редактирование у всех, кроме админа таблицы.
Это нужно, чтобы случайные правки в Sheets не разъехались с CRM.

## Rollback plan

Применять, если в первые часы после cutover всплывает критичный блокер
(например, UI не открывается у половины команды или ETL начал писать мусор).
Данные не теряются: Sheets неизменён до шага 5 cutover sequence — пока заголовок READ-ONLY ещё не висит, Sheets пригоден к работе.

1. **Разморозить Sheets writes.** В чате: "**Откат на Sheets, CRM временно отключаем. Работаем в Sheets как раньше.**" Если шаг 5 cutover sequence уже выполнен — снять header, вернуть права на редактирование.
2. **Отключить wookiee-cron CRM ETL.** В `deploy/docker-compose.yml` закомментировать строку с `services.influencer_crm.scripts.etl_runner`, на сервере: `docker compose up -d wookiee-cron`. Это остановит запись в CRM из cron — пока чиним, новые правки в Sheets не будут перезаписываться.
3. **Сообщить команде.** В чат: "**Откат завершён, работаем в Sheets. Дата нового cutover — TBD.**" Параллельно — открыть incident-тикет с описанием причины и плана фикса.

## Day-1 monitoring

Первые 24 часа после cutover держать `/ops` открытым и проверять каждый час.

- **`etl_last_24h.failed = 0`** — после cutover sheets ETL продолжает гонять каждые 6ч (страховка от расхождений). Любой `failed > 0` — разбирать сразу, до накопления.
- **`mv_age_seconds < 600`** — MV `v_blogger_totals` обновляется pg_cron-ом каждые 5 минут. Возраст больше 10 минут = pg_cron упал, фронт показывает старые тоталы.
- **`cron_jobs[].active = true` у всех 3 строк.** Если хоть одна `active = false` — `SELECT cron.alter_job(jobid, active := true)` или пере-применить миграцию `010`/`011`.
- **`retention.audit_log_eligible_for_delete` растёт линейно** — норма. Каждый PATCH в CRM пишет строку в `audit_log`, через 90 дней она попадает в очередь на удаление, раз в неделю pg_cron чистит. Резкий скачок (×10 за час) = ловить аномалию в API.

Если все 4 метрики в норме сутки подряд — cutover можно считать завершённым.
