## [2026-05-16] ADR-010: Deterministic Hygiene Scan for Night DevOps

### Статус
Принято

### Контекст
Night DevOps должен начинаться с read-only hygiene scan: один JSON-отчёт в
`.hygiene/reports/`, без веток, коммитов и PR. Фактический запуск
`/hygiene --json-output` через Claude Code action открыл PR #150 и не записал
JSON artifact. Дополнительно Cloudflare publish не получал credentials из
GitHub Actions env, потому что publisher читал только `.env`.

### Решение
1. Для nightly `hygiene-scan.yml` использовать deterministic runner
   `python -m scripts.nightly.hygiene_scan`.
2. Оставить интерактивный `/hygiene` skill для ручных уборок, где PR и
   Telegram-публикации допустимы.
3. Генерировать оба артефакта: JSON для `night-coordinator` и Markdown для
   Cloudflare Pages.
4. Разрешить `cloudflare-pub` читать `CF_ACCOUNT_ID`/`CF_API_TOKEN` из shell env.

### Последствия
- (+) 03:00 scan не может случайно открыть PR.
- (+) `night-coordinator` получает стабильный JSON-контракт.
- (+) Cloudflare publish работает в GitHub Actions без локального `.env`.
- (-) Ночной hygiene scan покрывает только deterministic checks; LLM-heavy
  проверки остаются в ручном `/hygiene`.

## [2026-02-21] ADR-007: Narrow runtime contour and retire Lyudmila + Vasily agent runtime

### Статус
Принято — частично пересмотрен в **ADR-009 (2026-04-26)**: контур, зафиксированный в п.1 ниже, далее сужен (Oleg тоже выведен в архив). Активный контур см. в [docs/architecture.md](architecture.md).

### Контекст
После миграции на новый сервер активный production-контур должен быть упрощён до компонентов, которые реально поддерживаются и покрыты тестами. В репозитории оставались лишние runtime-модули (`agents/lyudmila`, агентный runtime Василия), что усложняло деплой, документацию и контроль качества.

### Решение
1. Зафиксировать активный контур: `agents/oleg`, `agents/ibrahim`, `services/marketplace_etl`, `services/sheets_sync`, `services/wb_localization`, `deploy`.
2. Удалить `agents/lyudmila` из активного runtime; перенести код в архив.
3. Удалить агентный runtime Василия и оставить только сервис WB-локализации:
   - новый entrypoint: `python -m services.wb_localization.run_localization`
4. Ввести обязательный CI quality gate перед deploy.

### Обоснование
- Меньше runtime-компонентов: ниже операционный риск и проще сопровождение.
- Сервисная форма WB-локализации соответствует фактическому использованию (утилита + API trigger).
- Деплой без quality gate приводил к риску выпуска с красными тестами.

### Последствия
- (+) Прозрачный и поддерживаемый прод-контур
- (+) Документация синхронизирована с фактическим runtime
- (+) Деплой последовательный: CI -> deploy
- (-) Исторический код агентов вынесен в архив и не участвует в активной разработке

## [2026-02-16] ADR-006: Уровни автономии, классификация по роли, экономические правила

### Статус
Принято

### Контекст
Проект — система AI-агентов, но не было формализовано: какие действия агент делает сам, какие с одобрения человека. Также отсутствовали экономические правила выбора моделей и классификация агентов по роли (сенсор/аналитик/исполнитель).

### Решение
1. Уровни автономии (0-3) в `docs/guides/agent-principles.md` — каждый инструмент получает явный уровень
2. 3-слойная классификация в `docs/architecture.md` — сенсоры, аналитики, исполнители
3. Экономические правила в `AGENTS.md` — минимальная достаточность модели, confidence-based routing

### Обоснование
- Источник: `06-rules-and-templates.md`, Блок 3 (агентный проект)
- Уровни автономии предотвращают неконтролируемые действия при масштабировании
- Экономические правила формализуют уже существующую практику (95% z.ai / 5% Claude)

### Последствия
- (+) Формализованные правила для проектирования новых агентов
- (+) Контроль затрат при росте системы
- (-) Новые требования при добавлении инструментов (указание уровня автономии)

### Backlog (будущее)
- ROI-трекинг агентов (бизнес-метрика на агента, ежемесячная оценка)
- Auto-downgrade (упрощение агента при отрицательном ROI)
- Бюджет токенов/месяц (лимит на агента, требует биллинг-интеграции)
- Триангуляция (вывод надёжен при 3+ независимых источниках)

## [2026-02-16] ADR-005: Регулярное обновление документации через /update-docs

### Статус
Принято

### Контекст
Документация устаревала при добавлении новых агентов/сервисов — не было процесса проверки актуальности. Ибрагим, например, отсутствовал в `docs/agents/README.md` и `docs/index.md`.

### Решение
Лёгкий многоуровневый подход:
1. Правило в AGENTS.md: обязательное обновление документации при архитектурных изменениях
2. Команда `/update-docs` для Claude Code: аудит документации по запросу
3. Расширенный DoD и PR-шаблон: конкретные чеки на документацию агентов/архитектуры

### Обоснование
- Claude Code команда лучше git hook'а: интеллектуальное сравнение структуры с документацией
- Не блокирует workflow, запускается по запросу
- Правило в AGENTS.md автоматически читается всеми AI-агентами разработки

### Последствия
- (+) Документация не будет устаревать при росте проекта
- (+) Аудит одной командой `/update-docs`
- (-) Требует дисциплины запуска (но AGENTS.md правило подстраховывает)

## [2026-02-16] ADR-004: Рефрейминг документации — от Data Hub к AI Agent System

### Статус
Принято

### Контекст
Проект эволюционировал из аналитической платформы ("Data Hub") в систему AI-агентов для управления бизнесом. Документация не отражала новую концепцию: агенты описывались как "боты", архитектура была data-pipeline-центричной.

### Решение
Обновить позиционирование во всех ключевых документах. Агенты — ядро системы, боты — интерфейсы. Архитектура агент-центричная: данные → shared → агенты → интерфейсы.

### Обоснование
- Фреймирование определяет мышление — разработчики и AI-агенты будут проектировать agent-first
- Бот ≠ агент: разделение помогает проектировать агентов, которые могут работать через разные интерфейсы
- Техническая суть проекта не изменилась — только нарратив документации

### Последствия
- (+) Документация отражает реальную архитектуру и цели проекта
- (+) Единая терминология: "AI-агент" vs "бот"
- (-) Файлы с именами *-bot.md не переименованы (для стабильности ссылок)

## ADR-008 — Refactor v3 Phase 1 (2026-04)

**Статус:** Принято

**Контекст:** Накопленный технический долг после Oleg V2 (200+ файлов), 604 MB бинарных артефактов, stale docs.

**Решение:** 7-PR рефакторинг:

2. Hardening .gitignore
3. Коммит неотслеживаемых сервисов

7. Унификация документации (ONBOARDING, docs/skills/, module READMEs)

**Последствия:** Репозиторий сократился ~65% по файлам Hub, освободилось ≥780 MB диска. Active runtime чётко документирован.
