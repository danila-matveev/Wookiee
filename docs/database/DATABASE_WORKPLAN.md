### 1.1 Разница `marga` vs `marga_union` (WB) — ЧАСТИЧНО РЕШЕНО

**Вывод:** Паттерн такой же как на OZON — предрассчитанная маржа **не включает НДС**. Нужно вычитать `nds`.

| Поле | Формула | Расхождение с PowerBI | Статус |
|------|---------|----------------------|--------|
| `marga_union` | Напрямую | ~10% | Не включает НДС |
| `marga_union - nds` | `marga_union - nds` | Требуется проверка (ожидаем <1%) | Гипотеза |

| `marga` | Напрямую | Неизвестно | Назначение неясно |

**Аналогия с OZON (верифицировано):**
- OZON: `marga` напрямую = неправильно → `marga - nds` = точное совпадение с PowerBI
- WB: `marga_union` напрямую = ~10% расхождение → `marga_union - nds` = вероятно корректно

### 2.4 Связь рекламных таблиц с abc_date

Рекламные таблицы **НЕ связаны с abc_date напрямую** — в них нет поля `article`. Связь устанавливается через промежуточные ключи.

#### WB: wb_adv → abc_date (через nomenclature)

```
wb_adv.nmid → nomenclature.nmid → nomenclature.vendorcode = abc_date.article
```

- `wb_adv` хранит данные по `nmid` (номенклатурный ID WB), а `abc_date` — по `article` (артикул поставщика)
- Чтобы связать, нужен JOIN через таблицу `nomenclature`, которая содержит оба ключа
- Дополнительно совпадают даты: `wb_adv.date = abc_date.date`

```sql
-- Пример: рекламные расходы + маржа по артикулу
SELECT n.vendorcode, SUM(a.sum) as ad_spend, SUM(a.views) as views
FROM wb_adv a
JOIN nomenclature n ON a.nmid = n.nmid
WHERE a.date >= '2026-02-01' AND a.date < '2026-02-06'
GROUP BY n.vendorcode;
```

#### WB: content_analysis → abc_date (напрямую)

```
content_analysis.vendorcode = abc_date.article + date
```

- `content_analysis` уже содержит `vendorcode` (= артикул поставщика), поэтому связь **прямая**
- Совпадение по `vendorcode = article` + `date = date`

#### OZON: ozon_adv_api → abc_date (по SKU)

```
ozon_adv_api.sku = abc_date.sku + operation_date = date
```

- `ozon_adv_api` хранит данные по `sku`, которое есть в `abc_date`
- Связь прямая, без промежуточных таблиц

#### OZON: adv_stats_daily — ТОЛЬКО по РК, НЕ по артикулу

```
adv_stats_daily: id_rk + operation_date (нет sku / article)
```

- `adv_stats_daily` — **агрегатная** таблица на уровне рекламной кампании (РК), не по товару
- В ней есть `id_rk` (ID кампании), `views`, `clicks`, `orders_count`, `rk_expense`
- **Нет** поля `sku` или `article` → невозможно привязать к конкретному товару
- Для потоварной рекламной аналитики на OZON использовать `ozon_adv_api`
- `adv_stats_daily` полезна для общих метрик РК: общий расход, CTR, CPC по кампании

**Вопрос к разработчику:** Подтвердить, что `adv_stats_daily` действительно не связывается с артикулами. Есть ли способ привязать РК к товарам (через `ozon_adv_history` или `details_adv`)?
