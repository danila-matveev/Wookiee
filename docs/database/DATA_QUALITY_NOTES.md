Документ фиксирует обнаруженные проблемы качества данных, их причины и применённые исправления.
Цель — избежать повторения ошибок в будущих отчётах.

## 1. Регистр имён моделей (CRITICAL)

**Проблема:** В базах данных WB и OZON артикулы хранятся с разным регистром.

- **WB `abc_date.article`:** преимущественно lowercase (`wendy/S`), но встречаются Capitalized (`Wendy/S`)
- **OZON `abc_date.article`:** преимущественно Capitalized (`Wendy/S`), но встречаются lowercase

**Последствия без LOWER():**
- `SPLIT_PART(article, '/', 1)` возвращает `wendy` и `Wendy` как разные модели
- GROUP BY создаёт отдельные строки для каждого варианта
- WB ТОП-10 показывал `wendy` = 1,980,624 руб, скрывая `Wendy` = -369,760 руб (негативная маржа)
- Комбинированный рейтинг WB+OZON не объединял каналы (разные ключи)
- Итог: завышенная маржа по моделям на WB, комбинированный рейтинг = WB рейтинг

**Масштаб ошибки (январь 2026):**
| Модель | Без LOWER() | С LOWER() | Разница |
|--------|-------------|-----------|---------|
| wendy (WB) | 1,980,624 | 1,610,864 | -369,760 (-19%) |
| audrey (WB) | 755,319 | 505,712 | -249,607 (-33%) |
| ruby (WB) | 623,119 | 457,723 | -165,396 (-27%) |

**Исправление:**
- `get_wb_by_model()`: `LOWER(SPLIT_PART(article, '/', 1)) as model`
- `get_ozon_by_model()`: `LOWER(SPLIT_PART(article, '/', 1)) as model`
- Файл: `scripts/data_layer.py`

**Правило на будущее:** Все запросы с GROUP BY по имени модели ОБЯЗАНЫ использовать `LOWER()`.

## 2. СПП бренда (ИТОГО) = 0% (BUG)

**Проблема:** При расчёте ИТОГО (WB + OZON) поле `spp_pct` было захардкожено в 0:
```python
t['spp_pct'] = 0  # БЫЛ БАГ
```

**Причина:** При суммировании каналов нельзя просто сложить проценты СПП — нужен средневзвешенный.

**Исправление:**
```python
total_spp_amount = wb_spp_amount + ozon_spp_amount
t['spp_pct'] = (total_spp_amount / total_revenue_before_spp * 100) if total_revenue > 0 else 0
```

**Правило на будущее:** Процентные метрики (СПП%, маржинальность, ДРР) нельзя суммировать напрямую при объединении каналов — только через средневзвешенное от абсолютных значений.

## 6. Известные особенности данных

### WB
- `revenue_spp` = выручка ДО скидки площадки (не интуитивно, суффикс `_spp` = "before SPP")

## 7. WB retention == deduction (дубликация пайплайна) — CRITICAL

**Обнаружено:** 2026-02-10, при сверке отчёта за 09.02.2026 с Power BI

**Проблема:** На дату 09.02.2026 поля `retention` и `deduction` в `abc_date` (WB) содержат **идентичные значения** в каждой из 1668 строк:
- `SUM(retention)` = 21,619.3 руб
- `SUM(deduction)` = 21,619.3 руб
- Совпадение: 1668/1668 строк (100%)

На всех остальных днях февраля 2026 эти поля различаются:
| Дата | retention | deduction | Одинаковые? |
|------|-----------|-----------|-------------|
| 01.02 | 3,955 | 3,411 | нет |
| 02.02 | 11,748 | 5,197 | нет |
| 08.02 | 12,162 | 4,392 | нет |
| **09.02** | **21,619** | **21,619** | **ДА** |

**Последствия:**
- Формула маржи WB вычитает оба поля: маржа занижена на SUM(deduction) = 21,619 руб
- Отчёт показывал маржу бренда 182,799 руб, Power BI показывает ~200,000 руб
- Расхождение: -8.6% (вместо штатных <1%)

**Исправление (автоматическое):**
- Функция `validate_wb_data_quality()` в `data_layer.py` детектирует дубликацию
- `daily_analytics.py` при обнаружении корректирует маржу: +SUM(deduction)
- В отчёт добавляется предупреждение о качестве данных

**Корневая причина:** Баг пайплайна загрузки данных в PostgreSQL. Требуется исправление на стороне ETL.

**Правило:** Всегда проверять `retention != deduction` при сверке с Power BI. При совпадении — один из столбцов содержит дубль.

## 8. Supabase Security — RLS не был включён (CRITICAL)

**Обнаружено:** 2026-02-11, письмо Supabase Security Advisor (25 errors)

**Проблема:** При развертывании схемы `database.sku` в Supabase **не был включён RLS** (Row Level Security) ни на одной из 16 таблиц. Роли `anon` и `authenticated` имели полный доступ (SELECT, INSERT, UPDATE, DELETE, TRUNCATE) ко всем таблицам и views через публичный REST API Supabase.

**Масштаб:** Кто угодно с `anon key` проекта мог через REST API:
- Читать все данные товарной матрицы (артикулы, баркоды, цены, поставщики)
- Модифицировать или удалять любые записи
- Вызывать публичные функции

**Исправление (миграция 005):**
1. RLS включён на всех 16 таблицах
2. Роль `anon` — все права отозваны
3. Роль `authenticated` — ограничена до SELECT only
4. Созданы 32 RLS-политики
5. Default privileges настроены на запрет для `anon`

**Файл:** `database/sku/scripts/migrations/005_fix_supabase_security.py`

**Дополнительное исправление (миграция 006, 2026-02-25):**
Supabase Security Advisor продолжал показывать 9 ошибок после миграции 005.

**Фактическая причина 9 ошибок:** все 9 views в public schema использовали SECURITY DEFINER (default) — запросы к views выполнялись с правами владельца (postgres), обходя RLS. Любой пользователь через REST API мог обойти RLS-политики, запрашивая views вместо таблиц.

Исправлено миграцией 006:
- `ALTER VIEW ... SET (security_invoker = true)` для всех 9 views (основное исправление)
- `ALTER FUNCTION ... SET search_path = public` для всех 4 функций (доп. hardening)
- `REVOKE ALL ON SEQUENCE ... FROM anon` для всех sequences (доп. hardening)
- Заменены `anon_deny` политики на стандартные на 5 таблицах lyudmila_* (доп. hardening)
- `REVOKE CREATE ON SCHEMA public FROM public`
- `ALTER DEFAULT PRIVILEGES` для предотвращения будущих утечек на sequences

**Файл:** `database/sku/scripts/migrations/006_fix_remaining_security.py`
**Диагностика:** `database/sku/scripts/migrations/diagnose_security.py`

**Правило:** При создании новой таблицы в Supabase — обязательно RLS + политики (см. `database/sku/README.md`).

## 10. WB: 11-полевая формула маржи не учитывала возвраты (CRITICAL)

**Обнаружено:** 2026-02-11, при сверке отчёта за 10.02.2026 с OneScreen

**Проблема:** Формула маржи WB из 11 полей (`revenue_spp - comis_spp - logist - sebes - reclama - reclama_vn - storage - nds - penalty - retention - deduction`) давала 245 207 руб, а OneScreen — 239 199 руб. Расхождение: +2.51%.

**Корневая причина:** 11-полевая формула НЕ учитывает возвраты. Поле `revenue_spp` включает выручку по возвращённым товарам, но `revenue_return_spp` (11 304 руб) не вычитается. Аналогично `sebes` включает себестоимость возвратов (`sebes_return` = 2 074 руб), а `logist` — логистику возвратов (`logis_return_rub` = 250 руб).

**Правильная формула WB:** `SUM(marga) - SUM(nds) - SUM(reclama_vn)`
- Поле `marga` рассчитывается на стороне WB и уже учитывает все возвраты
- Аналогично OZON: `SUM(marga) - SUM(nds)` — но WB дополнительно вычитает `reclama_vn` (внешняя реклама)
- Верификация: 239 199.23 руб vs OneScreen 239 199 руб (расхождение 0.23 руб, < 0.001%)

**Также исправлено:**
- `SUM(revenue_spp)` → `SUM(revenue_spp) - COALESCE(SUM(revenue_return_spp), 0)` для чистой выручки
- Верификация: 1 120 531.3 руб vs OneScreen 1 120 531 руб (расхождение 0.3 руб)

**Не исправлено (известное ограничение):**
- WB `full_counts` — gross продажи (включая возвраты). Поле `count_return` НЕ СУЩЕСТВУЕТ в таблице `abc_date` (несмотря на документацию в DATABASE_REFERENCE.md). Чистые продажи WB отличаются от OneScreen на ~5 шт (< 1%).

**Файл:** `scripts/data_layer.py` — `WB_MARGIN_SQL`, все функции `get_wb_*`

**Правило:** WB маржа = `marga - nds - reclama_vn`. НЕ использовать 11-полевую формулу. WB выручка до СПП = `revenue_spp - revenue_return_spp`.

## 13. Ложноположительное уведомление "Данные готовы" (BUG)

**Обнаружено:** 2026-02-18

**Проблема:** Агент Олег отправил уведомление "Данные за 17.02.2026 готовы", но по Power BI данные WB по марже доступны только до 16.02. Уведомление было преждевременным — маржинальные данные за 17.02 загружены не полностью.

**Корневая причина:** Слишком мягкие пороги в `DataFreshnessService._check_mp()`:
- Порог выручки: 50% от предыдущего дня (недостаточно строго)
- Порог строк с маржой: 50% от предыдущего дня (недостаточно строго)
- Отсутствовала проверка абсолютного заполнения маржи (доля от общего числа строк)
- Отсутствовала санитарная проверка соотношения маржа/выручка

ETL-пайплайн загрузил строки с revenue из API маркетплейса, но marga ещё не была полностью рассчитана. Проверка с порогом 50% пропустила частично загруженные данные.

**Исправление:**

| Проверка | Было | Стало |
|----------|------|-------|
| Выручка vs пред. день | 50% | 70% |
| Строки с маржой vs пред. день | 50% | 90% |
| Строки с маржой vs всего строк | — | 90% (новое) |
| Маржа/выручка ratio | — | 5% (новое) |
| MAX(date) = вчера | — | exact match (новое) |

**Файлы:** `scripts/check_data_ready.py` (ранее: `agents/oleg/services/data_freshness_service.py` — модуль выведен в апреле 2026, проверки не выполняются автоматически)

**Правило:** При проверке готовности данных пороги заполнения маржи должны быть >= 90%. Частичная загрузка финансовых данных не должна считаться "готовой".

## 15. WB: Fan-out в JOIN wb_adv × nomenclature (CRITICAL)

**Обнаружено:** 2026-02-28, при сверке маркетингового отчёта за 16-22.02.2026

**Проблема:** `JOIN nomenclature n ON w.nmid = n.nmid` раздувал суммы расходов ~4x. Таблица `nomenclature` имеет уникальный ключ `(nmid, barcod, lk)` — один `nmid` = несколько строк (разные размеры/баркоды).

**Масштаб:**
- SUM(sum) по моделям через JOIN: ~1.9M ₽
- SUM(sum) raw (без JOIN): ~493K ₽
- Завышение: ~4x

**Затронутые функции:**
- `get_wb_traffic_by_model()` — модельная разбивка рекламного трафика
- `get_wb_model_ad_roi()` — ROI рекламы по моделям (ДРР, ROMI)

**Исправление:** Заменён JOIN:
```sql
-- Было:
JOIN nomenclature n ON w.nmid = n.nmid
-- Стало:
JOIN (SELECT DISTINCT nmid, vendorcode FROM nomenclature) n ON w.nmid = n.nmid
```

Добавлена функция `get_wb_ad_totals_check(start_date, end_date)` — sanity check для автоматической верификации: сравнивает raw SUM vs model-aggregated. Расхождение должно быть <1%.

**Правило:** При JOIN к `nomenclature` по `nmid` ВСЕГДА дедуплицировать: `SELECT DISTINCT nmid, vendorcode FROM nomenclature`.

## 16. WB: Органика vs Платное — сравнение несопоставимых метрик

**Обнаружено:** 2026-02-28, пользователь видел ~50% органических показов, отчёт показывал 24.3%

**Проблема:** `organic_views` (card_opens из content_analysis) складывался с `paid_views` (ad_impressions из wb_adv). Это разные метрики:
- card_opens = пользователь открыл карточку товара (переход)
- ad_impressions = рекламный баннер показан в поиске (показ)

**Исправление:** Сопоставимые пары:
- Переходы: organic_card_opens + paid_clicks
- Корзина: organic_add_to_cart + paid_to_cart
- Заказы: organic_orders + paid_orders
- Рекламные показы (impressions) — отдельная метрика, НЕ суммируется с органикой

**Правило:** НЕ складывать card_opens с ad_impressions. Сопоставимые метрики: card_opens ↔ clicks (оба = переходы в карточку).

## 17. OZON revenue_before_spp не вычитала возвраты (CRITICAL — FIXED)

**Проблема:** В `get_ozon_finance()` и `get_ozon_by_model()` поле `revenue_before_spp` считалось как `SUM(price_end)` без вычета возвратов. WB аналогично вычитала `revenue_return_spp`, а OZON — нет.

**Следствие:**
- OZON выручка завышалась на сумму возвратов (~53К₽/неделю при текущих объёмах)
- Маржинальность OZON занижалась (20,1% вместо реальных 21,1%)
- Маржинальность бренда занижалась (19,8% вместо 19,9%)
- Все отчёты до 13.04.2026 содержат эту ошибку

**Данные (неделя 06-12 апреля 2026):**
| Метрика | До фикса | После фикса (= OneScreen) |
|---|---|---|
| OZON выручка до СПП | 1 120 115 | 1 067 285 |
| OZON маржинальность | 20,1% | 21,1% |
| Бренд выручка | 8 474 026 | 8 421 196 |
| Бренд маржинальность | 19,8% | 19,9% |

**Причина:** Поле `return_end` в OZON `abc_date` хранится как отрицательное число. Для корректного вычета его нужно ПРИБАВЛЯТЬ к `price_end`.

**Исправление (2026-04-13):**
- `get_ozon_finance()`: `SUM(price_end) + COALESCE(SUM(return_end), 0) as revenue_before_spp`
- `get_ozon_finance()`: `SUM(price_end_spp) + COALESCE(SUM(return_end_spp), 0) as revenue_after_spp`
- `get_ozon_finance()`: `SUM(count_end) - COALESCE(SUM(count_return), 0) as sales_count`
- `get_ozon_by_model()`: аналогичные правки для `revenue_before_spp` и `sales_count`
- Файл: `shared/data_layer/finance.py`
