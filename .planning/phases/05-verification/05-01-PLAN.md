Search the Notion database (ID: `30158a2bd58780728785cfd6db66eb82`) for the best existing report of EACH of the 8 types. For each type, use the Notion MCP tools to:
1. Query the database filtering by "Тип анализа" property matching the type label from `shared/notion_client.py`:
   - `daily` -> "Ежедневный фин анализ"
   - `weekly` -> "Еженедельный фин анализ"

   - `localization_weekly` -> "Анализ логистических расходов"
2. Sort by "Период начала" descending, take last 5 per type
3. Read content of each candidate, select the best one by D-02 criteria: longest content, all sections present, real data (not placeholders), deepest analysis

The reference standards file format:
```markdown
# Reference Standards for Report Verification
## {Report Type} — {display_name_ru}
- **Notion Page ID:** {id}
- **Date:** {date}
- **Content Length:** {chars}
- **Sections Found:** {list of ## headings}
- **Quality Notes:** {what makes this the best example}
- **Key Metrics Present:** {list of metrics found in the report}
```

**Step 3 — Generate daily report (RPT-01):**

Run: `python3 scripts/run_report.py --type daily --date <determined_date>`

Capture the output (Notion URL, success/failure, warnings).

**Step 4 — Verify daily report quality (D-04 four criteria):**

1. **Completeness (D-04.1):** Read the generated Notion page. Verify ALL `## ` sections from `daily.md` template are present and filled with real numbers, not "Данные временно недоступны" placeholders.

2. **Depth (D-04.2):** Daily = compact summary (per D-04). Verify it contains: key daily metrics summary, WB + Ozon revenue, orders count, margin. NOT trends/hypotheses (that's weekly depth).

3. **Accuracy (D-04.3, D-05):** SQL-verify against DB using `shared/data_layer.py`:
   - Total daily revenue: `SELECT COALESCE(SUM(revenue_rub), 0) FROM fin_data WHERE date = '<date>'` — compare with report figure, must be within 5%
   - Daily orders: `SELECT COALESCE(SUM(orders_rub), 0) FROM abc_date WHERE date = '<date>'`
   - DRR: `SELECT SUM(advertising_cost) / NULLIF(SUM(orders_rub), 0) * 100 FROM advertising WHERE date = '<date>'`
   - Adequacy check: revenue > 0, revenue < 500M; orders > 0, orders < 100K; DRR between 0-100%

4. **Format (D-04.4):** Verify toggle-headings present (## with toggle markers), Russian language, professional structure matching the reference standard.

**Additional mandatory checks (per AGENTS.md rules):**
- DRR MUST have internal/external split — search report for "ДРР внутренняя" and "ДРР внешняя" (or equivalent split)
- Daily report MUST NOT use Выкуп% as causation driver — search for phrases like "выкуп" near "причина" or "маржа"

**Step 5 — Diagnose and fix issues (D-11):**

If any criterion fails, diagnose the root cause using the Pattern 4 table from RESEARCH.md:
- Wrong sections / missing depth → fix `agents/oleg/playbooks/templates/daily.md`
- Placeholder text → check tool output in pipeline, fix data_layer or agent tools
- Numbers mismatch → check SQL aggregation, LOWER() compliance
- Not published → check gate_checker logs

Make the fix, then re-run: `python3 scripts/run_report.py --type daily --date <date>`
Iterate until all 4 criteria pass.

After each code fix, run regression tests:
```bash
python3 -m pytest tests/agents/oleg/playbooks/ tests/agents/oleg/runner/ -q --tb=short
```

**Step 6 — Compare with reference:**

Compare the verified daily report with the reference standard from Step 1. Note improvements or regressions. The new report should be at least as good as the reference (same or more sections, real data, correct metrics).

**Weekly report (RPT-02) — verify per D-08 order:**

1. Determine date: use last Monday or most recent week with complete data. Run:
   `python3 scripts/run_report.py --type weekly --date <last_monday>`

2. Verify 4 criteria (D-04):
   - **Completeness:** All sections from `weekly.md` template present with real data
   - **Depth (D-04.2 — weekly = deep):** MUST contain: trends analysis (week-over-week comparisons), per-model breakdown, hypotheses for metric changes, revenue/margin trends. Check that report has phrases indicating trend analysis (e.g., "рост/снижение", "динамика", "тренд", "по сравнению с прошлой неделей")
   - **Accuracy (D-05):** SQL-verify via `shared/data_layer.py`:
     ```sql
     SELECT LOWER(SPLIT_PART(article, '/', 1)) as model,
            SUM(revenue_rub), SUM(margin), SUM(orders_rub)
     FROM fin_data
     WHERE date BETWEEN '<week_start>' AND '<week_end>'
     GROUP BY LOWER(SPLIT_PART(article, '/', 1))
     ORDER BY SUM(revenue_rub) DESC
     ```
     Compare top-3 model revenues with report figures. Must be within 5%.
     Verify DRR has internal/external split.
   - **Format:** Toggle headings, Russian, matches reference standard structure

3. If issues found — diagnose root cause (D-11), fix at correct layer:
   - Shallow analysis (no trends/hypotheses) → strengthen weekly.md template depth instructions: add explicit "ОБЯЗАТЕЛЬНО: анализ трендов неделя к неделе" and "ОБЯЗАТЕЛЬНО: гипотезы по изменениям ключевых метрик"
   - Missing model breakdown → check data-map.md tool mapping, verify `get_brand_finance` returns per-model data
   - Number mismatch → check SQL aggregation uses LOWER() per AGENTS.md

4. Re-run and iterate until all criteria pass.

**Monthly report (RPT-03) — verify per D-08 order:**

1. Determine date: use 1st of current month or recent month with complete data. Run:
   `python3 scripts/run_report.py --type monthly --date <month_start>`

2. Verify 4 criteria (D-04):
   - **Completeness:** All sections from `monthly.md` template present with real data
   - **Depth (D-04.2 — monthly = maximum):** MUST contain ALL of:
     - P&L (profit and loss) section with revenue, costs, margin
     - Unit economics (per-order/per-model economics)
     - Strategic recommendations section
     - Month-over-month comparisons
     Check for presence of keywords: "P&L" or "прибыль и убытки", "юнит-экономика" or "юнит" or "на единицу", "стратег" or "рекомендации"
   - **Accuracy (D-05):** SQL-verify monthly totals:
     ```sql
     SELECT SUM(revenue_rub), SUM(margin), SUM(orders_rub)
     FROM fin_data
     WHERE date >= '<month_start>' AND date < '<month_end>'
     ```
     Verify total revenue within 5% of report figure.
     Verify DRR with internal/external split.
     Adequacy: revenue > 10M (monthly should be at least 10M given ~35-40M/month business per user goals)
   - **Format:** Toggle headings, Russian, matches reference standard structure

3. If issues found — same diagnosis pattern (D-11). Monthly-specific fixes:
   - Missing P&L → add explicit "## P&L" section requirement to monthly.md template
   - Missing unit economics → add explicit depth instruction: "ОБЯЗАТЕЛЬНО: юнит-экономика — маржа на заказ, себестоимость на единицу"
   - Missing strategy → add "ОБЯЗАТЕЛЬНО: стратегические рекомендации на следующий месяц"

4. Re-run and iterate until all criteria pass.
