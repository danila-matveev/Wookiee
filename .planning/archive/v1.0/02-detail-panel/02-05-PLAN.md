## Task 2: Add children_count to ModelOsnova

**Goal**: `GET /api/matrix/models/{id}` returns `children_count` with actual count of child artikuly.

**Root cause**: `ModelOsnovaRead.children_count` is declared but never computed. `ModelOsnova` SQLAlchemy model has `modeli` relationship but no `children_count` property.

**Implementation**: Add `column_property` with correlated subquery to `ModelOsnova`.

**File**: `sku_database/database/models.py`

Add after the relationship declarations on `ModelOsnova` (after line 246):

```python
from sqlalchemy.orm import column_property
from sqlalchemy import select, func

# On ModelOsnova class, after modeli relationship:
children_count = column_property(
    select(func.count(Model.id))
    .where(Model.model_osnova_id == id)
    .correlate_except(Model)
    .scalar_subquery()
)
```

Note: `column_property` is evaluated at query time (single SQL), not lazy-loaded. This avoids N+1 on list endpoints.

Alternative (simpler, but N+1 on lists): `@hybrid_property` with `len(self.modeli)`. Prefer `column_property` for correctness.

**Verify**: `GET /api/matrix/models/7` → `children_count: 11` (or whatever the actual count is). Header badges appear in the panel.

## Risk

- **Task 1**: Migration uses hardcoded IDs (27-50). If field_definitions IDs differ in production, the migration will fail silently. Mitigation: use `field_name + entity_type` WHERE clauses instead of IDs.
- **Task 2**: `column_property` adds a correlated subquery to every ModelOsnova query. For 50-100 models this is negligible. If performance becomes an issue, replace with a `@hybrid_property` + explicit annotation on list endpoint.
