# Пульс команды · WOOKIEE (v2)

## Версия v2 — что изменилось

С версии v2 скилл собирает **6 источников** вместо 2:
1. Задачи Битрикса
2. Групповые чаты (включая обсуждения внутри задач)
3. **Календарь каждого штатного сотрудника** (`calendar.event.get`)
4. **Личные диалоги Данилы с каждым штатным сотрудником** (`im.dialog.messages.get`)
5. **Notion — планерка прошлой недели** (per-person отчёты, транскрипт, фоллоуапы)
6. Notion — планерка текущей недели (черновик на понедельник)

Финальный отчёт публикуется в Notion с красивым форматированием (callouts, цветовые акценты, таблицы).

```bash
cd modules/bitrix-analytics && python3 fetch_data.py --days $ARGUMENTS
```

Параллельно с шагом 1 (или после) — получить планерки:

1. Найти страницу планерки **прошлой недели** в базе «Планерка — Еженедельные отчёты команды» (page_id: `31e58a2bd5878017acf8cc6fa2b657a9`) через `notion-search` с `content_search_mode: workspace_search`, запрос формата «Итоги команды WOOKIEE за [даты]».
2. Через `notion-fetch` получить содержимое — включает: per-person отчёты, ссылку на запись встречи, транскрипт, фоллоуапы.
3. Также проверить текущую неделю (черновик) — если что-то уже заполнено, включить в анализ.

## 5. Privacy-фильтр (критично)

**Тест планерки:** если факт **не мог бы прозвучать на планерке при всей команде** — он **не идёт в отчёт**. Отчёт публикуется в общекомандный Notion и может быть прочитан всеми.

### НЕ публиковать:
- **Увольнения** — обсуждение чьего-то возможного увольнения, недовольство конкретным сотрудником.
- **Пугающие операционные решения** — реструктуризация, сокращения, урезание бюджетов, закрытие направлений.
- **Личная информация** — здоровье, беременность, семейные обстоятельства, финансовые проблемы. Если снижение активности влияет на работу — описывать нейтрально («в сокращённом режиме», «возвращается к работе»), без причины.
- **Оценочные суждения Данилы** — «плохо работает», «не тянет», «медленный». Факты можно, оценки — нет.
- **Конфиденциальные цифры** — долги, кассовые разрывы, если не анонсированы команде.
- **Корпоративные интриги** — конфликты между сотрудниками, которые Данила обсуждает с кем-то одним.
- **Source tags** — никогда не писать в отчёте `[из: лички]`, `[из: чатов]` и т.п. Источники используются только для внутренней верификации фактов при генерации.

### Правила формулировок
- Больничный/декрет/личные обстоятельства → «в сокращённом режиме» / «возвращается к работе».
- Стратегические сдвиги (передача процессов) → «плановый переход», не «выводим».
- Негативная оценка работы → описывать факт без цитаты оценки (например, «процессы контента оптимизируются», а не «Данила считает, что Алина тормозит»).

## 6. Шаблон отчёта (v4 — тоггл-заголовки + читаемая иерархия)

### Принципы форматирования

**Главное правило: воздух между абзацами.** Никогда не лепи всё в один блок. Notion — рендерит markdown с двойным переносом строки как новый параграф.

**Тоггл-заголовки — обязательны.** Все детальные секции должны быть свёрнуты по умолчанию, чтобы при открытии страницы читатель видел только главные итоги (callouts) и executive summary. Тоггл — это `{toggle="true"}` после заголовка, дочерние блоки внутри ОБЯЗАТЕЛЬНО индентятся табами (`\t`), иначе они не попадут внутрь тоггла.

```markdown
### 📦 Маркетплейсы · Настя {toggle="true"}
\t#### Что сделано
\t- буллет 1
\t- буллет 2
```

**Где применять тоггл:**
- ✅ `## 🏆 Молодцы недели` → `{toggle="true"}` (можно свернуть)
- ✅ `## 🧭 Вектор` → `{toggle="true"}`
- ✅ Каждое `### Direction` → `{toggle="true"}` (главный паттерн)
- ✅ `## 🔥 Просрочено / Горит` → `{toggle="true"}`
- ✅ Подсекции внутри `## 🔥 Пульс недели` (`### По цифрам`, `### Активность команды`) → `{toggle="true"}`
- ❌ `## 🔥 Пульс недели` (executive summary — всегда виден)
- ❌ `## 🏢 По направлениям` (это просто заголовок-разделитель)
- ❌ `## 🎯 Фокус` (CEO action items — всегда видны)
- ❌ Callouts «🎯 Главный итог» — НИКОГДА не прячутся в тоггл, они идут ПЕРЕД `### Direction`

**Иерархия:**
- `##` — главные секции (Пульс / Молодцы / Вектор / По направлениям / Фокус / Горит)
- `###` — направление (👑 Операционный слой · ... / 🛍 Продукт · ...) — всегда `{toggle="true"}`
- `####` — имя человека внутри направления с несколькими людьми (внутри тоггла, не сами по себе тогглы)
- `---` — горизонтальный разделитель между секциями и между направлениями

**Каждое направление ОБЯЗАТЕЛЬНО предваряется callout'ом с «Главным итогом» — ВНЕ тоггла:**

```markdown
<callout icon="🎯" color="...">
\t**📦 Маркетплейсы — главный итог:** [одна-две строки — ключевая мысль или решение]
</callout>

### 📦 Маркетплейсы · Настя {toggle="true"}
\t[содержимое — детали внутри тоггла]
```

Это даёт CEO дашборд из 9 цветных callouts (по одному на направление) сверху вниз — можно скан-чтением понять состояние каждого направления за 30 секунд, и развернуть только те, где нужны детали.

**Цвет callout'а «Главный итог» по направлению:**
- 👑 Операционный слой → `blue_bg`
- 🛍 Продукт → `purple_bg`
- 📦 Маркетплейсы → `orange_bg`
- 🎨 Контент → `yellow_bg`
- 🤝 SMM → `pink_bg`
- 📈 Маркетинг → `orange_bg` (если проблема) / `green_bg` (если победа)
- 🎭 Инфлюенс → `green_bg`
- 💰 Финансы → `red_bg` (если есть критичные цифры) / `gray_bg`
- 📦 Склад → `green_bg`

## 7. Принципы генерации

**Стиль — аналитика, не списки:**
- Каждая секция направления = 2-4 плотных предложения с конкретикой.
- Выделять **жирным** ключевые находки, метрики, идеи.
- Конкретные цифры, факты из обсуждений (цены, конверсии, сроки).
- Рутинные задачи не перечислять — «закрывает операционку».
- Фокус на том, что **происходит** (динамика), а не что **числится**.

**Что брать из каждого источника:**
- **Задачи** → что движется, что закрыто, что просрочено.
- **Групповые чаты** → обсуждения, идеи, блокеры, внешние факторы.
- **Чаты в задачах** → конкретные решения, согласования.
- **Календарь** → интенсивность, сеть связей, ритуалы (Dayli/планерки).
- **Лички Данилы** → стратегический слой (с фильтром).
- **Notion-планерка** → плановые метрики, фокусы, фоллоуапы, транскрипт.

**Запрещено:**
- Писать в отчёте, откуда взят факт (никаких `[из: ...]`).
- Раскрывать личные обстоятельства сотрудников.
- Делать негативные выводы на основе количества задач (роли измеряются по-разному: реклама — ДРР/ROMI, SMM — охваты).
- Цитировать оценочные суждения из лички Данилы.
- Дублировать задачу между секциями.
- Перечислять все открытые задачи — только ключевые и просроченные.

## Логирование (выполнить всегда в конце)

Прочитай `USER_EMAIL` из `.env`. Рассчитай длительность от старта до завершения публикации в Notion.

Выполни через Supabase MCP (`execute_sql`, project `gjvwcdtfglupewcwzfhw`):

```sql
WITH ins AS (
  INSERT INTO tool_runs (
    id, tool_slug, status, trigger_type, triggered_by,
    result_url, items_processed, notes,
    started_at, finished_at, duration_sec
  ) VALUES (
    gen_random_uuid(), '/bitrix-analytics',
    '{status}', 'manual', 'user:{USER_EMAIL}',
    '{notion_url}', {items_count}, '{brief_notes}',
    now() - interval '{duration_sec} seconds', now(), {duration_sec}
  ) RETURNING tool_slug
)
UPDATE tools SET
  total_runs = total_runs + 1,
  last_run_at = now(),
  last_status = '{status}',
  updated_at = now()
WHERE slug = '/bitrix-analytics';
```

Где: `{status}` = `success` или `error`, `{notion_url}` = URL страницы в Notion, `{items_count}` = количество задач/участников в отчёте, `{USER_EMAIL}` из `.env` (или `unknown`), `{duration_sec}` = целое число секунд.
