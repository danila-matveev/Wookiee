## Задачи

### A. Детекция аномалий по долям

Если доля метрики в структуре изменилась > 3 п.п. — это флаг.

| Метрика | Доля тек., % | Доля пред., % | Delta п.п. | Эквивалент, руб | Severity |
|---|---|---|---|---|---|

Проверить доли:
- Себестоимость / выручка
- Логистика / выручка
- Хранение / выручка
- Комиссия / выручка
- Реклама внутр. / выручка
- Реклама внешн. / выручка
- НДС / выручка
- Маржа / выручка

Пороги severity:
- **HIGH**: delta > 5 п.п. ИЛИ эквивалент > 100 000 руб
- **MEDIUM**: delta 3-5 п.п.
- **LOW**: delta < 3 п.п. (не флагуется)

### B. Проверка себестоимости

Если себестоимость в долях от выручки резко меняется (например, 20% -> 25%):
- Вероятно ошибка данных, а не реальное изменение
- Проверить: изменился ли ассортиментный микс? Изменились ли фактические себестоимости?
- Если по отдельным моделям себестоимость стабильна, а доля выросла — это ассортиментный сдвиг
- Если по конкретной модели себестоимость скакнула — вероятно ошибка в БД

### C. Анализ логистики

| Модель | Логистика/ед тек. | Логистика/ед пред. | Delta% | Тариф МП | Индекс локализации | Комментарий |
|---|---|---|---|---|---|---|

Проверить:
- Изменились ли тарифы WB/OZON?
- Изменилась ли география заказов (индекс локализации)?
- Изменился ли размерный/весовой микс?

### D. Data Quality проверки

1. **retention == deduction (дубли):** В WB abc_date поля `retention` и `deduction` часто дублируют друг друга. Проверить: если на одних датах оба поля ненулевые и равны — это дубль. НЕ суммировать.

2. **Fan-out JOIN:** При JOIN abc_date с orders/sales по баркоду без агрегации строки размножаются. Проверить: если рекламный расход оказался в X раз больше ожидаемого — вероятно fan-out.

3. **Трафик ~20% gap:** Данные `content_analysis` занижают трафик на ~20% vs PowerBI. Не использовать абсолютные значения для сравнений. Только тренды.

4. **OZON органика = 0:** OZON не предоставляет данные об органическом трафике. Если в данных есть ненулевая органика — это оценка, не факт.

5. **Неполные месяцы:** Если анализируемый период — текущий месяц, данные неполные. Пометить: "данные за N дней из M".

### E. Глубокая диагностика моделей

Для каждой модели с изменением заказов > +/-10%:

| Модель | Заказы Delta% | Проверка | Результат |
|---|---|---|---|

Диагностический чеклист:
1. **Цена до/после СПП** — изменения за последние 3 дня? Если да — ценовой фактор.
2. **Наличие на складе (OOS)** — выпали ли ключевые размеры? Если да — OOS-фактор.
3. **Органический трафик** — показы/переходы/конверсия карточки. Если трафик упал — позиции или конкуренты.
4. **OZON отдельно** — оборачиваемость, хранение, наличие по цветам/размерам. Если проблема только на одной площадке — локальный фактор.
5. **Рекламный бюджет** — изменился ли? Если да — рекламный фактор.
6. **Сезонность** — есть ли сезонный паттерн из прошлых лет?

Для каждой модели: 1-2 наиболее вероятные причины с обоснованием из данных.

### F. Cross-check арифметики

1. Сумма маржи моделей = маржа бренда (+/- 1%)
2. Сумма выручки WB + OZON = выручка бренда (+/- 1%)
3. ДРР = реклама / выручка (проверить расчет)
4. Оборачиваемость = остаток / дневные продажи (проверить)
5. ROI = маржин-ть * (365 / оборачиваемость) (проверить)

## Жесткие правила

1. **Каждая аномалия** — с severity (HIGH/MEDIUM) и эквивалентом в рублях.
2. **Data quality проблемы** — не интерпретировать как бизнес-изменения. Если подозрение на ошибку данных — маркировать.
3. **GROUP BY модели** = `LOWER(SPLIT_PART(article, '/', 1))`.
4. **Формат чисел:** `1 234 567 руб`, `24.1%`, `+3.2 п.п.`.
5. **Тон** — сухой, аналитический. Без спекуляций — только факты и обоснованные гипотезы.
