# Workflow Diagram Generator

Генерирует красивые интерактивные visual workflow как standalone HTML-файлы.
Пользователь открывает файл в браузере и видит профессиональную диаграмму с zoom, pan, dark mode и click-to-detail.

## Когда вызывать (автоматический триггер)

Вызывай этот скилл **автоматически** при brainstorming/design если задача содержит:
- **3+ последовательных шагов** (pipeline, chain, sequence)
- **Условные ветвления** (gates, decisions, if/else в логике)
- **Data flow между компонентами** (API -> ETL -> DB -> отчёт)
- **Agent pipeline или bot logic** (orchestrator -> agents -> output)
- **Бизнес-процесс с фазами** (как NATN: preparation -> generation -> QC -> publication)

### Когда НЕ вызывать
- Простой bugfix без flow
- Добавление UI-элемента (кнопка, поле)
- Изменение конфигурации
- Рефакторинг без изменения архитектуры

## Момент вызова

В рамках brainstorming: **после согласования дизайна, для визуальной валидации архитектуры**.
При ручном вызове `/workflow-diagram`: сразу генерировать.

## КРИТИЧНО: Человечный язык

Диаграммы читает бизнес-пользователь без технического бэкграунда. **Все тексты пишутся для человека, не для разработчика.**

### Правила написания текстов

1. **Названия нод** — что делает этот шаг с точки зрения пользователя:
   - Плохо: `Reporter Agent`, `APScheduler Cron`, `PostgreSQL wb_sales`
   - Хорошо: `Аналитик собирает цифры`, `Автоматический запуск по расписанию`, `База данных с продажами WB`

2. **Подписи на стрелках** — что происходит при переходе:
   - Плохо: `SQL query`, `API call`, `JSON response`
   - Хорошо: `запрашивает продажи`, `отдаёт данные`, `✅ всё ок, готовим отчёт`

3. **Заголовки subgraph** — фаза/группа с эмодзи:
   - Плохо: `Data Sources`, `ETL Pipeline`, `Output Layer`
   - Хорошо: `🌐 Откуда берутся данные`, `🔄 Обработка`, `📬 Куда приходят результаты`

4. **Sidebar описания** — подробно и понятно, объясняют ЧТО и ЗАЧЕМ:
   - Плохо: `ReAct loop with 30 SQL tools, hypothesis-driven search`
   - Хорошо: `Когда в цифрах что-то не сходится, этот модуль начинает разбираться: сравнивает с прошлыми периодами, ищет аномалии, проверяет, не было ли технических сбоев.`

5. **Запрещённые слова** в нодах и стрелках: API, SQL, JSON, Docker, PostgreSQL, agent, orchestrator, middleware, ETL, pipeline, ReAct, LLM, cron. Используй понятные аналоги.

6. **Допустимо в sidebar** (в поле notes): упоминать технологии кратко, если это помогает контексту. Но основной текст description — всегда для бизнес-пользователя.

## Как генерировать

### Шаг 1: Определи тип диаграммы

| Потребность | Mermaid тип | Направление |
|---|---|---|
| Agent pipeline | `flowchart TD` + subgraphs | сверху вниз |
| Data flow / ETL | `flowchart LR` | слева направо |
| Бизнес-процесс с gates | `flowchart TD` + diamonds | сверху вниз |
| API call sequence | `sequenceDiagram` | -- |
| State machine | `stateDiagram-v2` | -- |
| Таймлайн проекта | `timeline` | -- |

### Шаг 2: Напиши Mermaid-код

Используй стилевые классы для типов нод:

```
classDef agent fill:#dbeafe,stroke:#3b82f6,stroke-width:2px,color:#1e3a5f
classDef database fill:#dcfce7,stroke:#22c55e,stroke-width:2px,color:#14532d
classDef api fill:#ffedd5,stroke:#f97316,stroke-width:2px,color:#7c2d12
classDef gate fill:#fef3c7,stroke:#f59e0b,stroke-width:2px,color:#78350f
classDef output fill:#f3e8ff,stroke:#a855f7,stroke-width:2px,color:#581c87
classDef process fill:#e0f2fe,stroke:#0284c7,stroke-width:2px,color:#0c4a6e
classDef error fill:#fee2e2,stroke:#ef4444,stroke-width:2px,color:#991b1b
classDef human fill:#fce7f3,stroke:#ec4899,stroke-width:2px,color:#831843
```

### Шаг 3: Заполни HTML-шаблон

Скопируй содержимое `docs/workflows/_template.html` и замени:

1. `{{DIAGRAM_TITLE}}` -- название на русском, понятное бизнесу (пример: «Как Олег делает отчёты»)
2. `{{DIAGRAM_TYPE}}` -- badge на русском (примеры: «Отчётный пайплайн», «Поток данных», «Общая схема», «Бизнес-процесс»)
3. `{{MERMAID_CODE}}` -- Mermaid-код из шага 2
4. `{{NODE_DETAILS_JSON}}` -- JSON-объект с деталями нод:

```json
{
  "node_id": {
    "name": "Понятное название для человека",
    "type": "Роль в процессе (не технический тип)",
    "description": "Подробное описание что делает и зачем — 2-3 предложения, понятные бизнес-пользователю",
    "notes": "Дополнительные пояснения, нюансы, что может пойти не так"
  }
}
```

**Не используй поля `tech`, `inputs`, `outputs`** — они слишком технические. Вместо этого объясняй входы/выходы естественным языком внутри `description`.

### Шаг 4: Сохрани и открой

Сохрани в `docs/workflows/<slug>.html` (kebab-case).
Открой: `open docs/workflows/<slug>.html`

## Стилевые классы нод

| Класс | Цвет | Использование |
|---|---|---|
| `:::agent` | Синий | AI-помощники, умные модули |
| `:::database` | Зелёный | Хранилища данных, базы |
| `:::api` | Оранжевый | Внешние сервисы (маркетплейсы, мессенджеры) |
| `:::gate` | Жёлтый | Проверки, условия, развилки |
| `:::output` | Фиолетовый | Результаты (отчёты, уведомления) |
| `:::process` | Голубой | Шаги обработки, преобразование данных |
| `:::error` | Красный | Ошибки, повторные попытки |
| `:::human` | Розовый | Действия человека, ручные проверки |

## Правила

1. **Всегда используй classDef** для цветовой кодировки
2. **Максимум 15-20 нод** на диаграмму. Если больше — разбей на несколько
3. **Используй subgraphs** для группировки фаз/модулей, заголовки с эмодзи
4. **Labels на стрелки** — описательные, на русском, с эмодзи для статусов (✅/❌)
5. **NODE_DETAILS_JSON** заполняй для всех ключевых нод, подробно и человечно
6. **Именование файлов**: `<subject>-<type>.html` (kebab-case)
7. **Язык**: весь текст на русском, для бизнес-пользователя
8. **Стрелки**: прямые линии (curve: linear уже настроен в шаблоне)
