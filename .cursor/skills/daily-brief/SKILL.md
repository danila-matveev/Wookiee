# Daily Brief — ежедневный отчёт

## Когда запускать

Когда пользователь просит:
- «дэйли», «дайли», «ежедневный отчёт», «как мы вчера/сегодня»
- `/daily-brief`, `/daily-brief YYYY-MM-DD`

**TARGET по умолчанию:** вчерашний день (`date.today() - 1`). Пользователь может указать конкретную дату.

## Шаги

### 1. Сбор данных

```bash
.venv/bin/python scripts/daily_brief/run.py <YYYY-MM-DD>
```

Результат: `data/daily_brief/<YYYY-MM-DD>.json` (15+ KB JSON).

Источники:
- **Supabase** через `shared/data_layer/` — финансы WB+OZON, `plan_article`, `content_analysis` (воронка WB), `adv_stats_daily` (воронка OZON по рекламе)
- **Google Sheets** через `gws` CLI — блогеры (BLOGGERS_SHEET_ID), ВК (VK_SHEET_ID), SMM (SMM_SHEET_ID — лист «Понедельный отчёт», транспонированная таблица)

Если gws auth истёк → соответствующая секция `marketing_context.{bloggers,vk,smm}.available` = false. Python не падает. Попросить пользователя `gws auth login`.

### 2. Написать narrative

Шаблон — 9 разделов + итоги. Следуй правилам из секции «Правила narrative».

### 3. Опубликовать в Notion

**Database:** «Аналитические отчёты» (id `30158a2b-d587-8091-bfc3-000b83c6b747`)

**Properties:**
- `Name`: `Ежедневный отчёт — DD.MM.YYYY`
- `Статус`: `Актуальный`
- `Источник`: `Claude Code`
- `Тип анализа`: `Ежедневный фин анализ`
- `Корректность`: `Да`
- `date:Период начала:start` и `date:Период конца:start`: target date
- `icon`: 📊

Парент: `{"type": "data_source_id", "data_source_id": "30158a2b-d587-8091-bfc3-000b83c6b747"}`

Инструмент: `mcp__claude_ai_Notion__notion-create-pages`.

## Правила narrative

### ❌ Чего НЕ писать

**Обучающих/концептуальных пояснений**, которые повторяются из отчёта в отчёт. Эти знания у пользователя уже есть, они забивают страницу:

- ❌ «Ключевые различения. Выручка — это деньги по выкупленным заказам. Сумма заказов — это деньги по заказам, которые ещё доедут…»
- ❌ «Логика: заказы сегодня → выручка через 3-21 день → маржа следом…»
- ❌ «По WB нет сквозной атрибуции заказов с конкретной рекламы. Смотрим на сумму заказов и маржу с задержкой 3-21 день…»
- ❌ Общие фразы «Что важно видеть» без привязки к конкретным цифрам дня.

Концепции упоминаются максимум **одной короткой ссылкой** в хедере раздела, а не повторяются callout-ом каждый день.

### ✅ Что писать обязательно

**Модельная атрибуция** при любой агрегированной оценке «растём/падаем/отстаём»:

- Если маржа выросла — указать **топ-3 моделей-драйверов** с суммой вклада в ₽ и % маржинальностью.
- Если темп заказов замедлился — указать **какие модели тянут вниз** (declining WB+OZON, с ₽-вкладом в дельту).
- Если разрыв плана по марже — **таблица драйверов-моделей** с маржинальностью, выручкой, потенциалом подъёма в ₽ за оставшиеся дни месяца.
- Если вырос день — **по каким 2-3 моделям WB и 2-3 моделям OZON**.

Источники в JSON:
- `model_radar.wb.{growing,declining,critical_marginality}`
- `model_radar.ozon.{growing,declining,critical_marginality}`
- Каждый элемент содержит: `model`, `margin_today`, `margin_yesterday`, `delta_abs`, `delta_pct`, `marginality_pct`, `revenue_today`

**Никогда не галлюцинировать названия моделей.** Использовать только те, что пришли из JSON.

**Экономический эффект в ₽** для каждой рекомендации: «Подъём маржинальности Wendy WB с X% до 20% = +₽/день × дней = ₽ к марже месяца».

### Структура отчёта

Все разделы — на `## II-N заголовках`, подразделы — на `### Подраздел` (угловые markdown-заголовки, не callout-ы вместо заголовков).

| # | Раздел | Содержание |
|---|---|---|
| I | Итог дня | 3 callout-а: main result, main driver (модели), main risk/flag |
| II | Финансы — динамика за 5 дней | Таблица 5-дневная с подсветкой red_bg/green_bg/yellow_bg |
| III | Прогноз по динамике заказов | Таблица 5д vs 5д, 3д vs 3д + callout **с драйверами-моделями** при отклонении |
| IV | План-факт по марже | Таблица метрик + **### Драйверы разрыва — по моделям** + fokus-callout |
| V | Воронка WB (5 дней) | Таблица CRO/конверсии с акцентом на 21.04 |
| V.1 | Воронка OZON (реклама) | Таблица CTR/CR click→order |
| VI | Маркетинг | `### Реклама` (канал vs прошлый день), `### Блогеры (7 дней)`, `### Связка блогер→модель→маржа`, `### Запланировано на 3 дня`, `### ВК/Яндекс`, `### SMM (staleness)` |
| VII | Модели — движение за день | `### WB растут`, `### WB падают`, `### WB критическая маржинальность`, `### OZON растут`, `### OZON падают`, `### OZON критическая маржинальность` |
| VIII | Флаги | callout-ы из JSON `flags[]` с русскими severity_ru |
| IX | Что делать | 5-7 пронумерованных callout-ов с экономическим эффектом в ₽. Каждая рекомендация ссылается на **конкретные модели** |
| — | Итоги | 5-6 коротких callout-ов с цветовой индикацией: финансы, темп, воронка, блогер недели, SMM, критичное |

### Стилистика Notion

Используй Notion-enhanced markdown. Доступные синтаксисы:

```markdown
<callout icon="📊" color="blue_bg">
	**Заголовок.** Текст.
</callout>

<table fit-page-width="true" header-row="true" header-column="true">
<tr color="blue_bg">
<td>Col1</td><td>Col2</td>
</tr>
<tr color="green_bg">
<td>**Highlighted row**</td><td color="red_bg">**cell accent**</td>
</tr>
</table>
```

**Цвета:** `blue_bg` (нейтрально), `green_bg` (хорошо), `yellow_bg` (внимание), `red_bg` (плохо), `purple_bg` (forecast/план), `gray_bg` (дисклеймер/итого).

**Правила подсветки:**
- Header row в таблицах — всегда `blue_bg`
- Целевая дата (target) в 5-дневной таблице — `yellow_bg` (для акцента, не green/red — это для аномалий)
- Провальный день (если есть в окне) — `red_bg`
- Лучший день в окне — `green_bg`
- Отдельные ячейки с аномалиями — `<td color="red_bg">` или `<td color="green_bg">`

## Правила метрик (важно)

- **Доля рекламы** — от `orders_rub` (суммы заказов), **НЕ от выручки**. Реклама влияет на заказы сегодня, выручка = выкупы со сдвигом 3-21 день.
- **План по марже** — `plan_article` → индикатор «Маржинальная прибыль, руб». Другие индикаторы в таблице: «Сумма заказов, руб» → `plan.orders_rub`, «Сумма продаж, руб» → `plan.revenue`, «Кол-во заказов, шт» → `plan.orders_qty`.
- **Выкуп %** за target-дату — **лаговый** (3-21 день задержка). Не использовать как причину изменения маржинальности в дневном отчёте — только как информационный с пометкой «лаг 3-21 дн.».
- **СПП %** при объединении каналов — взвешенный: `sum(spp_amount) / sum(revenue_before_spp) * 100`.
- **SMM staleness** — когда `marketing_context.smm.staleness_days > 3`, писать дату закрытой недели в заголовке раздела и один короткий дисклеймер про задержку.

## Правила атрибуции (важно)

- **OZON не имеет органической воронки** по просмотрам карточки (в БД нет аналога `content_analysis` для OZON). Показывать только рекламную воронку — это фактическое ограничение данных, не забывать раз в заголовке V.1.
- **Блогеры WB без сквозной атрибуции** — не приписывать заказы конкретной рекламе. Смотрим на `orders_rub` бренда + маржу с задержкой выкупа.
- **Пустые метрики в таблице блогеров** (0 просмотров, 0 кликов, 0 заказов при ненулевом бюджете) — это пробел в разметке, **явно выделить в отчёте** как проблему «нельзя оценить ROI этих размещений».

## Инфраструктура

- **Python:** 3.12 (venv в `.venv/`)
- **Зависимости:** минимум `psycopg2-binary` + `python-dotenv`. Всё остальное — через `shared/data_layer/` (чтение pbi_wb_wookiee, PG 18.1) и `gws` CLI (системный бинарь).
- **Git ветка:** код на `feature/localization-redesign`, не на main.
- **Секреты:** `.env` (не в git).

## Failure modes

- **gws auth истёк** → SMM/блогеры пустые → отчёт продолжается, просто эти секции помечаются как unavailable.
- **БД недоступна** → `run.py` упадёт на подключении. Проверить VPN/сервер подрядчика.
- **plan_article пуст для target-месяца** → `plan_month` будет с нулями. Пользователь должен внести план в таблицу (Google Sheets → sync-sheets-to-supabase).
- **Модели-новинки не в model_mapping** → попадут в категорию «Other» в model_radar. Это нормально, не чинить «на лету».

## После публикации

1. Дать пользователю URL отчёта в Notion.
2. Сохранить JSON в `data/daily_brief/<date>.json` (делает `run.py`).
3. Если были правки кода — коммит на `feature/localization-redesign`, push в origin.
