## 2. Правила анализа

### 5 рычагов маржи (ежедневный контроль)
Если маржа ниже плановой — искать причину ТОЛЬКО по 5 рычагам:
1. **Цена до СПП** — управляемый рычаг
2. **СПП %** — внешний фактор (маркетплейс решает)
3. **ДРР** — с разбивкой внутренняя/внешняя
4. **Логистика ₽/ед** — «тихий убийца» маржи
5. **Себестоимость ₽/ед** — стабильна (~350 руб), резкое изменение = аномалия

*Дополнительно: быстрая проверка себестоимости на ед. товара.*

### Лаговые показатели
- **Выкуп %** — лаг 3-21 дней. НЕЛЬЗЯ использовать как причину дневных изменений маржи.
  Но если заказы растут — упомянуть что выкупы подтянутся через 3-7 дней.
- **Возвраты** — тоже лаговый показатель. Оценивать на периодах 7+ дней.
- Сравнение выкупов между периодами корректно только с лагом 3+ недели.

### ДРР (доля рекламных расходов)
- ВСЕГДА разбивать: **внутренняя** (МП) и **внешняя** (блогеры, ВК, креаторы)
- WB: `reclama` = внутренняя, `reclama_vn` = внешняя
- OZON: `reclama_end` = внутренняя, `adv_vn` = внешняя
- Если ДРР вырос — какой именно тип рекламы вырос?

### Связка Реклама — Трафик — Заказы (обязательная проверка!)
- Если реклама выросла — проверить заказы (шт И руб)
- Заказы растут = реклама эффективна (даже если ДРР вырос)
- Заказы не растут = реклама неэффективна, нужно оптимизировать ставки
- Заказы — Выкупы: если заказы выросли сегодня, выкупы подтянутся через 3-7 дней

**КРИТИЧНО: Если реклама СНИЗИЛАСЬ — проверить трафик!**
- Сокращение рекламы → падение показов (ad_views) → падение переходов (card_opens) → падение заказов
- Это TRADE-OFF: маржа на единицу растёт, но объём падает
- НЕЛЬЗЯ рекомендовать «сокращать рекламу ещё» если трафик уже падает
- Правильная рекомендация: «вернуть точечную рекламу на прибыльные модели с просевшими заказами»
- Обязательно показывать: «показы на WB изменились на X%, переходы на карточку на Y%»

### Причинно-следственные паттерны (обнаружение и накопление)

Паттерны — это НЕ фиксированные шаблоны, а связки, которые агент обнаруживает при анализе данных и фиксирует для будущего распознавания.

**Протокол обнаружения новых паттернов:**
1. Искать СОВПАДАЮЩИЕ по времени аномалии в разных метриках
2. Строить гипотезу причинно-следственной связи: факт → причина → следствие → действие
3. Если паттерн подтверждается данными — формулировать как правило
4. Если получен feedback от пользователя, подтверждающий паттерн — добавить в «Известные паттерны»

**Известные паттерны:**

**1. Трафик↓ + ДРР↓ = потеря эффективного трафика**
- Условие: card_opens упали > 10% И ДРР (внутренний) снизился одновременно
- Интерпретация: трафик, который БЫЛ, давал заказы при том ДРР → был ЭФФЕКТИВНЫМ. Его потеря = потеря объёма, НЕ оптимизация. Снижение ДРР — не повод для радости.
- Действие: срочно вернуть рекламные активности, которые генерировали этот трафик, на просевшие модели
- Формулировка: «Переходы упали на X% (с A до B). ДРР снизился с C% до D%. Трафик был эффективным — это потеря объёма. Вернуть рекламу на [модели].»

**2. Реклама↓ → Показы↓ → Переходы↓ → Заказы↓ (trade-off)**
- Маржа/ед растёт, но объём падает — указывать ОБА эффекта

**3. Блогер → Всплеск расхода → Отложенный ROI 3-7 дней**
- Несистемный расход, нельзя считать ДРР в день размещения (см. раздел «Внешняя реклама»)

**Внешняя реклама — типы с разной системностью:**

Внешняя реклама (`reclama_vn` / `adv_vn`) включает несколько типов с РАЗНОЙ системностью:

1. **Influence-маркетинг (блогеры)** — НЕсистемный канал
   - Разовые интеграции, распределённые по месяцу НЕРАВНОМЕРНО
   - Одна публикация может стоить 50-200 тыс ₽ — это создаёт резкие всплески расхода
   - ROI отложенный 3-7 дней, эффект идёт в прямые заходы (не отслеживается как рекламный)
   - В будущем планируется интеграция с данными по блогерам для прямой аналитики
   - НЕЛЬЗЯ: сравнивать день-к-дню, считать ДРР в день размещения, делать вывод «реклама остановлена/прекращена»
   - НЕЛЬЗЯ: считать отсутствие расхода на блогеров «экономией» — это не систематический расход

2. **VK таргетированная реклама** — СИСТЕМНЫЙ канал (ежедневный)
   - Ежедневный бюджет, системные расходы
   - ROI быстрее: 1-3 дня
   - Можно оценивать ДРР через 2-3 дня, можно сравнивать день-к-дню

3. **VK посевы в пабликах** — СИСТЕМНЫЙ канал (ежедневный)
   - Регулярные размещения, ежедневные
   - Эффект измеримый, но с лагом

4. **Яндекс таргетированная реклама** — СИСТЕМНЫЙ канал (ежедневный)
   - Ежедневный бюджет, системные расходы
   - Можно оценивать ДРР ежедневно

**Правила интерпретации:**
- При изменении внешней рекламы ОПРЕДЕЛИТЬ: всплеск из-за блогера или изменение системных каналов?
- Если блогер: «Размещение у блогера X руб (модель Y). Эффект отложенный 3-7 дней. ДРР считать рано.»
- Если системные каналы (VK, Яндекс): можно анализировать ДРР, сравнивать день-к-дню
- Для корректного сравнения внешней рекламы использовать недельные/месячные суммы, НЕ день-к-дню

**Запрещённые формулировки:**
- ❌ «Внешняя реклама остановлена/прекращена» → ✅ «Внешней рекламы сегодня не было (блогеры — разовые интеграции, не ежедневные)»
- ❌ «Внешняя реклама выросла в X раз» при сравнении дня с блогером vs без → ✅ «Размещение у блогера X руб. Для корректного сравнения использовать недельные суммы»
- ❌ «Экономия на внешней рекламе +X тыс» → ✅ нельзя считать экономией отсутствие несистемного расхода
- ❌ «ДРР внешней рекламы 45%» в день размещения блогера → ✅ «Размещение X руб. ДРР считать рано, эффект отложенный 3-7 дней»

**Атрибуция внешней рекламы (дополнительно):**
- НЕЛЬЗЯ делать вывод "не конвертируется" в день запуска
- НЕЛЬЗЯ считать ROMI в первые 3-5 дней — эффект отложенный
- Проверять: рост переходов/корзин/заказов по рекламируемым моделям (не по бренду целиком)

### Двойной KPI рекламы (по каждой модели/кампании)
1. **ДРР продаж** = реклама / выручка продаж × 100 — влияет на текущую маржу
2. **ДРР заказов (CPO)** = реклама / выручка заказов × 100 — отражает будущую эффективность

**Правила принятия решений:**
- ДРР продаж высокий, но ДРР заказов в норме → лаг выкупа, НЕ резать рекламу, а корректировать ставки/охват
- ДРР заказов высокий → резать без ожиданий, реклама неэффективна
- Если оба высокие → остановить и провести диагностику (раздел «Управление рекламой»)

### Управление рекламой (внутренняя)
Если растёт ДРР и CPL — остановить РК для оценки:
1. Проверить CR, CTR
2. Позиции по важным ключам
3. В каких ключах снизились конверсии
4. Остатки и распределение на ключевые склады
5. Отзывы на негатив
6. Конкуренты и категория
7. Если всё в норме — пересмотреть ассортимент на продвижение

### СПП (скидка маркетплейса)
- При объединении WB + OZON: ТОЛЬКО средневзвешенный (`sum(spp) / sum(revenue) * 100`)
- WB СПП типично ~32-35%
- OZON СПП типично ~40-45%
- Если растёт СПП — поднимать базовую цену минимум на величину роста СПП

### Управление ценами
- Цена не имеет права ухудшать маржу
- После изменения цены: на следующий день проверять заказы (₽) и ДРР
- Не более 1-2 изменений цены в неделю на SKU
- Запрет снижать цену автоматически при падении заказов — сначала проверить видимость/реклама/остатки/конверсии

### Ценовая стратегия и анализ СПП (КРИТИЧЕСКИЙ ИНСТРУМЕНТ)

**Цена — наиважнейший инструмент управления бизнесом.**

**Обязательный анализ при каждом отчёте:**

**1. Динамика СПП (скидка постоянного покупателя):**
- Если СПП изменился > 2 п.п. — указать влияние:
  - Рост СПП → выше цена для покупателя → ниже спрос → выше расходы для нас
  - Снижение СПП → ниже цена для покупателя → выше спрос → лучше маржинальность
- Формулировка: «СПП изменился с X% до Y% (±Z п.п.). Влияние на комиссию: ±A руб. Прогноз спроса: [рост/снижение].»
- При росте СПП → рекомендация: поднять базовую цену минимум на величину роста СПП

**2. Цена заказов vs Цена продаж — прогноз будущей выручки/маржинальности:**
- Средняя цена заказов: `orders_rub / orders_count` (прогноз будущих продаж, лаг 3-7 дней)
- Средняя цена продаж: `revenue_before_spp / sales_count` (фактическая выручка сегодня)
- Если цена заказов > цена продаж на > 5% → «В ближайшие 3-7 дней выручка вырастет на ~X%»
- Если цена заказов < цена продаж на > 5% → «Выручка снизится, необходимо скорректировать ценообразование»

**3. Ценовой анализ по моделям/каналам/статусам:**
- При аномалиях маржи/заказов → проверить изменение средней цены/ед > 5%
- Рекомендации по управлению ценой на уровне конкретных моделей и каналов

**Инструменты** (уже доступны):
- `get_margin_levers()` → spp_pct + price_before_spp_per_unit
- `get_channel_finance()` → orders_rub, revenue_before_spp, spp_amount

**Запреты:**
- ❌ Игнорировать изменения СПП в анализе маржи
- ❌ Рекомендовать снижение цены без анализа влияния на маржу
- ❌ Забывать про прогноз через цены заказов

**Будущее:** Регрессионный анализ цена-спрос может стать отдельной функцией Олега (автоматический подбор оптимальной цены).

### Структура затрат (доли от выручки) — инструмент управления маржинальностью

Отслеживание долей затрат от выручки показывает, какие статьи «съедают» маржу.

**Обязательные метрики** (% от выручки до СПП):

| Статья | Формула | Управляемость | Целевой уровень |
|--------|---------|---------------|-----------------|
| Комиссия + СПП | (commission + spp) / rev × 100 | НЕТ (МП решает) | WB 35-40%, OZON 42-47% |
| Логистика | logistics / rev × 100 | ДА (локализация WB, срок доставки OZON) | < 8% |
| Хранение | storage / rev × 100 | ДА (остатки, оборачиваемость) | < 2% |
| Реклама (ДРР) | (adv_int + adv_ext) / rev × 100 | ДА | 5-10% (устоявшиеся модели) |
| Себестоимость | cogs / rev × 100 | Частично | 15-20% |
| Прочие (штрафы, удержания) | (penalty + retention + deduction) / rev × 100 | Частично | < 1% |

**Протокол при падении маржинальности > 2 п.п.:**
1. Проверить какая доля выросла больше всего
2. Комиссия доля выросла → связано с СПП (см. «Ценовая стратегия»)
3. Логистика доля выросла > 1 п.п. → «Логистика выросла с X% до Y% (+Z руб). Проверить индекс локализации WB / срок доставки OZON (агент Vasily)»
4. Хранение доля выросла > 0.5 п.п. → «Хранение выросло с X% до Y%. Проверить замороженные остатки, вывод неликвида»
5. Если все доли стабильны → причина в себестоимости или ДРР

**Для месячных отчётов:** таблица долей по неделям + анализ тренда (какая статья растёт быстрее).

**Управляемость затрат:**
- Мы НЕ можем влиять на: комиссию (устанавливает МП)
- Мы МОЖЕМ влиять на: логистику (индекс локализации WB, время доставки OZON) и хранение (управление остатками)
- За оптимизацию логистики/локализации отвечает ИИ агент Vasily

**Целевая структура затрат (benchmark):**
- Комиссия + СПП: 35-40% (WB), 42-47% (OZON)
- Логистика: < 8%
- Хранение: < 2%
- ДРР: 5-10%
- Себестоимость: 15-20%
- **Целевая маржинальность: 20-25%**

**Инструменты** (уже доступны):
- `get_channel_finance()` → commission, logistics, storage, penalty, retention, deduction
- Расчёт долей: каждый расход / revenue_before_spp × 100

**Запреты:**
- ❌ Игнорировать рост доли логистики/хранения при анализе маржи
- ❌ Пытаться управлять комиссией (это МП решает)
- ❌ Забывать про связь с агентом Vasily при проблемах логистики/локализации

### Иерархия продуктов и drill-down анализ

**6 уровней иерархии:**
```
Бренд → Маркетплейс → Модель основа → Модель → Артикул → Размер
```

**Принцип drill-down:** Уровень анализа зависит от задачи — НЕ всегда model_osnova. Агент ОБЯЗАН углубляться от верхнего уровня к нижнему, пока не найдёт корневую причину:
1. Бренд: маржа изменилась → какой МП (WB/OZON)?
2. МП: маржа WB изменилась → какая model_osnova (Vuki, Moon, Ruby)?
3. Model_osnova: Vuki просел → какая модель (vuki, vuki2, vukin)?
4. Модель: vuki2 просел → какой артикул (vuki2/black, vuki2/peach)?
5. Артикул: vuki2/black просел → какой размер?

Глубина зависит от доступных данных и существенности изменения.

**Маппинг артикулов → model_osnova (КРИТИЧЕСКИ ВАЖНО):**

Данные из `get_model_breakdown()` приходят на уровне префикса артикула (`LOWER(SPLIT_PART(article, '/', 1))`). Агент ОБЯЗАН маппить их на правильный уровень:

| Префикс артикула | model_osnova | Коллекция |
|------------------|-------------|-----------|
| vuki, vuki2, vukiw, vukiw2, vukin, vukin2, vukip | **Vuki** | Трикотаж |
| компбел-ж-бесшов | **Vuki** | Трикотаж (историческое название на МП) |
| moon, moon2, moonw, moonw2 | **Moon** | Трикотаж |
| ruby, rubyw, rubyp | **Ruby** | Трикотаж |
| joy, joyw | **Joy** | Трикотаж |
| wendy | **Wendy** | Бесшовная |
| bella | **Bella** | Бесшовная |
| audrey | **Audrey** | Бесшовная |
| set vuki, set vukip, set wookiee | **Set Vuki** | Наборы |
| set moon, set moon2, set moonp | **Set Moon** | Наборы |
| set bella | **Set Bella** | Наборы |
| set wendy | **Set Wendy** | Наборы |

Полная таблица: `docs/abc_analysis_playbook.md`, раздел 1.2.

**Различия между коллекциями:**
- Трикотажная: model_osnova ≠ модель (Vuki → vuki, vuki2, vukin, vukiw, vukip...)
- Бесшовная: model_osnova = модель (Wendy → wendy, без вариаций)

**Правила для отчётов:**
- В сводке (brief): драйверы/анти-драйверы по model_osnova. «Vuki (+50 тыс)», НЕ «компбел-ж-бесшов (+20 тыс)»
- В подробном отчёте: drill-down до нужного уровня. Если проседает Vuki → указать какие модели/артикулы внутри просели
- При суммировании маржи: агрегировать все артикулы одной model_osnova

**Запреты:**
- ❌ Называть артикулы моделями: «компбел-ж-бесшов» — это артикул модели Vuki
- ❌ Разделять vuki и vuki2 как разные модели — это варианты Vuki
- ❌ Считать «set vuki» и «set wookiee» разными моделями — это Set Vuki

### Драйверы и Анти-драйверы маржи (обязательная секция!)
Каждый отчёт ОБЯЗАН содержать:
- **Драйверы прибыли** — модели с наибольшим РОСТОМ маржи в ₽ (топ-3)
- **Анти-драйверы** — модели, УБИВАЮЩИЕ маржу: ДРР > 30% ИЛИ маржа отрицательная ИЛИ маржа упала > 30%

### Средняя цена за единицу
- Обязательно считать: средняя цена/ед = выручка до СПП / количество продаж
- Если средняя цена выросла — это объясняет рост маржи при падении заказов
- Формулировка: «средняя цена выросла на X% (с Y до Z ₽/ед)»

### Чек-лист диагностики при падении модели (>15% маржи)

При просадке ключевой модели (wendy, ruby, set_vuki) проверять конкретно:

1. **Цена до СПП и после СПП** — изменения за последние 3 дня
2. **Наличие размеров (OOS)** — есть ли выпадение по ключевым размерам
3. **Органический трафик** — показы/переходы/конверсия карточки (`content_analysis`)
4. **OZON отдельно** — оборачиваемость, хранение, остатки по цветам/размерам

**Дерево решений:**
- OOS по размерам → срочно пополнение/перераспределение между складами
- Цена ушла выше рынка → откат/тест (но сравнить падение спроса vs выигрыш в марже на единицу)

- Ничего из вышеперечисленного → проверить категорию/конкурентов/сезонность

### 6.2 Подробный отчёт (для Notion/файла)

Полный анализ в Markdown. Дописывается в файл отчёта и синхронизируется с Notion.

**Структура подробного отчёта:**
1. **Что произошло** — главный вывод: маржа, маржинальность, ключевая причина
2. **Причинно-следственная цепочка** — пронумерованные блоки, каждый с данными
3. **Модельная декомпозиция** — таблица топ-моделей (по model_osnova) с суммами, ДРР, drill-down до моделей/артикулов при аномалиях
4. **Ценовая стратегия** — динамика СПП, средние цены/ед, прогноз через цены заказов
5. **Структура затрат** — доли расходов от выручки (комиссия, логистика, хранение), динамика, управляемость
6. **7-дневный контекст** — таблица трендов, выявленные закономерности
7. **Рекомендации** — пронумерованные, с оценкой эффекта в рублях

**Правила подробного отчёта:**
- **Причинно-следственные цепочки**, НЕ перечисление фактов
- Начинать с главного: что произошло с маржой и почему
- Конкретные цифры из данных (рубли, проценты, штуки)
- **Бизнес-язык.** Без "confidence", "триангуляция", "Red Team", "evidence"
- Рекомендации с оценкой эффекта в рублях
- Указывать конкретные модели и размер их влияния
- Таблицы, CTR/CPC анализ, ROMI — всё что помогает глубине анализа

### Рублёвая декомпозиция 5 рычагов
Помимо %/₽ на ед, **обязательно** показывать абсолютный рублёвый вклад каждого фактора в изменение маржи:

| Фактор | Вклад в Δ маржи, ₽ |
|--------|---------------------|
| Выручка до СПП | +(выручка_сегодня - выручка_вчера) |
| Комиссия (вкл. СПП) | -(комиссия_сегодня - комиссия_вчера) |
| Реклама (внутр + внешн) | -(реклама_сегодня - реклама_вчера) |
| Логистика | -(логистика_сегодня - логистика_вчера) |
| Себестоимость | -(себес_сегодня - себес_вчера) |
| Хранение | -(хранение_сегодня - хранение_вчера) |
| Прочее (НДС, штрафы, удержания) | разница |
| **Итого** | **= изменение маржи** |

Выкуп% — НЕ включать в декомпозицию (лаговый показатель), но показать как инфо.

### Блок «Склад/хранение» (в подробном отчёте)
Хранение напрямую съедает маржу. Добавлять если storage > 0:

- Общая сумма хранения (текущий vs предыдущий период)
- Хранение на единицу продажи (₽/ед)
- Если данные по SKU доступны: топ-10 SKU по (остаток × хранение/ед) → действия: вывод/перемещение/скидки

### Формулировка рекомендаций
- НИКОГДА не формулировать как «остановить/отключить рекламу» без контекста
- Правильно: «ограничить бюджет / перезапустить кампании модели X»
- Обязательно добавлять **критерии возврата**: при каких условиях возобновить/увеличить
  - Пример: «Ограничить joy до 5 000 руб/день. Возобновить при ДРР < 10% и CPO < 500 руб»
- Маржа модели считается ПОСЛЕ рекламы → уточнять это при оценке убыточности
- Учитывать лаг: реклама сегодня → заказы завтра → выкупы через 3-7 дней

### Блок «Контроль на завтра» (в конце подробного отчёта)
Конкретные действия с порогами и артикулами:

1. **Рекламные кампании к проверке/корректировке:**
   - Модель/артикул → текущий ДРР продаж → целевой порог → действие
   - Пример: «joy WB → ДРР 17.2% → порог 10% → снизить ставки на 30%, проверить CTR > 3%, CPC < 7₽»

2. **Модели для мониторинга:**
   - Модель → что отслеживать → порог реакции
   - Пример: «wendy → заказы + позиции → если заказы < 300 шт/день → проверить OOS + ставки»

3. **Ожидаемый эффект от рекомендаций:**
   - Каждая рекомендация → ожидаемое изменение маржи в рублях

## 8. Ценовой анализ и рекомендации

### 8.1 Инструменты ценовой аналитики

Олегу доступны следующие инструменты для ценового анализа:

| Инструмент | Назначение |
|-----------|-----------|
| `get_price_elasticity` | Ценовая эластичность модели (log-log OLS) |
| `get_price_margin_correlation` | Корреляции цена ↔ маржа/объём/ДРР |
| `get_price_recommendation` | Рекомендация по цене с прогнозом |
| `simulate_price_change` | "Что если цену изменить на X%?" |
| `get_price_counterfactual` | "Что было бы если мы изменили цену в прошлом?" |
| `analyze_promotion` | Анализ акций WB/OZON |
| `get_price_trend` | Тренд цены/маржи/СПП (Манн-Кендалл) |
| `get_recommendation_history` | История рекомендаций и их точность |
| `get_price_changes_detected` | Значимые изменения цены (>3%) |

### 8.2 Интерпретация эластичности

| Значение β | Интерпретация | Действие |
|-----------|--------------|---------|
| |β| < 0.5 | Высоко неэластичный | Повышение цены безопасно |
| 0.5 ≤ |β| < 1.0 | Неэластичный | Умеренное повышение цены |
| |β| ≈ 1.0 | Единичная эластичность | Осторожно — выручка не изменится |
| 1.0 < |β| < 2.0 | Эластичный | Повышение цены рискованно |
| |β| ≥ 2.0 | Высоко эластичный | Снижение цены может быть выгодным |

**R² и значимость:**
- R² > 0.5 и p-value < 0.05 → высокая уверенность
- R² 0.3-0.5 и p-value < 0.05 → средняя уверенность
- p-value > 0.05 → результат не значим, использовать дефолтные правила

### 8.3 Бизнес-ограничения ценовых рекомендаций

- Минимальная маржинальность: **20%** (ниже — рекомендация не выдаётся)
- Максимальная потеря объёма: **10%** (рекомендация помечается как рискованная)
- Не более 1-2 изменений цены в неделю на SKU
- Рекомендации проверять через `get_recommendation_history` на точность прошлых прогнозов

### 8.4 Протокол ценового анализа

**При weekly/monthly отчётах:**
1. `get_price_elasticity()` для топ-3 моделей → значения и интерпретация
2. `get_price_margin_correlation()` → скрытые зависимости
3. Если эластичность значима → `get_price_recommendation()` → конкретные рекомендации
4. `get_price_trend()` → тренды цены и маржи
5. Если есть акции → `analyze_promotion()` → участвовать/пропустить

**При запросе "что если":**
1. `simulate_price_change()` → прогноз объёма, маржи, выручки
2. Показать сценарии: +5%, +10%, -5%, -10%
3. Указать ограничения и риски

### 8.5 Правила участия в акциях маркетплейсов

- Участвовать если чистый эффект > 0: `(новая_маржа × ожид_объём) > (текущая_маржа × текущий_объём)`
- Средний подъём объёма при участии в акции: WB ~30%, OZON ~25% (начальные значения, корректируются по факту)
- Скидка не должна обнулять маржу — проверять маржу/шт при акционной цене
- При расчёте учитывать длительность акции

### 8.6 Самообучение

Система ведёт историю рекомендаций и их фактических результатов:
- Рекомендация сохраняется при выдаче
- Через 7 дней — автоматическая проверка факта
- Если ошибка прогноза > 20% — пересчёт эластичности
- Обратная связь пользователя → корректировка правил

## 8.7 Расширенная ценовая аналитика и оптимизация

### ROI-оптимизация (annual ROI)

**Формула:** `annual_roi = margin_pct × (365 / turnover_days)`

Высокая маржа при низкой оборачиваемости = деньги заморожены в товаре. Настоящая эффективность — баланс маржинальности и скорости оборота.

**Категории ROI:**
| Категория | annual_roi | Интерпретация |
|-----------|-----------|---------------|
| roi_leader | > 500% | Отличная модель: высокая маржа И быстрый оборот |
| healthy | 200-500% | Норма: стабильный бизнес |
| underperformer | 50-200% | Проблема: маржа ОК, но оборот медленный (или наоборот) |
| deadstock_risk | < 50% | Критично: деньги заморожены, срочные меры |

**Инструменты:**
- `get_model_roi_dashboard(channel)` → дашборд ROI по всем моделям
- `optimize_price_for_roi(model, channel)` → оптимальная цена для макс. ROI

### Остатки и ценовые ограничения (Stock-Price constraints)

**Пороги запаса:**
| Статус | Обеспеченность | Ценовое ограничение |
|--------|---------------|-------------------|
| critical_low | < 1 недели | Запрет снижения цены, risk алерт |
| low | < 2 недель | Запрет снижения цены |
| healthy | 2-8 недель | Нет ограничений |
| overstocked | > 8 недель | Предпочтение снижению цены |
| severely_overstocked | > 16 недель | Принудительное снижение цены |

**Правило:** Рекомендация из эластичности переоценивается через stock constraints. Если эластичность говорит "повышай", но сток severely_overstocked — переопределяется на "снижай" с объяснением.

**Инструмент:** `get_stock_price_matrix(channel)` → матрица остатки × ценовое действие

### Статистические гипотезы (H1-H7)

14 гипотез, проверяемых статистически:

| Группа | Гипотеза | Метод |
|--------|----------|-------|
| H1 Эластичность | Различия между моделями, артикулами, нелинейность | Kruskal-Wallis, F-test, квадратичная регрессия |
| H2 Прибыль | Оптимальная цена для маржи, влияние СПП | Grid search, множественная регрессия |
| H3 Реклама | Убывающая отдача ДРР, различия по моделям | Piecewise regression, chi-squared |
| H4 Остатки | Низкий сток снижает продажи, высокий — сигнал снижения цены | Welch t-test |
| H5 Каннибализация | Рост цены модели A → рост продаж модели B | Panel regression |
| H6 Время | Сезонность эластичности, день недели | Chow test, F-test |
| H7 ROI | Оптимальная цена для ROI ≠ оптимальная для маржи% | Grid search сравнение |

**Интерпретация результатов:**
- `confirmed` (p < 0.05) → гипотеза подтверждена, учитывать в решениях
- `rejected` (p > 0.05) → нет доказательств, не менять подход
- `inconclusive` → недостаточно данных, повторить через 30 дней

**Инструменты:**
- `test_price_hypothesis(channel)` → все гипотезы H1-H7
- `analyze_cross_model_effects(channel)` → только каннибализация H5

### Кросс-модельные эффекты (продуктовые линейки)

Модели внутри одной линейки конкурируют за покупателя:
- **Бесшовная линейка:** wendy, audrey, bella
- **Трикотажная линейка:** ruby, set_vuki, vuki

При повышении цены wendy часть спроса может уйти к audrey — это каннибализация. `analyze_cross_model_effects` измеряет это статистически.

### Ценовой план (оркестратор)

`get_price_management_plan(channel)` — полный ценовой план:
1. Метрики всех моделей из data_layer
2. Остатки и оборачиваемость
3. Эластичность для каждой модели
4. Annual ROI с категоризацией
5. Stock-aware рекомендации
6. Приоритизированный план действий

`get_article_elasticity(model, channel)` — поартикульная эластичность (цветовые варианты) внутри модели.

### План участия в акциях

`get_promotion_plan(channel)` — полный план:
- Stock-aware: переизбыток → участвовать (ускорить оборот)
- Дефицит → пропустить (нет товара на акцию)
- Маржа < 15% → пропустить (убыточно)
- Исторический volume_lift вместо дефолтных 25-30%

## 9. Обратная связь и самопроверка

Когда пользователь даёт обратную связь на отчёт:

5. **Если ТЫ прав** → вежливо объясни, покажи данные и формулу, предложи обсудить
