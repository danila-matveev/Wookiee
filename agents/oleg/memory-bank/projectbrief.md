# Олег — Проект

## Миссия
**Олег** — система финансовой аналитики Wookiee, состоящая из двух независимых компонентов:
1. **Agent Runner (`agent_runner.py`)**: Автономный процесс, который по расписанию генерирует отчёты и мониторит данные. Не зависит от Telegram.
2. **Telegram Bot (`main.py`)**: Интерфейс для пользователей (меню, аутентификация, интерактивные запросы) и доставка готовых отчётов из очереди (delivery queue).

**Главная задача:** Обеспечить 100% надёжность доставки отчётов и помочь управленческой команде принимать решения на основе данных.

## Бизнес-контекст
Wookiee — прибыльная компания с рентабельностью от 15% по чистой прибыли, работающая на маркетплейсах WB (~85% выручки) и OZON (~15%).

**Три ключевых показателя:**
1. Маржинальная прибыль (₽) и маржа (%) — ядро ежедневного управления
2. Годовая доходность (ROI) — возврат на вложенный капитал
3. Косвенные расходы (₽ и % от выручки) — рычаг общей эффективности

**Цели (январь 2026):**
- Минимум: 5 млн ₽ маржи, 20% маржинальность
- Средний: 6.5 млн ₽, 23%
- Высокий: 8 млн ₽, 25%
- Супер: 10 млн ₽, 30%

## Архитектура
- **Тип:** Распределённая система (Producer-Consumer)
- **Agent Runner (Producer):** Генерирует отчёты и складывает их в SQLite `delivery_queue`.
- **Telegram Bot (Consumer):** Читает очередь и доставляет HTML-сообщения пользователям.
- **Инструментарий:** ReAct-агент с tool-use для аналитики.
- **Интерфейс:** Telegram (aiogram 3.15).
- **LLM:** OpenRouter (kimi-k2.5 — основная аналитика, z.ai glm-4.5-flash — классификация).
- **Данные:** PostgreSQL (WB/OZON read-only), Supabase (товарная матрица), SQLite (хранилище отчётов и очередь доставки).

## Ключевые возможности
1. **Финансовая аналитика:** Декомпозиция маржи по 5 рычагам (цена, СПП%, ДРР, логистика, себестоимость)
2. **Маркетинговая аналитика:** Рекламная воронка, CPO/CTR/CR, органическая воронка
3. **Ценовой анализ:** Эластичность спроса на основе ПЕРВОИСТОЧНИКА заказов (`orders`), рекомендации по ценам, симуляция изменений, мультифакторный анализ с контролем маркетингового бюджета. Внедрены "Quality Gates" (HAC статистика, контроль вариации и покрытия данных) для минимизации риска неверных рекомендаций.
4. **Автоматизация:** Ежедневные отчёты (10:05 MSK), недельные (пн 10:15), месячные (пн 10:30)
5. **Мониторинг данных:** Проверка свежести данных, верификация vs OneScreen/PowerBI

## Известные ограничения
1. Feedback handler не подключён — Олег НЕ учится на ошибках автоматически
2. Tool crash убивает ReAct loop — нет graceful degradation
3. Нет checkpoint'ов в ReAct loop — crash = потеря прогресса
4. FSM в MemoryStorage — сессии теряются при перезапуске
