# Олег — История прогресса

## Журнал обучения

История корректировок на основе реальных отчётов и feedback:

### 2026-02-17 (Logic Refinement)
**Что произошло:** Выявлены расхождения в бизнес-логике между правилами аналитики и системным промптом агента.

**Что исправлено:**
1. **СПП (Скидка МП):** Логика исправлена на: Рост СПП = Снижение цены для клиента → Рост спроса. Рекомендация: при росте СПП повышать базовую цену (для маржи), при снижении — снижать (для удержания цены).
2. **Реклама (Инвестиции):** Внедрено правило: Высокий ДРР продаж + Нормальный ДРР заказов = Инвестиция (НЕ отключать рекламу).
3. **Качество трафика:** Добавлен паттерн: Рост корзин без роста заказов = Сигнал интереса (трафик есть), но есть барьер (Цена/Контент).
4. **Выручка vs Заказы:** Четкое разделение: Выручка — результат прошлого, Заказы — индикатор будущего.

**Источник:** Аудит `playbook.md` и `ANALYTICS_RULES.md`.

### 2026-02-17 (Architecture)
**Что произошло:** Переход на архитектуру Producer-Consumer для разделения логики аналитики и UI.

**Что исправлено:**
1. Создан `agent_runner.py` для автономной работы по расписанию.
2. Реализована `delivery_queue` в SQLite (WAL mode) для передачи отчётов в бот.
3. Добавлен механизм автоматического восстановления за 3 дня (Recovery).
4. Введена PID-блокировка (`oleg_agent.pid`) для защиты от дублирования процессов.

**Источник:** Архитектурная ревизия для повышения uptime.

### 2026-02-16

### 2026-02-12
**Что произошло:** Олег "праздновал" снижение рекламы, не проверив падение трафика на -25%.

**Что исправлено:** Добавлено правило **Trade-off анализ** — снижение рекламы → рост маржи/ед, НО падение показов/переходов/заказов. Нельзя рекомендовать "сократить ещё" если трафик уже падает.

**Источник:** Сравнение с отчётом живого аналитика

## Известные ограничения

❌ Feedback handler не подключён — Олег НЕ учится на ошибках автоматически  
❌ Tool crash убивает ReAct loop — нет graceful degradation  
❌ Нет checkpoint'ов — crash = потеря прогресса  
❌ Контекст не сжимается — длинные диалоги переполняют context window  
❌ FSM в MemoryStorage — сессии теряются при перезапуске  
❌ Нет rate limiting — один пользователь может спамить дорогие запросы
