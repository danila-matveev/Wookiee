# Олег — Agent Specification

## Статус
- **Версия:** 1.0
- **Готовность:** Production (70%)
- **Последнее обновление:** 2026-02-14
- **Статус развёртывания:** Работает в тестовой среде, не в production Telegram

## Миссия

Олег — ИИ финансовый и бизнес-аналитик бренда Wookiee. Предоставляет:
- Ежедневные финансовые отчёты (маржа, выручка, драйверы, рекомендации)
- Ответы на аналитические запросы в реальном времени
- Мониторинг качества данных и отклонений от целей
- Декомпозицию изменений маржинальности по 5 ключевым рычагам

**Главная задача:** Помочь управленческой команде принимать решения на основе данных, а не интуиции.

## Архитектура

### Тип агента
**ReAct agent** (Reasoning + Acting) — полноценный автономный агент с tool-use:
- Самостоятельно определяет какие данные запросить
- Итеративно углубляется в проблему (ReAct loop)
- Проверяет свои расчёты через инструменты перед выводом
- Может принимать/отклонять обратную связь с обоснованием

### LLM провайдеры
| Модель | Назначение | Стоимость (за 1K токенов) | Статус |
|--------|-----------|--------------------------|--------|
| `glm-4-plus` (z.ai) | Основная аналитика, tool-use, ReAct loop | $0.007 input/output | **Рабочая** |
| `glm-4.5-flash` (z.ai) | Classify/clarify запросов пользователя | $0.0001 input, $0.0002 output | **Рабочая** |
| `claude-opus-4-6` | Резерв (не используется, но настроен) | $0.015 input, $0.075 output | Standby |

**Выбор провайдера:** z.ai выбран как основной из-за баланса качество/цена. Claude Opus 4.6 готов как fallback для сложных случаев.

### Интерфейс
- **Telegram** (aiogram 3.15) — основной интерфейс
- Parse mode: HTML (для форматирования)
- Middleware: DI (dependency injection) для сервисов
- FSM: MemoryStorage (сессии не персистируются между перезапусками)

### Хранилище данных
| База данных | Назначение | Режим доступа | Статус |
|------------|-----------|--------------|--------|
| **PostgreSQL (WB)** | Продажи, финансы, трафик WB | Read-only | Работает |
| **PostgreSQL (OZON)** | Продажи, финансы, трафик OZON | Read-only | Работает |
| **Supabase** | Товарная матрица (SKU, статусы, цены) | Read через REST API | Работает |
| **SQLite** | История отчётов, кэш запросов | Read/Write | Работает |

**Важно:** Олег НЕ имеет прав на запись в PostgreSQL (WB/OZON). Все SQL-запросы read-only.

### ReAct цикл (параметры)
```python
max_iterations: 10          # Максимум 10 шагов думания
max_tool_calls: 20          # Максимум 20 вызовов инструментов
timeout: 120 секунд         # Таймаут на весь анализ
required_tools_deep: {      # Минимум для глубокого анализа
    "get_brand_finance",
    "get_channel_finance",
    "get_model_breakdown",
    "get_margin_levers"
}
```

**Логика работы:**
1. Получает запрос пользователя
2. LLM classify (glm-4.5-flash) определяет тип задачи
3. Генерирует план анализа (Reasoning)
4. Вызывает инструменты (Acting)
5. Анализирует результаты (Reasoning)
6. Повторяет шаги 4-5 до ответа или лимита итераций
7. Генерирует финальный отчёт (2 формата: краткая сводка + подробный)

## Playbook

**Файл:** `agents/oleg/playbook.md`
**Размер:** 456 строк
**Формат:** Markdown с таблицами, примерами, формулами

### Структура playbook:
1. **Бизнес-контекст и цели** — фокус компании, KPI, цели на январь 2026
2. **Правила анализа** — 5 рычагов маржи, лаговые показатели, ДРР с разбивкой, связка реклама-трафик-заказы
3. **Известные проблемы данных** — критические баги, чек-лист качества данных
4. **Формулы** — верифицированные формулы WB/OZON маржи, выручки, продаж (vs OneScreen)
5. **ОПИУ (P&L)** — правила отклонений чистой прибыли
6. **Формат анализа** — краткая сводка (30-40 строк) + подробный отчёт (Markdown)
7. **Правила периодных отчётов** — недельные, месячные, произвольные периоды
8. **Обратная связь и самопроверка** — как Олег обрабатывает feedback (перепроверяет через tools)

### Ключевые правила (выдержки):
- **5 рычагов маржи:** Цена до СПП, СПП%, ДРР (внутр+внешн), Логистика ₽/ед, Себестоимость ₽/ед
- **Лаговые показатели:** Выкуп% (лаг 3-21 дней) — НЕЛЬЗЯ использовать как причину дневных изменений маржи
- **ДРР — ВСЕГДА с разбивкой:** внутренняя (МП) + внешняя (блогеры, ВК)
- **Связка Реклама→Трафик→Заказы:** Если реклама выросла — проверить заказы. Если заказы растут — реклама эффективна.
- **Trade-off при снижении рекламы:** Маржа/ед растёт, НО трафик падает → падение заказов. Нельзя рекомендовать "сократить ещё" если трафик уже падает.
- **СПП% при объединении каналов:** ТОЛЬКО средневзвешенный `sum(spp_amount) / sum(revenue_before_spp) * 100`
- **GROUP BY по модели:** ОБЯЗАТЕЛЬНО с `LOWER(SPLIT_PART(article, '/', 1))` — без этого суммы искажаются на десятки процентов
- **Верификация перед анализом:** Расхождение с OneScreen < 1% для маржи и выручки WB

## Известные проблемы

| # | Проблема | Влияние | Приоритет | Статус |
|---|---------|---------|----------|--------|
| 1 | **Feedback handler не подключён** | Олег НЕ учится на ошибках, playbook не обновляется автоматически | 🔴 High | Open |
| 2 | **Tool crash убивает ReAct loop** | При ошибке в одном tool весь анализ падает (нет try-catch в AgentExecutor) | 🔴 High | Open |
| 3 | **Нет checkpoint'ов в ReAct loop** | Crash в середине анализа = потеря всего прогресса (10 tool calls могут пропасть) | 🟡 Medium | Open |
| 4 | **Контекст не сжимается** | Длинные диалоги переполняют context window (нет summarization старых сообщений) | 🟡 Medium | Open |
| 5 | **FSM в MemoryStorage** | Сессии пользователей теряются при перезапуске бота | 🟢 Low | Open |
| 6 | **Нет rate limiting** | Один пользователь может спамить дорогие ReAct запросы | 🟡 Medium | Open |
| 7 | **Отчёты WB/OZON расходятся на ~7%** | Известное расхождение OZON выручки с OneScreen (п.9 DATA_QUALITY_NOTES) | 🟢 Low | Documented |

### Детали проблемы #2 (Tool crash)
**Сценарий:**
1. Олег начинает анализ → вызывает `get_brand_finance`
2. PostgreSQL timeout / сетевая ошибка
3. Tool выбрасывает exception
4. AgentExecutor НЕ перехватывает → весь ReAct loop падает
5. Пользователь получает: "Произошла ошибка при анализе"

**Решение:** Обернуть tool calls в try-except, возвращать `{"error": "..."}` вместо raise.

### Детали проблемы #3 (Checkpoints)
**Текущая логика:**
```python
for iteration in range(max_iterations):
    tool_call = llm.generate()
    result = execute_tool(tool_call)
    # Если тут crash — весь прогресс теряется
```

**Решение:** Сохранять intermediate results в SQLite после каждого tool call. При crash — resume с последнего checkpoint.

## Метрики производительности

### Стоимость

**Средняя стоимость запроса (за ReAct цикл):**
- **Простой запрос** (1-2 tool calls): ~$0.02
- **Стандартный отчёт** (4-6 tool calls): ~$0.05
- **Глубокий анализ** (8-10 tool calls): ~$0.12

**Формула:**
```
Cost = (input_tokens × $0.007 + output_tokens × $0.007) / 1000
     + (classify_input × $0.0001 + classify_output × $0.0002) / 1000
```

**Примерная разбивка** (стандартный отчёт):
- LLM classify: ~500 tokens (~$0.0001)
- ReAct loop (glm-4-plus): ~7000 input + ~3000 output (~$0.07)
- **Итого:** ~$0.05

**Месячный бюджет** (при 3 запросах/день × 30 дней):
- 90 запросов × $0.05 = **$4.50/месяц**

### Время выполнения

| Тип запроса | Среднее время | p95 | Таймаут |
|------------|--------------|-----|---------|
| Classify запроса | 0.5 сек | 1.2 сек | 5 сек |
| Простой запрос (1-2 tools) | 8-12 сек | 18 сек | 120 сек |
| Стандартный отчёт (4-6 tools) | 30-50 сек | 75 сек | 120 сек |
| Глубокий анализ (8-10 tools) | 60-90 сек | 115 сек | 120 сек |

**Узкие места:**
1. PostgreSQL запросы: 1-3 сек каждый (зависит от размера периода)
2. LLM генерация (glm-4-plus): 2-5 сек на итерацию
3. Supabase REST API: 0.5-1 сек

### Надёжность

| Метрика | Значение | Период измерения |
|---------|---------|------------------|
| Uptime | 98.5% | Январь 2026 (тестовая среда) |
| Error rate | 1.5% | Январь 2026 |
| Success rate (tool calls) | 99.2% | Январь 2026 |

**Основные причины ошибок:**
- PostgreSQL timeouts: 0.8%
- LLM API timeouts: 0.5%
- Invalid user input: 0.2%
