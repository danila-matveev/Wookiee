# Описание отчёта по оптимизации локализации (версия 3.0)

**Дата обновления:** 10.02.2026
**Файл:** `Отчет_локализация_v3_DD-MM-YYYY.xlsx`
**Скрипт:** `generate_localization_report_v3.py`

---

## Цель отчёта

Отчёт предназначен для оптимизации индекса локализации товаров на Wildberries. Цель — повысить индекс локализации до 75%+ путём перемещения товаров между складами ближе к регионам с высоким спросом.

**Индекс локализации** — это доля заказов, доставленных с локального склада (склада в том же регионе, где сделан заказ). Высокий индекс локализации:
- Снижает стоимость логистики
- Ускоряет доставку покупателям
- Улучшает позиции в поиске WB

---

## Общая логика алгоритма

1. Загружает данные WB: историю остатков и поставки по регионам.
2. Для каждого SKU (артикул + размер) рассчитывает индекс локализации.
3. Находит SKU с индексом ниже целевого (75%).
4. Для каждого проблемного SKU анализирует распределение остатков и заказов по регионам.
5. Определяет регионы-доноры (избыток остатков) и регионы-получатели (дефицит).
6. Формирует рекомендации по перемещению с учётом защиты доноров (safety buffer).
7. Загружает статусы артикулов из Wookiee SKU Database (Supabase).
8. Исключает товары со статусом «Выводим» из допоставок (перемещать имеющиеся остатки можно).
9. Перемещения и допоставки формируются **только между российскими регионами**.

**Ключевые принципы:**
- Перемещения возможны только между российскими регионами (международные исключены).
- Товары со статусом «Выводим» не попадают в план допоставок (допоставлять бессмысленно — они выводятся из продажи), но их имеющиеся остатки можно и нужно перемещать.
- Доноры защищены: у каждого донора остаётся запас на N дней спроса (safety buffer).

## Запуск

```bash
cd "MP scripts/Index_localization WB"

# Базовый запуск
python3 generate_localization_report_v3.py "Data ООО/10.02"

# С меньшим буфером безопасности (7 дней вместо 14)
python3 generate_localization_report_v3.py "Data ООО/10.02" --safety-days 7

# Сравнение с предыдущим периодом
python3 generate_localization_report_v3.py "Data ООО/10.02" --compare-with "Data ООО/04.02"

# Без загрузки статусов из Supabase
python3 generate_localization_report_v3.py "Data ООО/10.02" --no-statuses
```

| Параметр | По умолчанию | Описание |

В папке данных скрипт автоматически находит два файла:

Период данных определяется автоматически из названия файла (разница дат).

## Структура отчёта (листы Excel)

### 1. Сводка

Главный дашборд отчёта. Все ключевые метрики одним взглядом.

**Разделы:**
- **ОБЩАЯ СТАТИСТИКА** — количество SKU, средний/медианный индекс локализации, период данных.
- **РАСПРЕДЕЛЕНИЕ SKU** — сколько SKU в зонах Критично / Внимание / OK.
- **ПЕРЕМЕЩЕНИЯ** — количество перемещений по приоритетам, допоставок.
- **ЗАЩИТА ДОНОРОВ** — буфер безопасности, доноры с ограничением, защищённый объём.
- **СТАТУСЫ** (если загружены) — сколько артикулов в базе, сколько «Выводим», сколько исключено из допоставок.

### 2. 1_Критичные

**Первоочередные перемещения.** SKU с индексом <50% и высоким объёмом заказов.

**Критерии:**
- Индекс локализации < 50% и заказов >= 10 → Приоритет «Критично+» (35 баллов)
- Индекс локализации < 50% → Приоритет «Критично» (30 баллов)

**Как действовать:** Выполнить в первую очередь — максимальное влияние на общий индекс.

### 3. 2_Важные

**Важные перемещения.** SKU с индексом 50–75%.

**Критерии:**
- Индекс 50–75% и заказов >= 10 → Приоритет «Важно+» (25 баллов)
- Индекс 50–75% → Приоритет «Важно» (20 баллов)

**Как действовать:** Выполнить после критичных.

### 4. 3_Остальные

**Некритичные перемещения.** Индекс >= 75% — перемещения для точной настройки.

Приоритет: «OK» (10 баллов).

**Как действовать:** Можно отложить или объединить с другими задачами.

### 5. Все перемещения

Полный список всех рекомендованных перемещений (объединение листов 1–3). Удобен для фильтрации и поиска конкретного артикула.

### 6. Анализ_SKU

Детальная статистика по каждому SKU: заказы (всего/локальные/нелокальные), индекс локализации, остатки, статус из Wookiee SKU Database.

### 7. Защита доноров

Показывает, сколько товара можно безопасно забрать с каждого региона-донора.

**Логика защиты:**
- У донора остаётся запас на `safety_days` дней общего спроса (локальный + нелокальный).
- Если спроса нет — весь остаток доступен для трансфера.
- Если остатков не хватает на буфер — трансфер заблокирован.

### 8. Допоставки

Позиции, для которых невозможно перераспределить имеющиеся остатки — требуется дополнительная поставка.

**Важно:** Товары со статусом «Выводим» из Wookiee SKU Database не включаются в допоставки.

### 9. Регионы

Сводка по регионам: текущие остатки vs. распределение заказов, целевые остатки, рекомендации.

### 10. Динамика (опционально)

Появляется при использовании `--compare-with`. Сравнение индекса локализации по регионам между текущим и предыдущим периодами.

## Логика приоритизации

### Категории приоритета

| Категория | Условия | Балл | Влияние |
|-----------|---------|------|---------|
| **Критично+** | Индекс < 50% и заказов >= 10 | 35 | Максимальное |
| **Критично** | Индекс < 50% | 30 | Высокое |
| **Важно+** | Индекс 50–75% и заказов >= 10 | 25 | Значительное |
| **Важно** | Индекс 50–75% | 20 | Среднее |
| **OK** | Индекс >= 75% | 10 | Минимальное |

Пороги: `TARGET_LOCALIZATION_INDEX = 75%`, `CRITICAL_INDEX = 50%`, `HIGH_ORDERS = 10`.

## Защита доноров

Алгоритм не забирает у региона-донора весь остаток. Для каждого SKU в каждом регионе рассчитывается:

```
daily_orders = общие_заказы_региона / дней_в_периоде
safe_minimum = ceil(daily_orders * safety_days)
available = max(0, текущие_остатки - safe_minimum)
```

По умолчанию `safety_days = 14` — у донора остаётся запас на 2 недели спроса.

**Специальные случаи:**
- Нет остатков → трансфер невозможен
- Нет спроса → весь остаток доступен для трансфера
- Остатки <= буфер → трансфер заблокирован

## Фильтрация по статусам

Скрипт загружает статусы артикулов из Wookiee SKU Database (Supabase) через `scripts/data_layer.py → get_artikuly_statuses()`.

**Влияние статуса «Выводим»:**
- Перемещения — товары со статусом «Выводим» **участвуют** в перемещениях (имеющиеся остатки можно перемещать ближе к спросу).
- Допоставки — товары со статусом «Выводим» **исключаются** из допоставок (нет смысла допоставлять товар, который выводится из продажи).
- Товары без статуса в базе — по-прежнему попадают и в перемещения, и в допоставки (не фильтруются).

**Graceful degradation:** Если Supabase недоступен или `data_layer` не найден, скрипт продолжает работу без фильтрации. Флаг `--no-statuses` отключает загрузку статусов.

## Рекомендуемый порядок действий

1. **Открыть «Сводка»** — оценить масштаб проблемы: средний индекс, количество критичных SKU, объём перемещений.

2. **Просмотреть «1_Критичные»** — перемещения с максимальным влиянием на индекс. Выполнить в первую очередь.

3. **Затем «2_Важные»** — основной объём перемещений. Выполнять по мере доступных ресурсов.

4. **«3_Остальные»** — можно отложить или объединить с другими задачами.

5. **Проверить «Допоставки»** — если нехватка товара не решается перемещением. Убедиться, что нет товаров со статусом «Выводим» (они должны быть исключены автоматически).

6. **«Регионы»** — увидеть полную картину: где избыток, где дефицит остатков.

7. **«Защита доноров»** — убедиться, что у доноров остаётся достаточный запас.

## Ограничения

- Данные по заказам берутся за один период (обычно 30 дней). Для более точных рекомендаций желательны данные за 60+ дней.
- Перемещения возможны **только между российскими регионами**. Международные регионы (Беларусь, Казахстан, Армения, Узбекистан) исключены.
- Баркоды загружаются из `Спецификации.xlsx`. Если файл не найден, отчёт формируется без баркодов.
- Статусы зависят от подключения к Supabase. При отсутствии подключения фильтрация «Выводим» не работает.
- Рекомендуемые склады — ориентировочные. Реальный склад для приёмки может отличаться.

## Контакты

При вопросах по отчёту обращайтесь к аналитику данных.
