# Оптимизация индекса локализации WB

## Что это
Скрипт анализирует распределение остатков по складам WB и генерирует рекомендации по перемещению товаров для повышения индекса локализации до 75%+.

## Запуск
```bash
python3 generate_localization_report.py
```

## Входные данные

| Файл | Откуда | Что содержит |
|------|--------|--------------|
| `Итория остатков...xlsx` | WB ЛК → Аналитика → История остатков | Остатки по складам |
| `Поставки по регионам...xlsx` | WB ЛК → Аналитика → Поставки по регионам | Заказы локальные/нелокальные |
| `Спецификации.xlsx` | База SKU | Баркоды (лист "Все товары") |

**Обновление данных:** скачать свежие файлы из ЛК WB, положить в `Data/`, обновить имена в скрипте (строки 18-19).

## Выходной файл
`Data/Отчет_оптимизация_локализации.xlsx`

### Листы

| Лист | Что содержит |
|------|--------------|
| **Сводка** | Общая статистика |
| **1_Критичные** | Перемещения первой очереди (индекс <50% + заказы >=10) |
| **2_Важные** | Перемещения второй очереди (индекс <75% + заказы >=5) |
| **3_Остальные** | Можно отложить |
| **Анализ_SKU** | Все SKU с индексами и баркодами |
| **Допоставки** | Нужна дозакупка (нечего перераспределять) |
| **Регионы** | Баланс остатков vs заказов по регионам |

## Логика расчёта

```
Индекс локализации = Локальные заказы / Всего заказов × 100%
```

**Локальный заказ** — заказ, выполненный со склада в том же регионе, где находится покупатель.

### Алгоритм перемещений
1. Для каждого SKU с индексом <75% рассчитываем долю заказов по регионам
2. Определяем целевые остатки = доля заказов × текущие остатки
3. Регионы с избытком → доноры, с дефицитом → получатели
4. Генерируем перемещения от доноров к получателям
5. Если остатков недостаточно → рекомендация на допоставку

### Приоритеты
| Приоритет | Условие | Действие |
|-----------|---------|----------|
| 🔴 Критично+ | Индекс <50% + заказы >=10 | Делать сразу |
| 🔴 Критично | Индекс <50% | Высокий приоритет |
| 🟠 Важно+ | Индекс <75% + заказы >=10 | Средний приоритет |
| 🟠 Важно | Индекс <75% | Обычный приоритет |
| 🟢 OK | Индекс >=75% | Не требует действий |

## Структура колонок

### Перемещения
- **Баркод** — для сканирования при отборе
- **Артикул** — артикул продавца
- **Размер** — S/M/L/XL
- **Откуда регион/склад** — где забираем
- **Куда регион/склад** — куда везём
- **Кол-во** — сколько штук
- **Индекс SKU, %** — текущий индекс
- **Заказов** — количество заказов за период

### Анализ SKU
- **Приоритет** — категория срочности
- **Баркод** — основной баркод
- **Всего заказов** — общее количество
- **Локальные/Нелокальные** — разбивка заказов
- **Индекс, %** — текущий индекс локализации
- **Остатки** — текущие остатки на WB

## Версия 3 (с защитой доноров)

### Что нового
Скрипт `generate_localization_report_v3.py` максимизирует индекс локализации, но при этом **не допускает ухудшения** индекса ни в одном регионе-доноре. Это решает проблему, когда забирая товар из региона для повышения общего индекса, мы случайно роняем локализацию в этом регионе.

### Запуск v3
```bash
# Базовый запуск (папка с данными как аргумент)
python3 generate_localization_report_v3.py "Data 08.02"

# Со сравнением с предыдущим периодом
python3 generate_localization_report_v3.py "Data 08.02" --compare-with "Data 04.02"

# С другими параметрами защиты
python3 generate_localization_report_v3.py "Data 08.02" --safety-days 7 --min-donor-localization 75
```

### Параметры CLI

| Параметр | По умолчанию | Описание |
|----------|-------------|----------|
| `data_folder` | (обязательный) | Папка с данными, например `"Data 08.02"` |
| `--sku-db` | `../wookiee_sku_database/Спецификации.xlsx` | Путь к файлу баркодов |
| `--compare-with` | нет | Папка предыдущего периода для сравнения |
| `--safety-days` | 14 | Дней запаса для донора |
| `--min-donor-localization` | 70 | Не ронять локализацию донора ниже этого % |

### Защита доноров

При перемещении товара из региона-донора проверяется два ограничения:

1. **По спросу**: у донора должен остаться запас на `safety-days` дней собственных заказов
2. **По локализации**: если текущая локализация донора <= `min-donor-localization`, он блокируется как донор

Если доступного излишка не хватает — дефицит уходит в допоставки.

### Дополнительные листы в отчёте v3

| Лист | Что содержит |
|------|--------------|
| **Защита доноров** | По каждому донору: остатки, безопасный минимум, доступно, фактор ограничения |
| **Динамика** | Сравнение индекса с предыдущим периодом (если `--compare-with`) |

### Обновление данных (v3, ручной режим)

1. Скачать из WB ЛК → Аналитика: «История остатков» и «Поставки по регионам»
2. Создать папку `Data XX.XX` (например `Data 15.02`)
3. Положить два файла в эту папку
4. Запустить: `python3 generate_localization_report_v3.py "Data 15.02" --compare-with "Data 08.02"`
5. Результат в `Отчеты готовые/`

## Автоматический режим (через API)

Скрипт `run_localization.py` — автоматически получает данные через WB API + МойСклад и генерирует отчёт без ручного скачивания Excel.

### Что делает

1. **WB API warehouse_remains** → остатки по складам WB
2. **WB API supplier/orders** → заказы за период → расчёт local/non-local
3. **МойСклад API** → остатки своего склада (для допоставок)
4. Вызывает v3 логику → Excel-отчёт с перемещениями

### Запуск

```bash
# Из корня проекта:

# Оба кабинета (ИП + ООО)
python "MP scripts/Index_localization WB/run_localization.py"

# Только один кабинет
python "MP scripts/Index_localization WB/run_localization.py" --cabinet ip
python "MP scripts/Index_localization WB/run_localization.py" --cabinet ooo

# С другим периодом и параметрами
python "MP scripts/Index_localization WB/run_localization.py" --cabinet ip --days 14 --safety-days 7

# Только посмотреть сводку (без генерации отчёта)
python "MP scripts/Index_localization WB/run_localization.py" --dry-run
```

### Параметры CLI

| Параметр | По умолчанию | Описание |
|----------|-------------|----------|
| `--cabinet` | `both` | `ip`, `ooo`, `both` |
| `--days` | 30 | Период заказов (дней назад) |
| `--safety-days` | 14 | Буфер безопасности донора |
| `--min-donor-localization` | 70 | Мин. локализация донора % |
| `--max-turnover-days` | 100 | Макс. оборот при допоставке (дней) |
| `--no-statuses` | false | Не загружать статусы из Supabase |
| `--dry-run` | false | Только показать сводку |

### Допоставки с учётом своего склада

В автоматическом режиме лист "Допоставки" содержит дополнительные колонки:

| Колонка | Описание |
|---------|----------|
| **Кол-во** | Сколько нужно допоставить (из v3 расчёта) |
| **Макс. по обороту** | Ограничение: не создавать запас > 100 дней оборота |
| **На своём складе** | Сколько реально есть на своём складе (из МойСклад) |
| **К допоставке (факт)** | Итого к отгрузке: min(нужно, макс. по обороту, на своём складе) |

### Время выполнения

~20-30 минут (из-за rate limit WB API: 1 запрос/мин и асинхронная генерация warehouse_remains)

### Требования

- API ключи в `.env`: `WB_API_KEY_IP`, `WB_API_KEY_OOO`
- Токен МойСклад в `.env`: `MOYSKLAD_TOKEN`
- Python пакеты: `httpx`, `pandas`, `openpyxl`, `python-dotenv`
