## 1. Контекст и проблема

### Что есть сейчас

- **Google Sheets «Спецификация Wookiee»** (`1bMyaUgVHvOl4lQDPMu3JgGEpzs8RsCWyu2GqxsB6XwI` или текущая основная) — заполняется командой вручную. Содержит листы: «Все модели», «Все товары», «Все артикулы», «Color Code», «Таблица склеек WB», «Таблица склеек Ozon».
- **Cron `sync_sheets_to_supabase` (06:00 UTC ежедневно)** — читает из Sheets и переносит в Supabase. Поведение: **Sheets → Supabase** (одностороннее, Sheets источник истины).
- **Hub (hub.os.wookiee.shop)** — после W10-цикла стал полноценным редактором каталога. Команда уже сейчас может править артикулы, цвета, склейки, атрибуты из интерфейса. Эти правки летят в Supabase напрямую, минуя Sheets.

### В чём боль

После W10 Hub стал реальным инструментом редактирования. Но команда продолжает работать в «Спецификации» (там удобнее массово править, фильтровать, выгружать). Получается:

- **Команда правит в Hub** → данные есть в Supabase, но **в Sheets их нет**.
- **Команда правит в Sheets** → ночной cron перетащит в Supabase, но **затрёт правки, сделанные в Hub** в течение дня.

То есть сейчас **источников истины два, синхронизации одного направления нет**, дрифт неизбежен.

### Что нужно

Один источник истины — **Supabase** (то, что пишет Hub). Команда продолжает работать в таблице, но таблица — **производная** от базы, не источник. Hub-кнопка «Обновить таблицу» переносит свежее состояние из БД в Sheets.

### Почему не «всё сразу»

Полное переключение прямо сейчас — рискованно: Hub ещё не прошёл боевую проверку всей командой. Если зальём данные из БД поверх «Спецификации» и где-то ошибёмся в маппинге — потеряем накопленную команду работу.

**Решение:** двухэтапный rollout.

1. **Этап 1 — зеркало (этот план):** создаём **новую таблицу** «Спецификация Wookiee — Зеркало Hub». Hub-кнопка пишет только туда. Команда продолжает работать в основной «Спецификации» как сейчас. На зеркало смотрим сбоку, сравниваем, ловим баги.
2. **Этап 2 — flip (отдельный план через 2–4 недели):** когда зеркало стабильно совпадает с основной — переименовываем зеркало в «Спецификация», старую таблицу архивируем, отключаем cron `sync_sheets_to_supabase`, ставим Protected ranges, переименовываем кнопку «Обновить зеркало» → «Обновить таблицу».

Откат на любом шаге — тривиальный: остановили запись в зеркало, основная таблица не пострадала.

## 2. Принципы проектирования

| Принцип | Что это значит на практике |
|---|---|
| **Mirror-first** | Hub пишет только в зеркальную таблицу. Основная «Спецификация» не трогается до flip. |
| **Delta-update** | Не перезаписываем лист целиком. Сравниваем БД и таблицу построчно по якорю, отправляем точечные `batchUpdate` с конкретными ячейками. Это сохраняет ручное форматирование, цвета, заметки. |
| **Anchor-based identification** | У каждого листа есть якорная колонка (бизнес-ключ), по которой ищем строку: «Модель основа» / «Артикул» / «БАРКОД» / «Color code» / «Название склейки + БАРКОД». |
| **Idempotency** | Повторный запуск без изменений в БД ≡ ноль операций в Sheets. Это даёт безопасный ретрай при сбое сети. |
| **Hub-empty не затирает** | Пустое поле в Hub НЕ перезаписывает ячейку в Sheets пустотой. Только реальные значения едут наверх. (Иначе команда лишится исторических заметок, которые не дублируются в БД.) |
| **No schema migration in Sheets** | Колонки в Sheets не двигаем и не добавляем автоматом. Если в Hub появилось новое поле (новый атрибут категории) — добавляем колонку в шапку листа вручную, после этого код её увидит по имени. |
| **Audit-friendly** | Каждый запуск синка пишется в `tool_runs`: tool_slug, started_at, duration_ms, cells_updated, status, errors. Видно в Hub на странице «История запусков». |

## 3. Структура зеркала (6 листов)

Новая Google Sheets-таблица **«Спецификация Wookiee — Зеркало Hub»**. Создаётся вручную в Фазе 0 как **копия структуры** основной таблицы (только шапки и форматирование, без данных). Заполняется первым запуском синка.

| № | Лист (имя совпадает с основной) | Сущность Supabase | Якорная колонка | Особенности |
|---|---|---|---|---|
| 1 | **Все модели** | `modeli_osnova` + `modeli` (LEFT JOIN, по одной строке на каждую `modeli`-запись) | `Модель основа` (например `Wendy`) | Денормализованные FK: бренд, коллекция, категория, фабрика, тип коллекции — берутся из ref-таблиц. Атрибуты модели (материал, состав, теги, …) — из `model_atributy` через pivot. |
| 2 | **Все артикулы** | `artikuly` (одна строка = один артикул) | `Артикул` (`Wendy/black`) | Цвет (Color code, hex, название) подтягивается из `cveta`. Статус из `artikuly.status`. |
| 3 | **Все товары** | `tovary` (одна строка = один SKU = один баркод) | `БАРКОД` | Канальные статусы: `Статус товара` (WB), `Статус товара OZON`. Идентификаторы маркетплейсов: `Ozon Product ID`, `FBO OZON SKU ID`, `Seller SKU Lamoda`, `SKU CHINA SIZE`. |
| 4 | **Color Code** | `cveta` | `Color code` | Поля: `Color code`, `Сolor` (англ. название), `Цвет` (русское), `HEX`, `Статус` (Продаётся/Выводим), `Семейство` (family), `Палитра/Коллекция`. |
| 5 | **Таблица склеек WB** | `skleyki_wb` + `tovary_skleyki_wb` + `artikuly_skleyki_wb` | `Название склейки` + `БАРКОД` (составной) | Одна строка = одно вхождение SKU в склейку. Колонки: `Название склейки`, `БАРКОД`, `Артикул`, `Модель`, `Цвет`, `Размер`, `Канал`, `Создано`. Удалённая в Hub строка → **физически удаляется** из листа (склейки не имеют статуса «Архив»). |
| 6 | **Таблица склеек Ozon** | `skleyki_ozon` + `tovary_skleyki_ozon` + `artikuly_skleyki_ozon` | то же | то же |

**Правило для удалений (не склейки):** строка, удалённая в Hub, в листе помечается как `Статус = Архив` (физически не удаляется). Это сохраняет историю и не ломает формулы, ссылающиеся на номер строки.

### Фаза 2 — Python sync-сервис (2 дня)

**Цель:** `runner.sync_all()` идемпотентно подтягивает БД в зеркало.

**Структура папки `services/sheets_sync/hub_to_sheets/`:**

```
hub_to_sheets/
├── __init__.py
├── README.md              # как запускать вручную, какие переменные нужны
├── config.py              # CATALOG_MIRROR_SHEET_ID + sheet name → view name map
├── exporter.py            # SupabaseExporter: fetch_view(view_name) → DataFrame
├── anchor.py              # AnchorIndex: build map "anchor_value" → row_number из листа
├── diff.py                # diff_sheet(db_rows, sheet_rows, anchor) → List[CellUpdate]
├── batch.py               # SheetsBatchWriter: упаковывает CellUpdate в batchUpdate-запросы
├── runner.py              # sync_all() / sync_one(sheet_name) / --smoke CLI
└── tests/
    ├── test_diff.py       # unit: empty Hub, new row, archived row, color match by code
    └── test_anchor.py     # unit: anchor index с дубликатами, спецсимволы
```

**Ключевые алгоритмы:**

- **`diff_sheet`** (стержень сервиса):
  - На вход: `db_rows` (из view), `sheet_rows` (текущее состояние листа), `anchor_col`, `column_map` (имя колонки БД → имя колонки Sheets).
  - На выход: список `CellUpdate(sheet, row, col, value)`.
  - Логика:
    - Для каждой строки БД: ищем по якорю в `sheet_rows`.
      - Найдена → построчно сравниваем все колонки из `column_map`. Если значение в БД ≠ значение в листе **и значение в БД не пустое** → добавляем `CellUpdate`. Если значение в БД пустое — пропускаем (правило #2).
      - Не найдена → append-операция: новая строка в конце листа.
    - Для каждой строки Sheets, которой нет в БД:
      - Если это склейки — `DeleteRow`.
      - Иначе — `UpdateCell` в колонку `Статус` = `Архив`.
- **`SheetsBatchWriter`** — батчит обновления по 200 ячеек на один `batchUpdate` (Google API лимит — 100 запросов/100 сек; 200 ячеек в одном запросе нам с запасом подходит). Retry с экспоненциальным backoff на 429/500.

**Конфиг (`config.py`):**

```python
SHEETS_MAP = {
    "Все модели":         {"view": "vw_export_modeli",        "anchor": "Модель основа"},
    "Все артикулы":       {"view": "vw_export_artikuly",      "anchor": "Артикул"},
    "Все товары":         {"view": "vw_export_tovary",        "anchor": "БАРКОД"},
    "Color Code":         {"view": "vw_export_cveta",         "anchor": "Color code"},
    "Таблица склеек WB":  {"view": "vw_export_skleyki_wb",    "anchor": ("Название склейки", "БАРКОД")},
    "Таблица склеек Ozon":{"view": "vw_export_skleyki_ozon",  "anchor": ("Название склейки", "БАРКОД")},
}
```

**CLI:**
```bash
python -m services.sheets_sync.hub_to_sheets.runner --sheet "Все модели"     # один лист
python -m services.sheets_sync.hub_to_sheets.runner --all                    # все 6
```

### Фаза 5 — Тестовый период (2–4 календарные недели)

**Цель:** убедиться, что зеркало стабильно совпадает с основной таблицей на реальных данных.

**Шаги:**

### Фаза 6 — Flip (1 рабочий день, отдельный план)

**После согласования отдельного решения по flip:**

1. В Hub переименовать кнопку «Обновить зеркало» → «Обновить таблицу».
2. Прогнать финальный `sync_all` на зеркало.
3. Переименовать в Google Sheets: «Спецификация Wookiee — Зеркало Hub» → «Спецификация Wookiee». Старую таблицу — в «Спецификация Wookiee — Архив до 2026-XX-XX», доступ только на чтение.
4. В `.env` свопнуть: значение `CATALOG_PRIMARY_SHEET_ID` ← бывший `CATALOG_MIRROR_SHEET_ID`. Удалить `CATALOG_MIRROR_SHEET_ID`.
5. Отключить старый cron `sync_sheets_to_supabase` в `wookiee_cron`.
6. В Google Sheets настроить **Protected ranges** на всех 6 листах: всё, кроме шапки и колонок «Заметки», «Комментарий» — read-only для всех, кроме сервис-аккаунта.
7. Обновить документацию: `docs/index.md`, `services/sheets_sync/README.md`.
8. 1 неделя мониторинга, потом физически удалить архивную таблицу (или оставить навсегда — без разницы).

**Deliverable:** flip выполнен, один источник истины, документация актуальна.

## 7. Риски и митигации

| Риск | Митигация |
|---|---|
| Google Sheets API rate limit (100 req / 100 sec на user, 500 read req / 100 sec на project) | Батчинг по 200 ячеек на запрос + exponential backoff на 429. Один `sync_all` = ~6 батчей max. |
| Расхождение схемы (в Hub новое поле, в листе нет колонки) | Код игнорирует поле, если колонки нет. В лог — warning «column X not found in sheet Y». Решается дописыванием колонки в шапку (правило #3). |
| Сетевой сбой посередине синка | Идемпотентность: повторный запуск без новых правок ≡ no-op. Прерванный синк безопасен (некоторые ячейки обновлены, остальные обновятся в след. раз). |
| Service-account токен истёк | Централизованные credentials в `services/sheets_sync/credentials/sa.json`, мониторинг expiry через существующий health-check. |
| Команда правит зеркало вручную (case в тестовый период) | На зеркале Protected ranges не ставим (вариант #8), но в шапке листа красная заметка «Это зеркало Hub. Правки будут стёрты». Если случайно поправили — следующий sync затрёт. |
| Дрифт во время теста: основная и зеркало разошлись из-за параллельных правок | Это **ожидаемое поведение** во время теста (правда основной таблицы делается командой, правда зеркала — Hub). QA-фаза 5 как раз для отлова таких случаев. На flip — финальный sync_all и расхождения исчезают. |
| Сложные атрибуты модели (динамический pivot) тормозят view | Если `vw_export_modeli` > 2 сек — переписать через materialized view с refresh-триггерами. Не делать сразу, добавить только если боль реальна. |

## 8. Scope — что НЕ делаем

- **Двусторонняя синхронизация.** Только Hub → Sheets. Sheets → Supabase остаётся через старый cron до flip.
- **Realtime push.** Никаких webhook-ов из Supabase в Sheets. Синк руками по кнопке + опциональный ночной cron.
- **Schema management из кода.** Колонки в Sheets — ручное добавление в шапку. Это feature, а не bug.
- **History/audit в Sheets.** История правок остаётся в `audit_log` Supabase (W10 уже сделал). В Sheets история — только через Google Sheets «Version history».
- **Множественные зеркала.** Одно зеркало, одна основная, флип. Не плодим окружения.
- **Поддержка других таблиц.** Только «Спецификация». «Финансы», «Заказы» и т. п. — отдельные планы.
