## W9.1 — Permission denied: `istoriya_izmeneniy` (НА ВСЕХ CRUD)

**Симптом:** при создании артикулов через модалку "Создать артикулы" появляется
ошибка `permission denied for table istoriya_izmeneniy`.

**Масштаб (важно!):** это аудит-триггер. Он срабатывает на **любой** INSERT/UPDATE/DELETE
по аудируемым таблицам каталога. То есть БАГ ВЛИЯЕТ ТАКЖЕ НА:
- inline-edit в матрице (W8.5)
- inline-edit в /catalog/artikuly (W9.10)
- inline-edit в /catalog/tovary (W9.10)
- редактирование SKU из карточки модели
- удаление любых сущностей
- bulk-операции W4.6

Все эти места сейчас, скорее всего, **молча падают** или показывают непонятную ошибку.

**Гипотеза:** W7.1 добавил аудит-триггеры, которые пишут в таблицу с русским
именем `istoriya_izmeneniy`. Триггер выполняется от имени роли вызывающего запроса
(`authenticated`), но `GRANT INSERT ON istoriya_izmeneniy TO authenticated` не выдан
(или таблица создана позже самого триггера / переименована).

**Что сделать:**
1. Через Supabase MCP проверить:
   - существует ли таблица `istoriya_izmeneniy`, owner, GRANT-ы
   - на каких таблицах висят аудит-триггеры (`pg_trigger`)
   - `SECURITY DEFINER` или `SECURITY INVOKER` у функции триггера
2. Применить миграцию (`apply_migration` → `024_audit_grants.sql`):
   - `GRANT INSERT, SELECT ON istoriya_izmeneniy TO authenticated, service_role;`
   - Если функция SECURITY INVOKER — переключить на DEFINER + `SET search_path = public`.
   - Включить RLS + policy: `authenticated SELECT/INSERT свои записи`.
3. Прогнать smoke-сценарий по ВСЕМ операциям:
   - Создание артикула в модалке
   - Inline-edit в матрице (поле статуса)
   - Inline-edit в /catalog/artikuly (после W9.10)
   - Inline-edit в /catalog/tovary
   - Удаление SKU
   - Bulk-операции

**Acceptance:**
- Все CRUD-операции каталога работают без permission-ошибок.
- В `istoriya_izmeneniy` появляются записи о действиях.
- RLS не открывает таблицу для `anon`.

## W9.3 — Поиск на ВСЕХ реестрах не находит существующие записи

**Симптом:** существующий SKU `Lucky/black` есть в БД, но поиск на /catalog/tovary
по строке `Lucky/black` или `Lucky` ничего не возвращает.

**Масштаб:** проверить поиск на ВСЕХ трёх реестрах:
- `/catalog` (матрица моделей) — поиск по названию, бренду, артикулу
- `/catalog/artikuly` — по имени артикула, модели, цвету, баркоду
- `/catalog/tovary` — по баркоду, артикулу, модели, цвету, размеру

**Где смотреть:**
- `wookiee-hub/src/pages/catalog/matrix.tsx`
- `wookiee-hub/src/pages/catalog/artikuly.tsx`
- `wookiee-hub/src/pages/catalog/tovary.tsx`
- `wookiee-hub/src/lib/catalog/service.ts` → функции загрузки/фильтрации

**Что сделать:**
1. Унифицировать паттерн поиска: серверный fuzzy search через `ILIKE %term%` по
   конкатенации ключевых полей (или PostgreSQL `to_tsvector` + `plainto_tsquery` для
   полноценного FTS).
2. Если поиск на клиенте, но пагинация рубит выборку — перенести на сервер.
3. Регистр-инвариантность: всегда `ILIKE` или `LOWER()`.
4. Дебаунс 300ms на input, чтобы не дёргать сервер каждое нажатие.

**Acceptance:**
- На каждом из 3 реестров поиск находит существующие записи.
- Регистр не влияет.
- Поиск работает по нескольким полям.

## W9.8 — Неверное отображение размеров

**Симптом:** для модели Ruby в колонке "Размеры" отображается 4 значения, фактически
в комплект входит 3.

**Гипотеза:** агрегатор размеров суммирует все SKU модели вместо размеров одного
комплекта (вариации цветов имеют разные размеры — попадает union, а должен быть
размерный ряд комплекта).

**Что сделать:**
1. Уточнить продуктовую логику: размер привязан к артикулу или к SKU? Если размерный ряд
   общий для модели — брать `array_agg(DISTINCT size)` по модели. Если размерный ряд
   зависит от цвета — показывать диапазон для каждой вариации.
2. Скорее всего корректнее: в колонке "Размеры" матрицы показать размерный ряд комплекта
   (тот, что относится к набору, а не сумму всех SKU). Уточнить с пользователем перед
   реализацией, либо взять минимальный размерный ряд (intersection всех артикулов).
3. Проверить на Ruby глазами после фикса.

**Acceptance:**
- Ruby показывает 3 размера.
- Проверены 5 моделей разных типов, размеры совпадают с реальностью.

## W9.18 — Человекочитаемые toast-ошибки вместо raw SQL

**Симптом:** в проде вылетают непонятные ошибки типа
`permission denied for table istoriya_izmeneniy`. Это сырой текст PostgreSQL,
пользователь не понимает что делать.

**Что сделать:**
1. Глобальный error-handler в `wookiee-hub/src/lib/catalog/service.ts`:
   - Маппинг известных ошибок Supabase → человеческий русский текст.
   - Сетевые ошибки → `Не удалось связаться с сервером, проверьте подключение`.
   - Permission/RLS → `Недостаточно прав на это действие. Обратитесь к администратору`.
   - Unique violation → `Запись с таким значением уже существует`.
   - Foreign key violation → `Связанные данные не позволяют выполнить операцию`.
2. Все catch-блоки в UI должны звать `toast.error(translateError(e))`, а не `toast.error(e.message)`.
3. Логировать оригинал в `console.error` для отладки.

**Acceptance:**
- Нигде в UI нет сырого SQL-текста ошибок.
- Все ошибки на русском, понятны пользователю.
